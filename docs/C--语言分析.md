## 0. 先认识 C-- 这门语言

你写 C 语言写过 `int a = 1;`，C-- 长得跟它很像。但 C-- 是这门课为了教学特意"砍"出来的一门小语言——把 C 里复杂的、用不上的东西全删掉，只留下最核心、最值得学的部分，让你能在课程里亲手写出一个编译器来翻译它。

砍掉了什么呢？没有指针，没有 `for` 循环，没有 `switch`，没有 `++` `--`，类型也只剩下 `int`、`float` 和 `struct`（结构体），连字符型 `char` 都没有。你可能会觉得"这也太弱了吧"，但这恰恰是关键：语言越小，你能完整掌握它的全部规则，写编译器时就越清楚每一步该干什么。

举个例子，下面这段是一段合法的 C-- 代码，你可以感受一下它的样子：

```c
int factorial(int n) {
    if (n <= 1) return 1;
    return n * factorial(n - 1);
}
```

看起来和 C 几乎一样对吧？是的，C-- 在"长相"上和 C 高度相似，区别在于它**规则更少、限制更明确**。所以接下来分析 C-- 时，有一个最重要的原则：**一切以附录 A 为准，不要凭 C 的经验下结论**。比如你要是默认"肯定有 `for` 循环"，那就错了——C-- 没有。

## 1. "分析一门语言"到底要分析什么

很多人一上来就说"分析语言嘛，不就是看看词法、看看语法"。其实不止。要看懂一门语言，要分三个层面去看，附录 A 也是按这个顺序写的。

第一个层面是**词法**，回答的是"字面上，哪些字符凑在一起算一个'词'"。比如 `123` 这三个字符拼在一起算一个整数，`if` 这两个字符拼在一起算一个关键字。这是最底层的、关于"怎么写"的问题。

第二个层面是**语法**，回答的是"这些词怎么组合起来，才算是合法的句子和程序"。比如 `int a = 1;` 是合法的，但 `int = 18 age ;` 虽然每个词都认识，组合起来就是胡言乱语——这不合法。语法管的就是这种"句子结构对不对"的事。

第三个层面经常被忽略，叫**语义约束**（或者叫"潜规则")，回答的是"有些写法语法上是允许的，但到底算不算错"。举个例子，`int;` 这样一条什么都没定义的语句，你会本能觉得是错的，但 C-- 明确说它是合法的、不该报错。这种"藏在补充说明里的边界情况"，就是语义约束要管的。

为什么要特意拎出第三个层面？因为它最容易让你写出错误的编译器——把不该报错的报了，或者该报错的漏了。所以这份文档会专门花一节讲它。前三节分别分析这三层，最后一节再讲怎么把这些分析用到 Flex/Bison 实验上。

## 2. 词法层：C-- 的"词"长什么样

### 2.1 先想清楚：什么是"词"

你拿一行代码 `int age = 18;` 给计算机看，在计算机眼里它不是"int、age、=、18、分号"这几个有意义的单位，而是一串毫无意义的字符：`i`、`n`、`t`、空格、`a`、`g`、`e`、空格、`=`…… 如果让后面的程序直接面对这一堆零散字符，会非常痛苦，因为它得自己判断"i、n、t 三个连起来其实是 `int` 这个词"。

所以词法分析做的事情，就是先把字符"拼"成一个个有意义的单位，这些单位就叫 **Token**（词法单元，也有人叫"单词"）。上面那行代码经过词法分析，就变成了五个 Token：`int`（类型关键字）、`age`（标识符）、`=`（赋值号）、`18`（整数）、`;`（分号）。这下后面的程序就好处理多了。

在 C-- 里，Token 分成两大类。一类是**值会变的词**，比如整数、浮点数、变量名——你不能写死，因为 `18` 和 `999` 都是整数、`age` 和 `salary` 都是变量名，得用一套规则去描述"什么样的算整数、什么样的算变量名"。这套规则就是**正则表达式**，这是实验 1.1 你要亲手写的东西。另一类是**写死的符号**，比如 `+` 永远是加号、`;` 永远是分号、`if` 永远是关键字，这些一一对应，直接认就行。

### 2.2 那些值会变的词：整数、浮点数、标识符

先看**整数（INT）**。最普通的整数就是一串数字，但有个小规定：除了 `0` 本身，首位不能是 0。也就是说 `0`、`234`、`10000` 都是合法的十进制整数，但 `012` 就不算普通的十进制整数了——它会被当成**八进制**（以 0 开头）。C-- 还支持**十六进制**，以 `0x` 或 `0X` 开头，比如 `0xFF32`。课程让你假设整数都不超过 32 位，这样你处理起来不用操心超大数。

再看**浮点数（FLOAT）**。最基本的写法是"一串数字 + 一个小数点 + 一串数字"，关键是**小数点前后都得有数字**。所以 `0.7`、`12.43`、`9.00` 都是合法浮点数，但 `5.` 或 `.5` 这种小数点一边没数字的，基础规则里是不行的（不过在指数形式里另有说法）。C-- 也支持科学记数法，比如 `01.23E12` 表示 1.23×10¹²，`43.e-4` 表示 43.0×10⁻⁴。课程让你假设浮点数都符合 IEEE754 单精度标准，也就是 C 语言里的 `float`。

最后看**标识符（ID）**。标识符就是变量名、函数名这些名字。规则是：**以字母或下划线开头，后面可以跟字母、数字、下划线的任意组合**。所以 `age`、`_count`、`x1`、`total_score` 都是合法的标识符，但 `1x`（数字开头）不合法，`a-b`（中间有横杠）也不合法——横杠会被当成减号。还有一个重要规定：**保留字（`if`、`while`、`int` 这些）不能当标识符用**，你不能给变量起名叫 `if`。课程让你假设标识符长度都不超过 32 个字符。

### 2.3 那些写死的符号和关键字

除了上面三个"值会变"的词，C-- 里其它 Token 都是固定字符，看到什么就是什么。

**运算符和标点**有：分号 `;`、逗号 `,`、赋值号 `=`、加减乘除 `+ - * /`、取负 `-`、逻辑非 `!`、点 `.`、关系运算符 `> < >= <= == !=`、逻辑与 `&&`、逻辑或 `||`，还有六种括号 `(` `)` `[` `]` `{` `}`。

这里有个小窍门帮你记括号的 Token 名字：每个名字三个字母，第一个字母 L 表示"左"、R 表示"右"，第三个字母表示括号形状——P 是圆括号（Paren，就是 `()`），B 是方括号（Bracket，就是 `[]`），C 是花括号（Curly，就是 `{}`）。所以 `LP` 就是左圆括号 `(`，`RB` 就是右方括号 `]`，以此类推，六个括号的名字一下就记住了。

**关键字**就五个：`struct`、`return`、`if`、`else`、`while`，再加上两个类型名 `int` 和 `float`。这些是保留字，有固定的 Token 类型。词法分析时要**先把它们认出来，再当成普通标识符**——否则 `if` 会被误识别成一个叫 `if` 的变量名，那就乱套了。

最后还有一个细节要在这里提一句：C-- 有两种注释。一种是 `//` 开头的单行注释，从 `//` 到这一行结尾的全部内容都算注释；另一种是 `/* */` 包起来的多行注释，`/*` 和它后面最近的那个 `*/` 之间的内容都算注释。注释在词法分析这一步就被**直接丢掉**了——它只是给人看的，编译器不需要。注意 `/* */` 不能嵌套，这个后面语义约束那节会专门说为什么。

## 3. 语法层：词怎么组合成合法的程序

### 3.1 什么是"文法"，怎么读它

词法拼出了 Token，但光有一堆 Token 还不够，还得知道这些 Token 怎么排列组合才算一个合法的 C-- 程序。描述这种"怎么组合"的规则，就叫**文法**。

文法是用一种叫"产生式"的东西写的，长这样：`左边 → 右边`，意思是"`左边`这个语法单位，可以写成`右边`的样子"。右边可以有好几种选择，用竖线 `|` 隔开，表示"或者"。我直接拿 C-- 里关于"语句"的产生式来举例，你一看就懂：

```
Stmt → Exp SEMI
Stmt → RETURN Exp SEMI
Stmt → IF LP Exp RP Stmt
```

这几行的意思是：一条语句（Stmt），可以是一个"表达式加分号"，也可以是 `return 表达式;`，也可以是 `if (表达式) 语句`。换句话说，下面三行都是合法的 C-- 语句：

```c
a = 1;
return x + y;
if (n > 0) a = a + 1;
```

文法里还有两个概念顺带说一下。一种叫**终结符**，就是那些不能再拆的最小单位，其实就是上一节讲的 Token（比如 `IF`、`SEMI`、`INT`）；另一种叫**非终结符**，是能继续展开成别的东西的语法单位（比如 `Stmt`、`Exp`、`Program`）。整个文法有一个总的起点叫**初始符号**，C-- 里就是 `Program`——一个合法的 C-- 程序，从 `Program` 出发，按照产生式一层一层展开，最终全变成 Token，就得到了你的代码。

下面分六组，把 C-- 的文法从头到尾讲一遍。每组我都会配代码例子，你对着例子看规则，就不会觉得抽象。

### 3.2 程序的骨架：一串"外部定义"

一个完整的 C-- 程序长什么样？文法说：

```
Program → ExtDefList
```

`Program`（整个程序）就是一串 `ExtDef`（外部定义）。什么叫"外部定义"？就是在所有函数外面写的那些东西，主要有三种：**定义全局变量**、**定义结构体**、**定义函数**。下面三种都是合法的"外部定义"：

```c
int global1, global2;              // 全局变量定义
struct Point { int x; int y; };    // 结构体定义
int add(int a, int b) { ... }      // 函数定义
```

对应到文法上，就是这三条产生式：

```
ExtDef → Specifier ExtDecList SEMI        // 全局变量：类型 + 变量列表 + 分号
ExtDef → Specifier SEMI                   // 结构体定义：类型 + 分号
ExtDef → Specifier FunDec CompSt          // 函数定义：返回类型 + 函数头 + 函数体
```

这里 `Specifier` 是"类型"，`ExtDecList` 是"变量列表"，`FunDec` 是"函数头"，`CompSt` 是"函数体"。这些名字后面会一一讲到。你只要先记住：**一个 C-- 程序，就是一串全局变量定义、结构体定义、函数定义，一个接一个排下去。**

顺带说一个文法里反复出现的小套路。你会看到很多写成 `xxList → xx xxList | 空` 的产生式，意思是"一个列表，可以是'一个 xx 接着一个 xxList'，也可以是空的"。说白了就是"零个或多个 xx"。比如程序里可以定义零个、一个或多个函数，就是用这个套路表达的。看习惯了就好。

### 3.3 类型：除了 int/float，还有结构体

C-- 的"类型"有两种来源。最简单的是直接写 `int` 或 `float`，这叫基本类型。另一种是结构体。结构体在 C-- 里有两种用法，我用例子区分：

第一种是**定义一个结构体**，比如：

```c
struct Complex {
    int real, image;
};
```

这相当于"我发明了一种新类型，名字叫 Complex，里面有 real 和 image 两个成员"。这里 `Complex` 这个名字其实可以省略，写成 `struct { int real, image; };` 也合法，这叫匿名结构体。

第二种是**复用一个已经定义过的结构体**，比如上面定义过 Complex 之后，后面就可以直接用它来声明变量：

```c
struct Complex a, b;
```

这里不用再重新写一遍成员，只要指明"我用之前那个 Complex"就行。注意这两种用法的区别：第一种带 `{ 成员 }`，是"定义"；第二种只有名字没有 `{}`，是"复用"。文法就是这样区分的。

### 3.4 变量和函数长什么样

这一组讲的是"一个变量或函数的形状"。先看变量。一个变量可能就是个普通名字，比如 `a`；也可能是数组，比如 `a[10]` 或者二维数组 `a[10][2]`。文法写成：

```
VarDec → ID
VarDec → VarDec LB INT RB
```

第一条说"变量就是一个标识符"，第二条说"变量后面可以跟一个 `[整数]`，而且可以反复加"，所以数组就这么表达出来了。

再看函数。一个函数的"头部"由函数名和形参列表组成，比如 `foo(int x, float y)`。文法是：

```
FunDec → ID LP VarList RP      // 函数名(形参列表)
FunDec → ID LP RP              // 函数名()，没有参数
```

里面的 `VarList` 就是形参列表，由一个或多个形参组成，每个形参又是一个"类型 + 变量"，比如 `int x`、`float y[10]`。所以一个完整的函数头 `foo(int x, float y[10])`，就是用这条规则一层层拼出来的：函数名 `foo` + `(` + 形参列表（`int x` 和 `float y[10]`，中间用逗号隔开）+ `)`。

### 3.5 语句：if、while、return，就这几种

C-- 语句块（用花括号 `{}` 括起来的那段）的内部结构是固定的：**先定义局部变量，再写一串语句**。也就是说，变量必须在一个语句块的开头集中定义好，不能写到一半突然插一个 `int c;` 进来。这是 C-- 的硬规定，后面语义约束会再强调。

语句本身有五种形态。我直接列出来，配上例子你就明白了：

```c
x = a + b;          // Exp SEMI：一个表达式加分号，是最常见的语句
{ int t; t = 1; }   // CompSt：一个嵌套的语句块
return result;      // RETURN Exp SEMI：返回语句
if (score >= 60) pass();                 // IF LP Exp RP Stmt：if 语句
if (score >= 60) pass(); else fail();    // IF LP Exp RP Stmt ELSE Stmt：if-else
while (i < n) i = i + 1;                 // WHILE LP Exp RP Stmt：while 循环
```

注意 C-- **没有 `for`，没有 `switch`，没有 `do-while`**。想写循环，只有 `while`。这一点要记住，别下意识以为有 `for`。

### 3.6 表达式：能算出一个值的东西

表达式是 C-- 里最灵活的部分。简单说，表达式就是"能算出一个值的东西"。它可以是：

- 一个变量或常量：`a`、`18`、`3.14`
- 各种运算：`a + b`、`x * y`、`a > b`、`a && b`、`a = 1`
- 加括号改变运算顺序：`(a + b)`
- 取负或逻辑非：`-a`、`!flag`
- 函数调用：`add(1, 2)` 或 `add()`
- 数组访问：`arr[i]`
- 结构体成员访问：`point.x`

这些都能"算出一个值"，所以都是表达式。文法把所有这些形态都列成 `Exp → ...` 的产生式。你不用背，只要知道：**凡是能出现在等号右边、或能参与运算的东西，都是表达式。**

现在有个问题：当一个表达式里同时出现好几个运算符，比如 `1 + 2 * 3`，到底先算加还是先算乘？再比如 `a = b = 1`，是先算左边的 `a = b` 还是先算右边的 `b = 1`？这就需要规定**优先级**和**结合性**了，下一小节专门讲。

### 3.7 优先级和结合性：消除"先算谁"的歧义

C-- 给了一张表，规定了所有运算符的优先级和结合性。优先级高的先算，优先级相同的看结合性。这张表你不用硬背，需要的时候查就行：

| 优先级（数字越小越优先） | 运算符 | 结合性 |
|---|---|---|
| 1（最高） | `( )` `[ ]` `.` | 左结合 |
| 2 | 取负 `-`、逻辑非 `!` | 右结合 |
| 3 | `*` `/` | 左结合 |
| 4 | `+` `-` | 左结合 |
| 5 | `<` `<=` `>` `>=` `==` `!=` | 左结合 |
| 6 | `&&` | 左结合 |
| 7 | `\|\|` | 左结合 |
| 8（最低） | `=` | 右结合 |

举个例子，`1 + 2 * 3` 为什么等于 7 而不是 9？因为 `*`（优先级 3）比 `+`（优先级 4）优先级高，所以先算 `2 * 3 = 6`，再算 `1 + 6 = 7`。

再举一个结合性的例子，`a = b = 1`。`=` 是右结合的，意思是"从右往左算"，所以先算右边的 `b = 1`（把 1 赋给 b），再算 `a = (b = 1)`（把刚才的结果赋给 a）。如果是左结合，就会先算 `a = b`，那是另一回事了。

这张表后面写 Bison 语法分析器时会用到，你得用特定的写法（`%left`、`%right`）把优先级告诉分析器。到时候我会再带你做。

## 4. 语义约束层：那些"合语法但有潜规则"的地方

这是最容易被忽略、也最容易让你写错编译器的一层。前面讲的文法产生式，描述的是"结构上能不能组合出来"；但有些写法结构上明明组合得出来，却涉及"到底算不算错、要不要报错"的问题，这些答案藏在附录 A 的"补充说明"里，不在产生式本身。我用几个具体的例子来讲。

**第一个例子：`int;` 这样的空定义。** 你看到 `int;` 这种"只有类型、什么变量都没定义"的语句，第一反应肯定是"这是错的吧"。但 C-- 明确告诉你：**它是合法的，不要报错**。为什么？因为标准 C 语言里这种写法本来就是允许的，课程为了和 C 保持一致，也允许它。如果你的编译器在这里报了错，反而是你的 bug。

**第二个例子：变量必须在语句块开头定义。** 前面提过，C-- 的语句块（花括号里的内容）固定是"先变量定义、后语句"的顺序。也就是说，下面这样是**不允许**的：

```c
{
    int a = 1;
    a = a + 1;
    int b = 2;      // 错！语句已经开始写了，不能再插变量定义
    b = b + a;
}
```

你要定义变量，得在一进花括号就集中定义，定义完了再开始写语句。这个限制其实是从文法产生式（`CompSt → { DefList StmtList }`）来的，但很多人不细看产生式，会以为变量能随便插，结果写出 C-- 不支持的代码。

**第三个例子：注释不能嵌套。** 你可能会写这样的注释：

```
/* 外层注释 /* 里层注释 */ 外层继续 */
```

看起来挺合理，但 C-- 规定 `/* */` **不允许嵌套**。实际情况是：词法分析器看到第一个 `/*` 开始算注释，然后看到第一个 `*/` 就认为注释结束了——哪怕这个 `*/` 本来是你想结束"里层注释"的。于是后面那句"外层继续 */"就成了孤零零的代码，报错。这是 C 系语言的通病，C-- 也一样。所以词法分析时遇到嵌套注释，要能识别并报错。

**第四个例子：整数和标识符的规模限制。** 课程让你假设整数都不超过 32 位、浮点都符合单精度标准、标识符长度都不超过 32 个字符。这些不是"硬性语法"，而是"你可以放心做的假设"——意味着你写词法分析器时，不用费劲去处理超长整数或超长变量名，能简化不少工作。

这些"潜规则"单独拎出来讲，是因为它们决定了你的编译器**什么时候该报错、什么时候不该报错**。把这一层吃透，你写出来的编译器才不会"该报的不报、不该报的乱报"。

## 5. 把分析结果用到实验：Flex 和 Bison

前面四节是"分析语言本身"，跟怎么实现无关。这一节讲怎么把这些分析变成能跑的代码——也就是接下来的两次实验。

**实验 1.1（词法分析）用的工具叫 Flex**，它对应的是第 2 节的词法分析。你要做的事情，就是把第 2 节里讲的那几条词法规则，用一种叫"正则表达式"的写法告诉 Flex，Flex 就会自动帮你生成一个能做词法分析的 C 程序。比如你告诉它"整数长这样：一串数字、首位非 0"，它生成的程序就能识别整数。这里有一个特别重要的细节：**多字符的运算符要先于单字符的写**。比如 `==` 必须写在 `=` 之前，否则看到 `==` 时可能先被当成两个 `=`。关键字也要先于普通标识符写，否则 `if` 会被当成变量名。

**实验 1.2（语法分析）用的工具叫 Bison**，它对应的是第 3 节的文法。你要做的事情，就是把第 3 节里那些产生式，翻译成 Bison 能看懂的格式告诉它，Bison 就会帮你生成一个能做语法分析的 C 程序。3.7 节那张优先级表也要告诉 Bison（用 `%left`、`%right` 这些写法）。Bison 的工作方式是：它需要下一个词时，就去调用 Flex 提供的 `yylex()` 函数，Flex 从文件里读字符、拼出一个词返回给它。所以 Flex 和 Bison 是搭档关系——**Bison 是主，Flex 是辅**，数据从字符流变成 Token 流、再变成语法判断，是一条流水线。

关于第 4 节那些语义约束，要分清楚它们各自在哪个阶段处理。`int;` 合法、变量要在块首定义，这些是**语法阶段**就该处理好的（文法产生式本身就能保证）。注释嵌套要报错，是在**词法阶段**就要处理的（注释在词法阶段就被吃掉了）。而像"类型对不对、变量有没有定义过"这种，要等到后面的**实验 2（语义分析）**才处理，实验 1.2 不用管。把这个分工搞清楚，你就知道每个规则该在哪一步实现，不会乱。

最后说一下两次实验的产出。实验 1.1 做完，你的程序能读入一个 `.cmm` 文件，把每个词的类型和值打印出来。实验 1.2 做完，你的程序能读入一段代码，告诉你"语法对"还是"语法错"——这个阶段**只判断对错**，不搭语法树、不算值，那些是后面阶段的事。

## 6. 整体流程：字符怎么一步步变成判断结果

最后我们把整条线串起来看一遍，这对理解 Flex 和 Bison 怎么配合很关键。

你写好的 C-- 源代码（比如 `test.cmm`），本质上是一串字符。Flex 先登场，它读入这些字符，按照你写的词法规则，把它们拼成一个一个 Token，比如 `int` `age` `=` `18` `;`。接着 Bison 登场，它从 Flex 那里一个一个地要 Token，按照你写的文法规则，判断这些 Token 能不能组合成一个合法的程序——能就通过，不能就报语法错误。第 4 节那些约束，就像流水线上的质检规则，在相应的环节发挥作用。再往后（语义分析、中间代码、目标代码），那是后面几个实验的事了，现在不用管。

一句话总结这条流水线：**字符 →（Flex）→ Token →（Bison）→ 语法判断**。Flex 负责"拼词"，Bison 负责"判句"，两者一前一后，就完成了编译的前两道工序。

## 7. 怎么验证自己学懂了

光看文档不够，建议你做几件小事来检验理解：

第一，**对照原文**。打开 `Appendix.pdf` 附录 A，把这份文档和原文对一遍，看看有没有我讲漏的规则，也顺便确认我讲的对不对——附录 A 才是唯一权威。

第二，**自己当一次词法分析器**。打开 `Tests/inputs/` 里的几个 `.cmm` 测试文件，挑一行代码，试着用第 2 节的规则，把它一个词一个词地拼出来，标出每个词是什么类型。能顺畅做完，说明词法层你懂了。

第三，**自己当一次语法分析器**。拿一句简单的代码，比如 `int main() { return 0; }`，试着用第 3 节的产生式，从 `Program` 一层一层往下展开，看看能不能"展开"成这句代码。能展开，说明语法层你懂了。

第四，**专门想想边界情况**。比如 `int;` 算不算错？`a[10][2]` 合法吗？注释嵌套会怎样？对照第 4 节确认你的判断，能答对，说明语义约束层你也吃透了。
