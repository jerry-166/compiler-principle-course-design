### 0.2 测试数据不一致说明（非代码问题）

本项目 `.gitattributes` 强制 LF 换行，但 Tests1 有 3 个 .exp 文件违反了这一约定：

| 用例                   | 问题                                       | 验证                                                                                          |
| -------------------- | ---------------------------------------- | ------------------------------------------------------------------------------------------- |
| **B-1.exp, B-2.exp** | CRLF 换行（`\r\n`），其他 21 个 .exp 都是 LF（`\n`） | `file` 命令显示 "with CRLF line terminators"；`tr -d '\r'` 后 diff 通过 |
| **A-10.exp**         | 末尾多 1 个 `\n`，A-1~A-9 都没有                 | `od -c` 末尾是 `.\n`，其他错误 .exp 末尾是 `.`                                                         |

这 3 个用例的 parser 输出**内容完全正确**，仅因测试数据的格式问题在严格 byte-by-byte 比对下判 FAIL。处理方式（用户决定）：**保留测试数据不动**，在此记录为已知问题。

另有 3 个用例（E1-2, E2-2, E3-2）仓库从未提供 .exp 文件，无法比对。

### 2.4 D 类用例（合法建树）

#### D-1 — 十六进制/八进制整数 ✅ STRICT PASS

输入：

```c
int main() {
  int a = 0xFF00;    // → INT: 65280
  int b = 0x0F0F;    // → INT: 3855
  int c = 0644;      // → INT: 420
  int d = 0x3A;      // → INT: 58
  return 0;
}
```

进制转换正确，INT 全部按十进制打印，与 D-1.exp 一致。

#### D-2 — 科学计数法浮点（float32 精度）✅ STRICT PASS

输入：

```c
int main() {
  float p = 3.14e0;    // → FLOAT: 3.140000
  float q = 6.02e23;   // → FLOAT: 602000017271895229464576.000000
  float r = 1.38e-2;   // → FLOAT: 0.013800
  float s = 2.72E1;    // → FLOAT: 27.200001
  return 0;
}
```

`6.02e23` 输出 `602000017271895229464576.000000`——这是 **float32（单精度）** 的精度特征，必须用 `strtof` 而非 `strtod` 才能精确匹配（详见 [`语法分析.md`](语法分析.md) 踩坑 2）。与 D-2.exp 逐字节一致。

#### D-3 — 注释穿插 ✅ STRICT PASS

输入在表达式中间穿插注释：

```c
int x /* counter */ = 0;
x = x + 1;
if /* check */ (x > 0) { return x; }
```

注释被正确丢弃，语法树结构与无注释时一致。与 D-3.exp 逐字节一致。

### 2.5 E 类用例

#### E1-1 — 地址范围检查 ✅ STRICT PASS

输入：`check_range(addr)` 用十六进制（`0x0000`/`0xFFFF`）和八进制（`0755`）做边界判断。完整语法树与 E1-1.exp 逐字节一致。进制转换：`0755`→`493`、`0xFFFF`→`65535`。

#### E1-2 — 非法十六进制/八进制 ⚠️ 无 .exp（仓库未提供）

输入第 3、4 行 `0xQQQ`、`099`：

```
Error type A at Line 3: Undefined character in number "0xQQQ".
Error type A at Line 4: Undefined character in number "099".
```

#### E2-1 — 物理常量计算 ✅ STRICT PASS

输入：质量/速度/重力/能量等浮点常量运算。完整语法树与 E2-1.exp 逐字节一致。浮点值：`9.1e-2`→`0.091000`、`2.99e8`→`299000000.000000`。

#### E2-2 — 指数浮点缺尾数 ⚠️ 无 .exp（仓库未提供）

输入第 3、4 行 `2.5e`、`1.0E+`：

```

Error type A at Line 4: Undefined character in number "1.0E".
```

#### E3-1 — 矩阵初始化 ✅ STRICT PASS

输入：二维数组 `mat[3][3]`，双重 while 循环初始化。注意程序前有两行块注释，`Program (4)` 的行号取自首个实际代码行（注释不计）。完整语法树与 E3-1.exp 逐字节一致。

#### E3-2 — 嵌套注释 ⚠️ 无 .exp（仓库未提供）

输入第 1 行 `/*/* 嵌套注释不被支持，此处为错误示例 */*/`：

C-- 不支持嵌套注释。lexer 从第一个 `/*` 读到第一个 `*/` 闭合，吃掉 `/*/* ... */`，剩下的 `*/` 被解析为 STAR + DIV，导致后续无法构成合法程序。parser 输出为空（无树、无错误信息——因为剩余 Token 流既不构成完整程序也未命中错误产生式）。仓库未提供 E3-2.exp。

## 3. 复现方法

### 3.1 单个用例

```bash
./C/bin/parser Tests/Tests1/inputs/B-1.cmm
```

### 3.2 全量验证

```bash
bash C/scripts/verify.sh          # 内容比对（容许 CRLF），日常用这个
bash C/scripts/verify_strict.sh   # 严格逐字节比对
```

`verify.sh` 的输出状态：

| 状态        | 含义                                |
| --------- | --------------------------------- |
| `PASS`    | 内容与 .exp 完全一致（已容许 CRLF）           |
| `KNOWN`   | 内容正确，但 .exp 有格式问题（CRLF / 末尾 `\n`） |
| `MISSING` | 仓库未提供该用例的 .exp                    |
| `FAIL`    | 内容不匹配（当前为 0）                      |

### 3.3 字节级核对（针对 B-1/B-2/A-10）

```bash
# 确认 B-1.exp 是 CRLF
file Tests/Tests1/expects/B-1.exp
# 剥掉 CR 后比对（应无差异）
diff <(tr -d '\r' < Tests/Tests1/expects/B-1.exp) <(./C/bin/parser Tests/Tests1/inputs/B-1.cmm)

# 确认 A-10.exp 末尾多 \n
tail -c 5 Tests/Tests1/expects/A-10.exp | od -c
```
