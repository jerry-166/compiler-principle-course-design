## 0. 一句话先说明白

**语法分析**就是把词法分析器吐出来的一串 Token，按"语法规则"组装成一棵**语法树**（Syntax Tree），并验证这串 Token 是不是一段合法的程序。

打个比方：词法分析把 `int main ( ) { return 0 ; }` 切成了 9 个 Token。但光有 Token 还不够——你还想知道"这是一段函数定义吗？返回什么类型？里面有几条语句？"。**语法分析的工作就是回答这些问题**，方式是<u>按一套预先写好的"语法规则"（叫做**文法**）去匹配 Token 流，匹配成功就建出一棵树，匹配失败就报错</u>。

这棵树是后续所有阶段（语义分析、中间代码生成……）的**承重墙**：编译器后面不再看源代码本身，而是看这棵树。

## 1. 为什么（Why）

### 1.1 为什么需要单独一个语法分析阶段？

回想词法分析：它只管"切词"，每个 Token 都是孤立的。但程序是一个**有结构**的东西——`if (...) {...}` 里的语句属于这个 if，外层函数的形参列表有几个参数、是什么类型……这些"谁属于谁、谁嵌套在谁里面"的信息，词法分析器给不了。

**语法分析的价值**：

- 把扁平的 Token 流"折叠"成层次化的树，让"包含关系"显式化。
- 顺便检查源代码**是不是合法**——`int = ; main` 这种乱序写法，词法分析能切出 4 个合法 Token，但语法分析会立刻发现"等号左边不能是类型名"。

这是编译器"分阶段"思想的第二层：词法层负责"切"，语法层负责"组装 + 检查结构"。

### 1.2 为什么不手写而要用 Bison？

理论上语法分析可以手写——按文法规则写一堆 if/else/递归函数就行（PL/0 编译器就是手写的递归下降）。但 C-- 文法有 30+ 条产生式，手写要处理大量细节：

- 看到哪个 Token 应该走哪条规则
- 表达式优先级（`a + b * c` 应该先算 `*`）
- 出错时怎么恢复（不能因为一个错误就放弃整个文件）
- 怎么把每条规则的"成果"组装成树

**Bison**（GNU 版的 yacc）就是干这事的工具：你给它一份**文法描述**（.y 文件），它自动生成一个 C 函数 `yyparse()`，里面实现了<u> LALR(1) 解析算法——上面那些细节它全包</u>。你只需要：

1. <u>用类似 BNF 的语法写文法规则；</u>
2. <u>在每条规则后面写一段 C 代码（叫 **action**），告诉它"匹配到这条规则时，怎么用子节点的成果拼出父节点"。</u>

工具干苦力，你专注业务。这就是为什么课程要求用 Bison。

### 1.3 1.2 阶段要把 1.1 的词法分析器改造

1.1 的词法分析器是**独立程序**：有自己的 `main`，用 `printf` 直接打印每个 Token。1.2 里它要变成**被 Bison 调用的子模块**——Bison 生成的 `yyparse()` 会反复调用 `yylex()` 拿 Token，词法分析器不再"自己打印"，而是把 Token 编号 `return` 给 Bison，并把语义值（INT 的数值、ID 的字符串等）通过 `yylval` 这个全局变量传过去。

**一句话**：词法分析器从"自己说"变成"被问就答"。改造细节见本文 3.1 节。

## 2. 是什么（What）

这一节讲清楚做语法分析必须懂的概念。先讲文法和语法树，再讲 Bison 工具。

### 2.1 文法（Grammar）：用规则描述语言

**文法**是一组"替换规则"，描述一种语言里所有合法的写法。每条规则长这样：

```
左部 → 右部
```

意思是"左部这个东西，可以用右部这一串替代"。例如 C-- 的一条规则：

```
Stmt → RETURN Exp SEMI
```

读作："语句（Stmt）可以写成 `RETURN exp SEMI`"——也就是 `return <表达式>;` 这种形式。这里 `RETURN` 和 `SEMI` 是终结符（Token，再不能展开），`Exp` 是非终结符（还能继续展开，比如 `Exp → INT`、`Exp → Exp PLUS Exp`）。

C-- 完整文法见 [`资料/.../Appendix.pdf`](../资料/现课程要求(AICanUse)/Appendix.pdf) 附录 A。本阶段把它们几乎一对一翻译成 Bison 的产生式。

### 2.2 语法树（Syntax Tree）：规则的"执行结果"

光有文法不够——文法只回答"合不合法"。<u>但编译器后面要用程序的**结构**做语义分析、生成代码。这个结构就是语法树</u>。

**语法树的节点分两类**：

1. **非终结符节点**：对应文法里的非终结符（`Program`、`ExtDef`、`Stmt`、`Exp` 等）。打印格式：`Name (lineno)`。
2. **终结符节点**（Token）：对应 Token（`INT`、`ID`、`SEMI` 等）。打印格式：
   - 有值的：`TOKEN: value`（如 `INT: 65280`、`ID: main`、`TYPE: int`）
   - 无值的：`TOKEN`（如 `SEMI`、`LP`、`IF`）

举例，源代码 `int main(){return 0;}` 的语法树（缩进 2 空格）：

```
Program (1)
  ExtDefList (1)
    ExtDef (1)
      Specifier (1)
        TYPE: int
      FunDec (1)
        ID: main
        LP
        RP
      CompSt (1)
        LC
        StmtList (2)
          Stmt (2)
            RETURN
            Exp (2)
              INT: 0
            SEMI
        RC
    ExtDefList (12)
```

每个非终结符节点后面的 `(N)` 是**行号**，规则是：**取该节点第一个子节点的行号**（如 `Program (1)` 取自第一个 `ExtDef` 的行号 1）。这个规则是 Tests1 测试集定的，行号怎么实现见 3.4 节。

### 2.3 LALR(1) 解析：Bison 用的算法

Bison 默认用 **LALR(1)** 算法解析（Look-Ahead LR，1 个向前看符号）。原理不展开，只说要点：

- Bison 从左到右读 Token，维护一个**状态栈**。
- 每读一个 Token，它要决定两件事之一：
  - **移进（Shift）**：把这个 Token 压栈，等更多信息。
  - **归约（Reduce）**：栈顶的几个符号正好匹配某条规则的右部，把它们"折叠"成左部。
- 当栈顶是 `Program` 且 Token 流耗尽时，解析成功。

我们写 .y 文件时**不用关心算法**——Bison 自动算。但要关心两个工程问题：

- **优先级冲突**：`a + b * c` 应该先归约 `+` 还是先归约 `*`？Bison 默认行为不一定对，需要用 `%left`/`%right` 显式声明优先级（见 3.3 节）。
- **错误恢复**：遇到错误 Token 时怎么不放弃整个文件，跳过一些 Token 继续解析？用 Bison 的 `error` 伪终结符（见 3.5 节）。

### 2.4 yylval：词法分析器 → 语法分析器的"数据通道"

`yylex()` 返回的 Token 编号只是个"标签"（比如 `INT = 258`）。但语法分析器还要知道这个 INT 的**具体数值**是多少、出现在**第几行**。这些"语义值"通过一个全局变量 `yylval` 传递。

`yylval` 的类型由 .y 文件里的 `%union` 声明决定。本阶段的设计：

```c
%union {
    struct TokVal {
        long   ival;    /* INT 的十进制值 */
        double fval;    /* FLOAT 的 double 值 */
        char  *sval;    /* ID/TYPE 的词素字符串 */
        int    line;    /* 该 Token 出现的行号 */
    } tok;
    Node *node;         /* 非终结符的语法树节点指针 */
}
```

每个终结符 Token 都用 `<tok>` 类型，每个非终结符用 `<node>` 类型。这样 Bison 在归约时能正确地把 `$1`/`$2`（每个右部符号的语义值）当对应类型用。

**为什么要专门塞个 `line` 字段？** 因为 Bison 默认不维护每个 Token 的行号（它的 `@n.first_line` 机制依赖 flex 维护 `yylloc`，而 flex 默认不做）。我们用 `YY_USER_ACTION` 宏让 flex 在每次匹配前把 `yylineno` 写进 `yylval.tok.line`，这样每个 Token 都自带行号，建树时随时可取。这是本阶段踩的一个关键坑，详见 3.4 节。

## 3. 怎么做（How）

### 3.1 改造 lexer.l：从独立程序 → Bison 子模块

[`C/src/lex/lexer.l`](../C/src/lex/lexer.l) 的改动（相对 1.1）：

1. **包含 Bison 头文件**：`#include "node.h"` 和 `#include "parser.tab.h"`。注意顺序——`node.h` 必须在前，因为 `%union` 用到了 `Node` 类型。
2. **删 main/yywrap**，加 `%option noyywrap`。
3. **action 改为 `return TOKEN`**：如 `";" { return SEMI; }`，INT 用 `yylval.tok.ival = TO_LONG(16)`，ID/TYPE 用 `yylval.tok.sval = strdup(yytext)`。
4. **`YY_USER_ACTION` 统一设行号**（关键设计）：`#define YY_USER_ACTION yylval.tok.line = yylineno;`，flex 每次匹配前自动调用，所有 Token 行号自动正确。
5. **type A 词法错误用 `out()` 不用 `printf`，且不 return**：错误 Token 被丢弃、扫描继续，语法分析看到的还是干净 Token 流。
6. **新增复合赋值 `+=`/`-=`/`*=`/`/=`**：整体识别成 `COMPOUND_ASSIGN`，让语法层报 "Unsupported compound assignment"。
7. **新增两类非法词素整体识别**（数字开头后跟字母如 `2x`、多个小数点如 `3.14.15`）：靠 flex 最长匹配优先整体报错。

### 3.2 写 node.h + tree.c：语法树节点和打印

[`C/src/syntax/node.h`](../C/src/syntax/node.h) 定义节点结构：

```c
typedef struct Node {
    char *name;          /* 节点名（"Program" 或 Token 名） */
    int   lineno;        /* 行号：终结符=出现行；非终结符=首个子节点行号 */
    int   is_token;      /* 1=终结符，0=非终结符 */
    int   val_kind;      /* 终结符的值类型：NO_VAL/VAL_INT/VAL_FLT/VAL_STR */
    long ival; double fval; char *sval;   /* INT/FLOAT/ID 的值 */
    struct Node *children[MAX_CHILDREN];
    int   nchild;
} Node;
```

**关键 API**：

- `newNode(name)`：新建非终结符节点。
- `newTokenNode(...)`：新建终结符节点。
- `addChild(parent, child)`：把 child 挂到 parent 下。**首个被挂的子节点决定 parent 的 lineno**——这就是 Tests1 "非终结符行号取自第一个子节点"规则的实现。

[`C/src/syntax/tree.c`](../C/src/syntax/tree.c) 的打印逻辑用递归按 2 空格缩进：终结符按 `val_kind` 分四种格式（`INT: 65280` / `FLOAT: 3.140000` / `ID: main` / `SEMI`），非终结符打印 `Name (lineno)` 后递归子节点。

**输出缓冲（踩坑后改的设计）**：所有输出（树、错误信息）先进 `open_memstream` 内存流，最后在 `flush_output` 时统一写到 stdout。这样能控制"错误用例末尾不带 `\n`、树用例末尾带 `\n`"——Tests1 的 .exp 文件就是这种不一致格式。详见第 5 节踩坑。

### 3.3 写 parser.y 主干：合法程序建树

[`C/src/syntax/parser.y`](../C/src/syntax/parser.y) 的三段式结构：

**第 1 段：声明** —— `%union`（见 2.4）、`%token <tok> ...`（终结符）、`%type <node> ...`（非终结符）、优先级声明（**从低到高**，赋值最低）：

```c
%right ASSIGNOP                  /* = 右结合 */
%left  OR AND RELOP              /* ||  &&  关系运算 */
%left  PLUS MINUS                /* + - */
%left  STAR DIV                  /* * / */
%right NOT UMINUS                /* 一元 ! 和一元负号 */
%left  LP RP LB RB DOT           /* 括号、点号最高 */
```

`UMINUS` 是伪终结符，配合 `%prec UMINUS` 提升 `MINUS Exp`（一元负号）的优先级。这套表源自 Appendix A 的运算符优先级表——比如 `a + b * c`，`STAR` 比 `PLUS` 高，Bison 会先归约 `b * c`，对应"先乘后加"。

**第 2 段：规则** —— 每条产生式形如：

```c
Program : ExtDefList   { root = NT("Program"); addChild(root, $1); }
        ;
```

`$1` 指右部第 1 个符号的语义值（这里是 `ExtDefList` 的 Node 指针）。`{...}` 是 action——告诉 Bison 归约这条规则时怎么建节点。两个简化宏 `TKS(...)` 建终结符节点、`NT(...)` 建非终结符节点，让 action 写起来短。空产生式（`ExtDefList → 空` 等）action 直接 `$$ = NULL;`，`addChild` 跳过 NULL——纯空产生式不产生节点。

**第 3 段：用户代码** —— 主要是 `main`：

```c
int main(int argc, char **argv) {
    if (argc > 1) { yyin = fopen(argv[1], "r"); }
    yyparse();
    if (!has_error && root) {
        printTree(root);
        flush_output(0);   /* 树末尾保留 \n */
    } else {
        flush_output(1);   /* 错误末尾去掉 \n */
    }
}
```

`has_error` 是全局标志——任何 type B 错误都置 1，置位后整棵树都不打印（符合 Tests1 规则："遇到语法错误只打印错误信息，不打印树"）。

### 3.4 行号传播的踩坑历程

**踩坑**：最初用 Bison 的 `%locations` + `@N.first_line` 取行号——结果**所有节点都显示行号 1**。

**原因**：flex 默认不维护 `yylloc`（Bison 的位置结构），需要手写 `YY_USER_ACTION` 显式维护。但维护 `yylloc` 还要算列号，太麻烦。

**解决方案**：放弃 `%locations`，把行号塞进 `%union` 的 `TokVal` 结构体。flex 用 `YY_USER_ACTION` 把 `yylineno` 写进 `yylval.tok.line`，这样每个终结符 Token 都自带行号。建非终结符节点时，`addChild` 自动从**第一个非空子节点**继承行号。两层机制叠加，整棵树的行号都正确。

### 3.5 B 类精确错误产生式（核心难点）

Tests1 要求**精确错误信息**（不是通用的 "Syntax error"）。做法是为每种错误写专门的"错误产生式"，让 Bison 在匹配到错误模式时直接走对应规则、打印精确信息。

完整的 18 条错误清单（每条的触发用例 + 产生式 + 代码位置）见 [`测试记录.md`](测试记录.md) 的"错误信息清单"一节。这里只讲两个**非显然的设计**——它们是本阶段最绕的地方：

**设计一：尾逗号的 LALR shift/reduce 冲突**

形参列表尾逗号 `int f(int a,)` 看起来只要加一条规则 `FunDec → ID LP VarList COMMA RP` 就能抓到。但 Bison 实际解析时，看到 `ParamDec COMMA` 会面临两个选择：

- **shift**：把 COMMA 压栈，准备走"尾逗号"分支
- **reduce**：先把 `ParamDec` 归约成 `VarList`，再看后面

LALR 默认行为是 **reduce**，结果尾逗号分支**永远进不去**，错误抓不到。

**解决**：额外加一条"不递归 VarList"的变体规则 `FunDec → ID LP ParamDec COMMA RP`（单参数版）。这样 look-ahead 是 COMMA 时，Bison 会 shift 进这条变体，错误被精确捕获。函数调用实参的尾逗号 `add(3,)` 同理处理。详见 [parser.y:179-186](../C/src/syntax/parser.y)。

**设计二：匿名结构体用标志位传递状态**

`struct { int tag; };` 这种匿名结构体声明，`OptTag` 是空产生式。单看一条规则无法识别"匿名 + 用作声明"——因为 `StructSpecifier` 归约时还不知道外层是声明还是定义。

**解决**：在 `StructSpecifier` 的 action 里设置标志 `last_struct_anonymous = (OptTag 为 NULL)`，在外层 `ExtDef` 或 `Def` 的 `Specifier SEMI` 分支检查这个标志位。检查完立刻清零。详见 [parser.y:99-107](../C/src/syntax/parser.y)。

也就是说，本阶段把语法树建正确，是后续 4 个阶段的承重墙——树错了后面全错。所以**本阶段的精确 diff 通过率至关重要**。

## 5. 踩坑记录

1. **行号传播是最隐蔽的坑**：默认以为 Bison 的 `%locations` 会自动处理，结果所有节点都显示行号 1。**教训**：用工具时不能假设它的"默认行为"符合预期——要先跑通最小用例验证关键属性（如行号），再扩展。详见 3.4。

2. **FLOAT 是单精度不是 double**：`6.02e23` 期望 `602000017271895229464576.000000`。用 `atof`/`strtod`（double）得到 `601999...`，**对不上**；`strtold`（long double）也不对；最终用 `strtof`（float32）→ double 提升后才精确匹配。说明 **C-- 的 FLOAT 是 IEEE754 单精度**。详见 [lexer.l:127-132](../C/src/lex/lexer.l)。

3. **错误产生式不是"加一条规则"那么简单**：Bison 的 LALR 算法有 shift/reduce 冲突，光写规则没用，得理解 look-ahead 行为才能让错误分支真正命中。尾逗号就是典型——必须额外写一条"不递归 VarList"的变体规则。详见 3.5。

4. **输出缓冲解决格式不一致**：Tests1 树用例 .exp 末尾带 `\n`、错误用例不带。直接 `printf` 无法事后区分。用 `open_memstream` 收集所有输出、最后 `flush_output(strip)` 统一控制，干净利落。

5. **测试数据本身可能不规范**：B-1.exp/B-2.exp 是 CRLF（违反项目 `.gitattributes` 的 LF 约束），A-10.exp 末尾多 `\n`（与其他错误用例不一致），E1-2/E2-2/E3-2 仓库从未提供 .exp。**遇到对不上时先检查测试数据**，别一上来就怀疑自己代码。详见 [`测试记录.md`](测试记录.md)。

6. **Bison `.y` 要兼容 3.5.1**（协作约束）：本地是 Bison 3.8.2，同伴真机是 3.5.1。代码不用 3.8 才有的特性（如某些 `%define` 选项）。`%union` 用结构体而非 `%define api.value.type`，3.5.1 也支持。
