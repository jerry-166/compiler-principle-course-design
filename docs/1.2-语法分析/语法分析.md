# 语法分析（实践 1.2）—— 为什么、是什么、怎么做

> 配套代码：`C/src/syntax/parser.y`（Bison 文法）、`C/src/syntax/tree.c` + `node.h`（语法树）、`C/src/lex/lexer.l`（改造后的词法分析器）
> 适用阶段：C-- 编译器第 2 阶段（语法分析）
> 阅读前提：建议先读 [`词法分析.md`](词法分析.md)，了解 Token 是什么。

---

## 0. 一句话先说明白

**语法分析**就是把词法分析器吐出来的一串 Token，按"语法规则"组装成一棵**语法树**（Syntax Tree），并验证这串 Token 是不是一段合法的程序。

打个比方：词法分析把 `int main ( ) { return 0 ; }` 切成了 9 个 Token。但光有 Token 还不够——你还想知道"这是一段函数定义吗？返回什么类型？里面有几条语句？"。**语法分析的工作就是回答这些问题**，方式是按一套预先写好的"语法规则"（叫做**文法**）去匹配 Token 流，匹配成功就建出一棵树，匹配失败就报错。

这棵树是后续所有阶段（语义分析、中间代码生成……）的**承重墙**：编译器后面不再看源代码本身，而是看这棵树。

---

## 1. 为什么（Why）

### 1.1 为什么需要单独一个语法分析阶段？

回想词法分析：它只管"切词"，每个 Token 都是孤立的。但程序是一个**有结构**的东西——`if (...) {...}` 里的语句属于这个 if，外层函数的形参列表有几个参数、是什么类型……这些"谁属于谁、谁嵌套在谁里面"的信息，词法分析器给不了。

**语法分析的价值**：

- 把扁平的 Token 流"折叠"成层次化的树，让"包含关系"显式化。
- 顺便检查源代码**是不是合法**——`int = ; main` 这种乱序写法，词法分析能切出 4 个合法 Token，但语法分析会立刻发现"等号左边不能是类型名"。

这是编译器"分阶段"思想的第二层：词法层负责"切"，语法层负责"组装 + 检查结构"。

### 1.2 为什么不手写而要用 Bison？

理论上语法分析可以手写——按文法规则写一堆 if/else/递归函数就行（PL/0 编译器就是手写的递归下降）。但 C-- 文法有 30+ 条产生式，手写要处理大量细节：

- 看到哪个 Token 应该走哪条规则
- 表达式优先级（`a + b * c` 应该先算 `*`）
- 出错时怎么恢复（不能因为一个错误就放弃整个文件）
- 怎么把每条规则的"成果"组装成树

**Bison**（GNU 版的 yacc）就是干这事的工具：你给它一份**文法描述**（.y 文件），它自动生成一个 C 函数 `yyparse()`，里面实现了 LALR(1) 解析算法——上面那些细节它全包。你只需要：

1. 用类似 BNF 的语法写文法规则；
2. 在每条规则后面写一段 C 代码（叫 **action**），告诉它"匹配到这条规则时，怎么用子节点的成果拼出父节点"。

工具干苦力，你专注业务。这就是为什么课程要求用 Bison。

### 1.3 1.2 阶段要把 1.1 的词法分析器改造

1.1 的词法分析器是**独立程序**：有自己的 `main`，用 `printf` 直接打印每个 Token。1.2 里它要变成**被 Bison 调用的子模块**——Bison 生成的 `yyparse()` 会反复调用 `yylex()` 拿 Token，词法分析器不再"自己打印"，而是把 Token 编号 `return` 给 Bison，并把语义值（INT 的数值、ID 的字符串等）通过 `yylval` 这个全局变量传过去。

**一句话**：词法分析器从"自己说"变成"被问就答"。详见 [`词法分析.md`](词法分析.md) 的 1.1 节，本阶段改造细节见本文 3.1 节。

## 2. 是什么（What）

这一节讲清楚做语法分析必须懂的概念。先讲文法和语法树，再讲 Bison 工具。

### 2.1 文法（Grammar）：用规则描述语言

**文法**是一组"替换规则"，描述一种语言里所有合法的写法。每条规则长这样：

```
左部 → 右部
```

意思是"左部这个东西，可以用右部这一串替代"。例如 C-- 的一条规则：

```
Stmt → RETURN Exp SEMI
```

读作："语句（Stmt）可以写成 `RETURN Exp SEMI`"——也就是 `return <表达式>;` 这种形式。这里 `RETURN` 和 `SEMI` 是终结符（Token，再不能展开），`Exp` 是非终结符（还能继续展开，比如 `Exp → INT`、`Exp → Exp PLUS Exp`）。

C-- 完整文法见 [`资料/.../Appendix.pdf`](../资料/现课程要求(AICanUse)/Appendix.pdf) 附录 A。本阶段把它们几乎一对一翻译成 Bison 的产生式。

### 2.2 语法树（Syntax Tree）：规则的"执行结果"

光有文法不够——文法只回答"合不合法"。但编译器后面要用程序的**结构**做语义分析、生成代码。这个结构就是语法树。

**语法树的节点分两类**：

1. **非终结符节点**：对应文法里的非终结符（`Program`、`ExtDef`、`Stmt`、`Exp` 等）。打印格式：`Name (lineno)`。
2. **终结符节点**（Token）：对应 Token（`INT`、`ID`、`SEMI` 等）。打印格式：
   - 有值的：`TOKEN: value`（如 `INT: 65280`、`ID: main`、`TYPE: int`）
   - 无值的：`TOKEN`（如 `SEMI`、`LP`、`IF`）

举例，源代码 `int main(){return 0;}` 的语法树（缩进 2 空格）：

```
Program (1)
  ExtDefList (1)
    ExtDef (1)
      Specifier (1)
        TYPE: int
      FunDec (1)
        ID: main
        LP
        RP
      CompSt (1)
        LC
        StmtList (2)
          Stmt (2)
            RETURN
            Exp (2)
              INT: 0
            SEMI
        RC
    ExtDefList (12)
```

每个非终结符节点后面的 `(N)` 是**行号**，规则是：**取该节点第一个子节点的行号**（如 `Program (1)` 取自第一个 `ExtDef` 的行号 1）。这个规则是 Tests1 测试集定的，详见 3.4 节。

### 2.3 LALR(1) 解析：Bison 用的算法

Bison 默认用 **LALR(1)** 算法解析（Look-Ahead LR，1 个向前看符号）。原理不展开，只说要点：

- Bison 从左到右读 Token，维护一个**状态栈**。
- 每读一个 Token，它要决定两件事之一：
  - **移进（Shift）**：把这个 Token 压栈，等更多信息。
  - **归约（Reduce）**：栈顶的几个符号正好匹配某条规则的右部，把它们"折叠"成左部。
- 当栈顶是 `Program` 且 Token 流耗尽时，解析成功。

我们写 .y 文件时**不用关心算法**——Bison 自动算。但要关心两个工程问题：

- **优先级冲突**：`a + b * c` 应该先归约 `+` 还是先归约 `*`？Bison 默认行为不一定对，需要用 `%left`/`%right` 显式声明优先级（见 3.3 节）。
- **错误恢复**：遇到错误 Token 时怎么不放弃整个文件，跳过一些 Token 继续解析？用 Bison 的 `error` 伪终结符（见 3.5 节）。

### 2.4 yylval：词法分析器 → 语法分析器的"数据通道"

`yylex()` 返回的 Token 编号只是个"标签"（比如 `INT = 258`）。但语法分析器还要知道这个 INT 的**具体数值**是多少、出现在**第几行**。这些"语义值"通过一个全局变量 `yylval` 传递。

`yylval` 的类型由 .y 文件里的 `%union` 声明决定。本阶段的设计：

```c
%union {
    struct TokVal {
        long   ival;    /* INT 的十进制值 */
        double fval;    /* FLOAT 的 double 值 */
        char  *sval;    /* ID/TYPE 的词素字符串 */
        int    line;    /* 该 Token 出现的行号 */
    } tok;
    Node *node;         /* 非终结符的语法树节点指针 */
}
```

每个终结符 Token 都用 `<tok>` 类型，每个非终结符用 `<node>` 类型。这样 Bison 在归约时能正确地把 `$1`/`$2`（每个右部符号的语义值）当对应类型用。

**为什么要专门塞个 `line` 字段？** 因为 Bison 默认不维护每个 Token 的行号（它的 `@n.first_line` 机制依赖 flex 维护 `yylloc`，而 flex 默认不做）。我们用 `YY_USER_ACTION` 宏让 flex 在每次匹配前把 `yylineno` 写进 `yylval.tok.line`，这样每个 Token 都自带行号，建树时随时可取。这是本阶段踩的一个关键坑，详见 3.2 节。

## 3. 怎么做（How）

### 3.1 改造 lexer.l：从独立程序 → Bison 子模块

[`C/src/lex/lexer.l`](../C/src/lex/lexer.l) 的改动：

**1) 包含 Bison 生成的头文件**：

```c
#include "node.h"         /* Node 类型 + out()/flush_output() */
#include "parser.tab.h"   /* Bison 生成的 Token 编号 + yylval 类型 */
```

注意顺序：`node.h` 必须在 `parser.tab.h` 前，因为 `%union` 用到了 `Node` 类型。

**2) 删除独立 main/yywrap**，加 `%option noyywrap` 让 flex 不需要 `yywrap` 函数。

**3) action 改为 `return TOKEN`**。比如：

```
";"   { return SEMI; }
"int" { yylval.tok.sval = strdup(yytext); return TYPE; }
0[xX][0-9A-Fa-f]+  { yylval.tok.ival = TO_LONG(16); return INT; }
```

**4) 用 `YY_USER_ACTION` 统一设置行号**（关键设计）：

```c
#define YY_USER_ACTION  yylval.tok.line = yylineno;
```

flex 在每次成功匹配一条规则、执行 action **之前**自动调用这个宏。这样所有 Token 都不用重复写"设置行号"，行号自动正确。

**5) type A 词法错误用 `out()` 不用 `printf`**，且**不 return**（让扫描继续）：

```
.  { out("Error type A at Line %d: Undefined character '%s'.\n", yylineno, yytext); }
```

这样错误 Token 被丢弃、扫描继续，语法分析看到的还是干净的 Token 流——词法错误不影响后续语法匹配。

**6) 新增复合赋值 `+=`/`-=`/`*=`/`/=`**（Tests1 A-8）：作为整体识别成 `COMPOUND_ASSIGN` Token，让语法层报 "Unsupported compound assignment"。

**7) 新增两类非法词素整体识别**：

- 数字开头后跟字母（如 `2x`）：规则比合法 INT 长，靠 flex 最长匹配优先整体报错。
- 多个小数点（如 `3.14.15`）：规则比合法 FLOAT 长，整体报错。

### 3.2 写 node.h + tree.c：语法树节点和打印

[`C/src/syntax/node.h`](../C/src/syntax/node.h) 定义节点结构：

```c
typedef struct Node {
    char *name;          /* 节点名（"Program" 或 Token 名） */
    int   lineno;        /* 行号：终结符=出现行；非终结符=首个子节点行号 */
    int   is_token;      /* 1=终结符，0=非终结符 */
    int   val_kind;      /* 终结符的值类型：NO_VAL/VAL_INT/VAL_FLT/VAL_STR */
    long  ival;          /* INT 的值 */
    double fval;         /* FLOAT 的值 */
    char *sval;          /* ID/TYPE 的词素 */
    struct Node *children[MAX_CHILDREN];
    int   nchild;
} Node;
```

**关键 API**：

- `newNode(name)`：新建非终结符节点。
- `newTokenNode(name, lineno, val_kind, ival, fval, sval)`：新建终结符节点。
- `addChild(parent, child)`：把 child 挂到 parent 下。**首个被挂的子节点决定 parent 的 lineno**——这就是 Tests1 "非终结符行号取自第一个子节点"规则的实现。

[`C/src/syntax/tree.c`](../C/src/syntax/tree.c) 的打印逻辑用递归，按 2 空格缩进：

```c
if (n->is_token) {
    switch (n->val_kind) {
        case VAL_INT: out("%s: %ld\n", n->name, n->ival); break;   /* INT: 65280 */
        case VAL_FLT: out("%s: %f\n", n->name, n->fval); break;    /* FLOAT: 3.140000 */
        case VAL_STR: out("%s: %s\n", n->name, n->sval); break;    /* ID: main */
        default:      out("%s\n", n->name); break;                 /* SEMI */
    }
} else {
    out("%s (%d)\n", n->name, n->lineno);                          /* Program (1) */
    /* 递归打印子节点 */
}
```

**输出缓冲（踩坑后改的设计）**：所有输出（树、错误信息）先进 `open_memstream` 内存流，最后在 `flush_output` 时统一写到 stdout。这样能控制"错误用例末尾不带 `\n`、树用例末尾带 `\n`"——Tests1 的 .exp 文件就是这种不一致格式。

### 3.3 写 parser.y 主干：合法程序建树

[`C/src/syntax/parser.y`](../C/src/syntax/parser.y) 的三段式结构：

**第 1 段：声明**（`%{ %}` 和 `%union`/`%token`/`%type`/`%right`/`%left`）

- `%union`：定义 yylval 类型（见 2.4 节）。
- `%token <tok> INT FLOAT ID ...`：声明所有终结符 Token，统一用 `<tok>` 类型。
- `%type <node> Program ExtDef ...`：声明所有非终结符，用 `<node>` 类型。
- 优先级声明（**从低到高**，赋值最低）：

  ```c
  %right ASSIGNOP                  /* = 右结合 */
  %left  OR                        /* || */
  %left  AND                       /* && */
  %left  RELOP                     /* > < >= <= == != */
  %left  PLUS MINUS                /* + - */
  %left  STAR DIV                  /* * / */
  %right NOT UMINUS                /* ! 和一元负号 */
  %left  LP RP LB RB DOT           /* 括号、点号最高 */
  ```

  这套表源自 Appendix A 的"运算符优先级表"。比如 `a + b * c`，`STAR` 比 `PLUS` 高，Bison 会先归约 `b * c`，再归约 `a + (b*c)`，正好对应"先乘后加"。

**第 2 段：规则**（`%% ... %%`）

每条产生式形如：

```c
Program : ExtDefList   { root = NT("Program"); addChild(root, $1); }
        ;
```

`$1` 指右部第 1 个符号的语义值（这里是 `ExtDefList` 的 Node 指针）。`{...}` 是 action——告诉 Bison 归约这条规则时怎么建节点。

**两个简化宏**：

```c
#define TKS(name, t, kind)  newTokenNode(name, (t).line, kind, (t).ival, (t).fval, (t).sval)
#define NT(name)            newNode(name)
```

`TKS` 用于建终结符节点，`NT` 用于建非终结符节点。这样 action 写起来短得多。

**空产生式的处理**：像 `ExtDefList → 空`、`StmtList → 空`、`DefList → 空`，action 直接 `$$ = NULL;`。`addChild` 会跳过 NULL 子节点——所以纯空产生式不会产生节点（符合 Tests1 规则）。

**第 3 段：用户代码**（`%%` 之后）

主要是 `main`：

```c
int main(int argc, char **argv) {
    if (argc > 1) { yyin = fopen(argv[1], "r"); }
    yyparse();
    if (!has_error && root) {
        printTree(root);
        flush_output(0);   /* 树末尾保留 \n */
    } else {
        flush_output(1);   /* 错误末尾去掉 \n */
    }
}
```

`has_error` 是全局标志——任何 type B 错误都置 1，置位后整棵树都不打印（符合 Tests1 规则："遇到语法错误只打印错误信息，不打印树"）。

### 3.4 行号传播的踩坑历程

**踩坑**：最初用 Bison 的 `%locations` + `@N.first_line` 取行号——结果**所有节点都显示行号 1**。

**原因**：flex 默认不维护 `yylloc`（Bison 的位置结构），需要手写 `YY_USER_ACTION` 显式维护。但维护 `yylloc` 还要算列号，太麻烦。

**解决方案**：放弃 `%locations`，把行号塞进 `%union` 的 `TokVal` 结构体：

```c
struct TokVal { long ival; double fval; char *sval; int line; } tok;
```

flex 用 `YY_USER_ACTION` 把 `yylineno` 写进 `yylval.tok.line`：

```c
#define YY_USER_ACTION  yylval.tok.line = yylineno;
```

这样每个终结符 Token 都自带行号。建非终结符节点时，`addChild` 自动从**第一个非空子节点**继承行号。两层机制叠加，整棵树的行号都正确。

### 3.5 B 类精确错误产生式（核心难点）

Tests1 要求**精确错误信息**（不是通用的 "Syntax error"）。我们为每种错误写专门的"错误产生式"，让 Bison 在匹配到错误模式时直接走对应规则、打印精确信息。

完整清单（见 [`C/src/syntax/parser.y`](../C/src/syntax/parser.y)）：

| 错误信息 | 触发用例 | 错误产生式 / 检测方式 |
|---------|---------|---------------------|
| `Error type A at Line X: Undefined character '$'` | A-1（`$` 字符） | lexer.l 第 180 行 |
| `Error type A at Line X: Undefined character in number "0xQQQ"` | E1-2（非法十六进制） | lexer.l 第 102 行 |
| `Error type A at Line X: Undefined character in number "099"` | E1-2（非法八进制） | lexer.l 第 118 行 |
| `Error type A at Line X: Undefined character in number "2.5e"` | E2-2（指数浮点缺尾数） | lexer.l 第 136 行 |
| `Error type A at Line X: Invalid identifier starting with digit '2x'` | A-2 | lexer.l 第 154 行 |
| `Error type A at Line X: Invalid floating point literal '3.14.15'` | A-5 | lexer.l 第 142 行 |
| `Error type B at Line X: Array dimension must be integer` | A-6/B-1（`int a[2.0]`） | `VarDec → VarDec LB FLOAT RB` |
| `Error type B at Line X: Trailing comma in function parameters` | B-1（`int f(int a,)`） | `FunDec → ID LP ParamDec COMMA RP`（特殊设计，见下） |
| `Error type B at Line X: Anonymous struct declaration` | B-1（`struct {...};`） | `last_struct_anonymous` 标志位检测 |
| `Error type B at Line X: Array initialization not supported` | B-1（`int a[4]={...}`） | `Dec → VarDec ASSIGNOP ...` 检查 VarDec 是否含 LB |
| `Error type B at Line X: Empty while statement body` | B-1（`while(c);`） | `Stmt → WHILE LP Exp RP SEMI` |
| `Error type B at Line X: Extra closing bracket in array declaration` | B-2（`int a[5]]`） | `VarDec → VarDec LB INT RB RB` |
| `Error type B at Line X: Variable declaration after statements in block` | A-9/B-2 | `Stmt → Specifier DecList SEMI`（在语句位置出现声明） |
| `Error type B at Line X: Missing member name after '.'` | A-10（`a.;`） | `Exp → Exp DOT`（DOT 后非 ID） |
| `Error type B at Line X: Unsupported compound assignment operator '+='` | A-8 | `Exp → Exp COMPOUND_ASSIGN Exp` |
| `Error type B at Line X: Missing return value in non-void function` | A-3（`return;`） | `Stmt → RETURN SEMI` |
| `Error type B at Line X: Global variable initialization not allowed` | A-4 | `ExtDef → Specifier ExtDecList ASSIGNOP Exp SEMI` |
| `Error type B at Line X: Trailing comma in function call arguments` | A-7（`add(3,)`） | `Exp → ID LP Exp COMMA RP` |

**两个非显然的设计**：

1. **尾逗号的 LALR 冲突**：`FunDec → ID LP VarList RP` 配合 `FunDec → ID LP VarList COMMA RP` 时，Bison 在看到 `ParamDec COMMA` 后会面临"shift COMMA 进 `VarList COMMA RP` 分支"还是"reduce `VarList` 后才发现尾逗号"。默认行为是 reduce，结果尾逗号分支永远进不去。**解决**：再加一条 `FunDec → ID LP ParamDec COMMA RP`（不带递归 VarList），look-ahead COMMA 时 Bison 会 shift 进这条。详见 [parser.y:179-186](../C/src/syntax/parser.y)。

2. **匿名结构体检测**：`struct { int tag; };` 中的 `OptTag` 是空产生式。无法用单一规则识别"匿名 + 用作声明"。**解决**：在 `StructSpecifier` 的 action 里记录 `last_struct_anonymous = (OptTag 为 NULL)`，在外层 `ExtDef` 或 `Def` 的 `Specifier SEMI` 分支检查这个标志位。详见 [parser.y:99-107](../C/src/syntax/parser.y)。

## 5. 难点回顾与经验

1. **行号传播是最隐蔽的坑**：默认以为 Bison 的 `%locations` 会自动处理，结果所有节点都显示行号 1。**教训**：用工具时不能假设它的"默认行为"符合预期——要先跑通最小用例验证关键属性（如行号），再扩展。

2. **FLOAT 精度坑**：`6.02e23` 期望 `602000017271895229464576.000000`。用 `atof`/`strtod`（double）得到 `601999...`，**对不上**。最终用 `strtof`（float32）→ double 提升后才精确匹配。说明 **C-- 的 FLOAT 是 IEEE754 单精度**，不是 double。详见 [lexer.l:127-132](../C/src/lex/lexer.l)。

3. **错误产生式不是"加一条规则"那么简单**：Bison 的 LALR 算法有 shift/reduce 冲突，光写规则没用，得理解 look-ahead 行为才能让错误分支真正命中。尾逗号就是典型——必须额外写一条"不递归 VarList"的变体规则。

4. **输出缓冲解决格式不一致**：Tests1 树用例 .exp 末尾带 `\n`、错误用例不带。直接 `printf` 无法事后区分。用 `open_memstream` 收集所有输出、最后 `flush_output(strip)` 统一控制，干净利落。

5. **测试数据本身可能不规范**：B-1.exp/B-2.exp 是 CRLF（违反 .gitattributes），A-10.exp 末尾多 `\n`（与其他错误用例不一致）。**遇到对不上时先检查测试数据**，别一上来就怀疑自己的代码。

## 6. 后续阶段

本阶段产出的语法树（`Node*`）和 1.1 的 Token 流是后续所有阶段的输入：

- **实践 2 语义分析**：遍历语法树，建符号表，检查类型、未定义引用、重复定义等。
- **实践 3 中间代码生成**：基于语法树做语法制导翻译，生成 irsim IR。
- **实践 4 目标代码生成**：把 IR 翻译成 MIPS32。
- **实践 5 代码优化**：对 IR / 目标代码做优化。

也就是说，本阶段把语法树建正确，是后续 4 个阶段的承重墙——树错了后面全错。所以**本阶段的精确 diff 通过率至关重要**。
