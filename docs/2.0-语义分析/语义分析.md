## 1. 为什么需要语义分析

词法分析和语法分析回答了"程序的字符流是否符合 C-- 的词法和文法"这两个问题。但符合文法的程序依然可能是错的，举几个例子：

```c
int main() {
    j = i + 1;        // j 和 i 都没声明——文法允许（Exp → ID = Exp），但语义上"未定义变量"
    int x = "hello";  // 文法允许（Exp → STRING，C-- 里其实没有 STRING，假设允许），语义上"类型不匹配"
    int x;            // 同一作用域里第二次声明 x——文法允许，语义上"重复定义"
    return 1.5;       // 函数声明返回 int，return float——文法允许，语义上"返回类型不匹配"
}
```

这些错误都涉及**上下文相关**的信息：变量是否声明过、类型是什么、作用域里有没有重名、函数签名匹不匹配。上下文无关文法（CFG）天然记不住这些信息——一个非终结符 `Exp` 在不同位置展开成 `ID = Exp` 时，文法规则没有任何差别。要描述这些约束，必须给语法树节点挂上"属性"（变量名→类型映射、当前作用域、函数返回类型等），并在树遍历中传播和检查。这就是**语义分析**（semantic analysis）的工作。

工程上的直接产物是**符号表**（symbol table）和**类型系统**（type system）。符号表回答"这个名字是什么、什么类型、在哪个作用域"；类型系统回答"这两个类型是否等价、这个运算的运算数类型是否合法"。在 C-- 这种静态强类型语言里，所有这些检查都能在编译期完成，所以也叫**静态语义检查**。

## 2. 是什么——核心概念

### 2.1 属性文法

属性文法 = 上下文无关文法 + 给每个非终结符附加**属性**（attribute），以及附加**语义规则**（semantic rules）描述属性如何计算。属性分两类：

- **综合属性**（synthesized）：节点的属性由它的**子节点**属性计算得来。例如表达式 `Exp → Exp PLUS Exp` 的 `.type` 由两个运算数的 `.type` 决定。自底向上传播。
- **继承属性**（inherited）：节点的属性由它的**父节点或兄弟节点**属性得来。例如变量声明 `DecList → VarDec` 的 `.type` 来自父节点 `Def → Specifier DecList` 中 `Specifier` 的 `.type`。自顶向下传播。

举个具体例子，`int x, y[3];` 的依赖图：

```
Def → Specifier DecList SEMI
      Specifier.type = int   （综合：来自 TYPE 终结符的词法值）
                            ↓（继承：传给 DecList）
      DecList.type = int
                ↓
      Dec → VarDec    VarDec.type = int   （DecList.type 传过来）
                         ↓
                    x.type = int           （综合：VarDec→ID 时把类型赋给 ID）
```

`Specifier.type` 是综合属性（自底向上），`DecList.type / VarDec.type` 是继承属性（自顶向下，从 Specifier 继承）。

### 2.2 语法制导翻译 SDT

把属性规则嵌入到文法产生式里就叫**语法制导翻译**（syntax-directed translation，SDT）。具体到工程实现有两种风格：

- **S 属性文法**（只用综合属性）：可以在自底向上语法分析时**边归约边计算**，bison 的 action（`{ $$ = ...; }`）就是这种。实践 1 的建树就是 S 属性：每个非终结符归约时构造一个 Node。
- **L 属性文法**（综合 + 一次继承）：可以在一遍深度优先遍历里计算，但要求继承属性只依赖"左边"的兄弟节点和父节点。

C-- 的语义检查**不是** S 属性文法：变量声明的类型从 Specifier 流向 DecList（继承），而符号表填充又依赖当前作用域（继承自外层）。这些属性在 bison 的 action 里没法都算清楚，所以本项目采用**两阶段**做法：

1. **语法分析阶段（实践 1）**：bison action 只负责**建语法树**，所有 `Exp → Exp PLUS Exp { $$ = NT("Exp"); ... }` 都是把子节点挂到父节点上，等价于综合属性"构造语法树结构"。
2. **语义分析阶段（实践 2）**：在已建好的语法树上做**后序遍历**，遍历到 `Def` 节点时把 Specifier 的类型"传给"DecList，遍历到 `Exp` 时根据子表达式的类型推导本节点类型。等价于一次 L 属性文法的求值。

这种"先建树后遍历"的设计有个好处：**符号表/作用域/类型等数据结构完全用 C 写，不嵌在 bison action 里**，便于阅读和维护。

### 2.3 符号表

符号表（symbol table）是一个从**名字到符号信息**的映射。C-- 里名字有三种：变量、函数、结构体（结构体名是一个类型，要单独登记）。

```c
enum { SYM_VAR, SYM_FUNC, SYM_STRUCT };
struct Symbol_ {
    char *name;        // 名字（ID 词素）
    int   kind;        // 三种之一
    Type  type;        // 类型（变量/结构体的类型；函数的返回类型）
    FieldList params;  // 函数形参链表（仅 SYM_FUNC）
    int   nparam;
    int   declared_only;  // 函数仅声明未定义（要求3.1）
    int   defined;        // 函数已定义
    int   lineno;         // 声明/定义出现的行号
    int   depth;          // 所在作用域深度
    Symbol next_hash;     // 哈希桶拉链
    Symbol next_scope;    // 同一作用域链表（弹栈用）
};
```

**查表很快：pjw 哈希函数**。Project_2.pdf 3.2.3 给的算法：

```c
static unsigned int hash_pjw(const char *name)
{
    unsigned int val = 0, i;
    for (; *name; ++name) {
        val = (val << 2) + (unsigned char)*name;
        if ((i = val & ~0x3fffU)) val = (val ^ (i >> 12)) & 0x3fffU;
    }
    return val;
}
```

桶数 0x3fff（16383），冲突用拉链法。每次插入时按哈希值定位桶，挂在桶头。

**作用域：函数式栈**。C-- 允许嵌套语句块 `{ ... }`，内层同名变量**遮蔽**外层。本项目用函数式作用域栈（每个作用域一个符号链表）：

- 进入 `CompSt` → `symtab_enter_scope()` 压一层；
- 离开 → `symtab_leave_scope()` 弹一层，并把该层符号从哈希桶里摘除。
- 查表 `symtab_lookup` 从栈顶往下扫，返回第一个匹配（内层遮蔽外层）。

为什么选函数式而不是命令式（同一张表，符号挂 depth 字段）？因为函数式"离开作用域就把符号删掉"语义更直观，调试时栈的当前状态一目了然，不容易残留过期符号。

> **注意：必做模式不开 ENABLE_SCOPE 宏**，`symtab_enter_scope/leave_scope` 退化为空操作，所有符号都在全局作用域（depth=0）。这是为了匹配 D-2 的行为——嵌套块内的同名重定义**必须**报错（必做标准）；选做开了 ENABLE_SCOPE，嵌套块内的同名变量是合法遮蔽（E-2 期望的行为）。

### 2.4 类型系统

C-- 三种类型（Project_2.pdf 3.2.4）：

```c
struct Type_ {
    enum { BASIC, ARRAY, STRUCTURE } kind;
    union {
        int basic;                          // 0=int, 1=float
        struct { Type elem; int size; } array;
        FieldList structure;
    } u;
};
struct FieldList_ {
    char *name;       // 域名（结构体字段）或形参名
    Type  type;
    FieldList tail;
};
```

- **BASIC**：int 或 float，用 `basic` 字段 0/1 区分。
- **ARRAY**：链表式多维。`int a[10][3]` → `array(10, array(3, int))`，外层是第一维。
- **STRUCTURE**：结构体域用 FieldList 链表。

**类型等价**有两种判据：

- **名等价**（name equivalence）：两个类型当且仅当"名字相同"才等价。两个 `struct A` 类型的变量是同型；`struct A` 和 `struct B` 即使内部字段完全一样也是不同型。
- **结构等价**（structural equivalence）：两个类型当且仅当"结构相同"才等价。逐域比较 name 和 type；数组只比"基类型 + 维数"（不比每维 size，所以 `int[10][2]` 和 `int[5][3]` 同型，PDF 3.3 明确）；结构体里的数组域不递归展开元素。

本项目默认用名等价；选做 3.3（开 `ENABLE_STRUCT_EQ` 宏）切换到结构等价。

**名等价的工程实现**：每个结构体**定义**（`struct A {...}`）构造一个**唯一**的 STRUCTURE Type 实例，所有这个结构体类型的变量共享这一个 Type 指针。比较两个类型时，`a == b`（指针相等）即名等价；不同结构体定义产生不同指针，必然不等。

```c
int type_equal(Type a, Type b) {
    if (a->kind != b->kind) return 0;
    switch (a->kind) {
    case BASIC:     return a->u.basic == b->u.basic;
    case ARRAY:     return array_compatible(a, b);  // 维数+基类型，不比 size
    case STRUCTURE:
#ifdef ENABLE_STRUCT_EQ
        /* 结构等价：逐域比 type（不展开数组），长度相同 */
        ...
#else
        return (a == b);   /* 名等价：同指针 */
#endif
    }
}
```

## 3. 怎么做——实现

### 3.1 数据流总览

```
test.cmm → flex → bison(建语法树) → sem_analyze 后序遍历 → 错误输出
              ↑                       ↓
              共享实践 1 的建树代码    符号表栈 + 类型系统
```

可执行两个：`cmmc`（必做，无宏）和 `cmmc_full`（选做全开，`-DENABLE_FUNC_DECL -DENABLE_SCOPE -DENABLE_STRUCT_EQ`）。两者共享 `parser.tab.c / tree.c / lex.yy.c / type.c / symtab.c / sem.c`，只是宏开关不同。

### 3.2 入口：sem_analyze

```c
void sem_analyze(Node *root) {
    symtab_init();
#ifdef ENABLE_FUNC_DECL
    /* 先扫一遍，标记所有会被定义的函数名（错误18 用，详见 3.6）*/
    for each ExtDef: prepass_extdef(extdef);
#endif
    /* 正式遍历：每个 ExtDef */
    for each ExtDef: visit_extdef(extdef);
}
```

### 3.3 顶层结构 visit_extdef

按 ExtDef 的子节点形态分发：

| ExtDef 形式 | 处理 |
|---|---|
| `Specifier ExtDecList SEMI`（全局变量） | Specifier→Type；遍历 ExtDecList（直接是 VarDec 链表，无 Dec 包装）逐个 insert_var |
| `Specifier SEMI`（结构体定义） | Specifier→Type 触发结构体注册，ExtDef 本身无变量 |
| `Specifier FunDec CompSt`（函数定义） | visit_fundef：填函数符号，进作用域，形参填表，遍历 CompSt，弹作用域 |
| `Specifier FunDec SEMI`（函数声明，3.1） | visit_fundecl：仅声明，declared_only=1 |

**Specifier → Type**：`int/float` → BASIC；`struct A {...}` → 新建 STRUCTURE 类型 + 注册结构体名；`struct A`（不带定义）→ 查表返回已注册的 Type 指针（名等价的关键）。

### 3.4 语句 visit_stmt

按 Stmt 第一个子节点分发：

- `Exp SEMI` → check_exp（仅推导类型，触发表达式错误）。
- `CompSt` → enter_scope + 递归 visit_compst + leave_scope（嵌套块）。
- `RETURN Exp SEMI` → check_exp 后与 `current_return_type` 比，不等报错误8。
- `IF LP Exp RP stmt [ELSE stmt]` → check_exp(条件)；条件须 BASIC（int 或 float 均可），否则错误7；递归 stmt。
- `WHILE LP Exp RP stmt` → 同 IF。

> `current_return_type` 是一个全局变量，进函数体时设为该函数的返回类型，return 时读，离开函数体时清空。

### 3.5 表达式 check_exp

`Type check_exp(Node *exp)` 是核心。返回表达式的 Type；遇到错误时返回 NULL（NULL 短路传播，避免基于错误类型连锁报错）。

| Exp 形式 | 推导 / 检查 |
|---|---|
| `ID` | 查表（SYM_VAR）；无→错误1，返回 NULL；有→返回变量类型 |
| `INT` / `FLOAT` | 返回 int / float |
| `ID LP Args RP`（函数调用） | 查表（SYM_FUNC）；无→错误2；是变量→错误11；逐实参 check_exp 比类型/数量；不符→错误9；返回函数返回类型 |
| `Exp ASSIGNOP Exp` | 左侧须左值，否则错误6；左右 type_equal，否则错误5；返回左侧类型 |
| `Exp AND/OR Exp` | 两边须 int（布尔），否则错误7；返回 int |
| `Exp RELOP Exp` | 两边须 BASIC（int 或 float 均可），否则错误7；返回 int |
| `Exp PLUS/MINUS/STAR/DIV Exp` | 两边须同型 BASIC，否则错误7（短路返回 NULL）；返回操作数类型 |
| `MINUS Exp` | 须 int/float，否则错误7 |
| `NOT Exp` | 须 int，否则错误7；返回 int |
| `Exp LB Exp RB`（数组访问） | 左侧须 ARRAY，否则错误10；下标须 int，否则错误12；返回元素类型 |
| `Exp DOT ID`（域访问） | 左侧须 STRUCTURE，否则错误13；域须存在，否则错误14；返回域类型 |
| `LP Exp RP` | 返回内部 Exp 类型 |

**左值判定 is_lvalue**：从语法层判断（PDF 3.2.5），左值只有三种产生式：`ID`、`Exp LB Exp RB`、`Exp DOT ID`。出现在赋值号左边的其他形式（INT/FLOAT/函数调用/算术表达式）→ 错误6。

### 3.6 错误 18 的两阶段扫描

错误 18 是"函数声明了但从未定义"。要在**声明位置**按行号顺序报错，必须事先知道该函数最终会不会被定义。本项目用两阶段扫描：

1. **prepass_extdef**：扫一遍 ExtDefList，把所有 `Specifier FunDec CompSt`（函数定义）的 ID 名加入 `defined_func_names` 集合。
2. **主遍历 visit_fundecl**：遇到声明时，如果名字不在 `defined_func_names` 里，立即按声明行号报错误18。

这保证 E-1 中错误18（行4）先于错误19（行21）输出，与其他错误按行号顺序一致。

### 3.7 错误 3 的两种文字

错误 3（变量重名）在不同用例有两种措辞：

- A-3（函数体顶层同作用域重定义）：`Redefined Variable.`
- D-2（必做模式下嵌套块内重定义外层变量）：`Redefined variable or structure conflict.`

判定依据是**语法嵌套深度 `nest_depth`**（与符号表作用域栈解耦）：
- 进函数体 = 1（visit_fundef 设置）；
- 进 Stmt→CompSt 嵌套块再 +1（visit_stmt 设置）；
- `nest_depth >= 2` 报 D-2 文字，否则 A-3 文字。

不开 ENABLE_SCOPE 时，函数式作用域栈退化为空操作（所有变量在同一 depth=0），但 `nest_depth` 仍按语法嵌套累加，所以两种文字能正确区分。

## 4. 选做的宏开关机制

| 宏 | 影响 |
|---|---|
| `ENABLE_FUNC_DECL` | 开：parser.y 允许 `Specifier FunDec SEMI` 函数声明产生式，sem.c 处理错误18/19；关：报 type B "Syntax error." |
| `ENABLE_SCOPE` | 开：symtab 真正压栈/弹栈，嵌套块内同名变量合法遮蔽；关：所有符号在 depth=0，嵌套重定义报错3 |
| `ENABLE_STRUCT_EQ` | 开：type_equal 对 STRUCTURE 逐域比 type；关：指针相等（名等价） |

`cmmc` 不开任何宏（必做行为），过 A/B/C/D 27 个用例；`cmmc_full` 三宏全开（选做行为），过 A/B/C/E 30 个用例中相应分组。两个可执行**共享所有 .c 源码**，只是编译时宏开关不同。

## 6. 踩坑记录

| 坑 | 现象 | 解 |
|---|---|---|
| `strdup` 隐式声明 | GCC 7.5 + `-std=c99` 严格模式下 strdup 未声明，返回 int 截断 64 位指针，lexer 跑两步就段错误 | lexer.l 自定义 `dup_yytext(yytext, yyleng)`（malloc+memcpy），不依赖 POSIX strdup |
| parser 空产生式 NULL 子节点 | `DefList → ε` 返回 NULL，`addChild` 直接 return 不加槽位，导致 `CompSt->children[1]` 可能是 StmtList 而非 DefList | 在 visit_compst / struct_specifier_type 里**按节点名识别**，不按 children 索引 |
| ExtDecList 与 DecList 结构不同 | 全局 `int a, b;` 是 ExtDecList → VarDec 链表（无 Dec 包装），局部 `int a, b;` 是 DecList → Dec（VarDec） 链表 | 拆成 `visit_extdeclist`（VarDec 直填）和 `visit_declist`（Dec 解包）两个函数 |
| 错误3 两种文字 | A-3 `Redefined Variable.` vs D-2 `Redefined variable or structure conflict.`，按符号表 depth 区分时不开 ENABLE_SCOPE 全是 depth=0 失效 | 用独立的 `nest_depth` 计数器（语法嵌套），与符号表作用域栈解耦 |
| D-1 vs E-1 互斥 | D-1 要求函数声明报 type B，E-1 要求函数声明合法（错误18/19）| 宏开关 `ENABLE_FUNC_DECL`：开 → 合法，关 → type B |
| 错误18 输出顺序 | E-1 中错误18（行4）应先于错误19（行21），但 visit_fundecl 走到行21 先报19，最后扫描报18 | 两阶段：prepass 先收集"会被定义的函数名"，主遍历声明时立即按行号报18 |
| 类型等价传播短路 | 错误7 后返回非 NULL，上层基于错误类型继续报错误5（连锁） | PLUS/MINUS/STAR/DIV 类型不符报错误7 后返回 NULL |
