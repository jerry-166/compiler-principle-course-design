# 实践 3 中间代码生成 — 测试指南

> 配套可执行：`C/bin/cmmc_ir`（必做，读 .cmm 生成 .ir）+ `C/bin/irvm`（IR 虚拟机，执行 .ir）
> 测试集：`Tests/Tests3/inputs/*.cmm`（16 个：A-1~A-5, B-1~B-3, C-1~C-2, E1-1~E1-3, E2-1~E2-3）
> 批量脚本：`操作手册及脚本/run_tests_ir.sh`

## 1. 验证机制为什么和实践 2 不一样

实践 2（语义分析）的判定很直接：每个 .cmm 配一个 `.exp`，编译器把错误信息打到 stdout，逐字 diff 就完事。

实践 3 没法这么做。中间代码本身是一段"程序"，它对不对，**不能靠肉眼读 IR**，更不能拿一段固定的 IR 文本去逐字 diff——同一个源程序可以有多种等价的 IR（临时变量编号不同、跳转标签顺序不同都合法）。所以 Tests3 目录下**没有 `.exp` 文件**，给的"期望"是一组**程序运行后应当输出的 `write` 值序列**。

判定思路因此变成一条完整的执行链：

```
.cmm 源程序 ──cmmc_ir──▶ .ir 中间代码 ──irvm──▶ 实际 write 序列 ──对比──▶ 期望 write 序列
                              ▲ 读 stdin 提供 read 输入
```

这条链里**任何一环错都会暴露**：cmmc_ir 生成的 IR 有语法/语义错误，irvm 会拒绝执行或算错；translate 写错了加法，irvm 算出来的数就和期望对不上。所以"跑出正确的 write 序列"是一个端到端的强验证——它把翻译器和虚拟机一起验了。

`irvm` 是我们自己写的 IR 解释器（实践 3 的 Task 1），它按 irsim 的语义执行 IR：`READ` 时从 stdin 读一个整数、`WRITE` 时把一个整数打到 stdout、`ARG`/`PARAM`/`CALL` 传参调用、`GOTO`/`IF x relop y GOTO` 控制流、`DEC v N` 给数组开 N 字节、`*t := x` / `x := *t` 做内存读写。

## 2. 编译两个可执行

**必做 cmmc_ir**（生成 IR，不开选做宏）：

```bash
gcc -std=c99 -Wall -o C/bin/cmmc_ir \
    C/src/syntax/parser.tab.c C/src/syntax/tree.c \
    C/src/lex/lex.yy.c \
    C/src/sem/type.c \
    C/src/ir/ir.c C/src/ir/operand.c \
    C/src/ir/translate.c C/src/ir/translate_exp.c C/src/ir/translate_addr.c \
    C/src/ir/main_ir.c \
    -I C/src/syntax -I C/src/sem -I C/src/ir -lm
```

**irvm**（IR 虚拟机，必做也用，单独编译）：

```bash
gcc -std=c99 -Wall -o C/bin/irvm C/src/irvm/irvm.c -lm
```

> 编译时 `lex.yy.c` 可能报 `fileno`/`yyunput` 两个 warning，这是 flex 生成代码的老问题，与翻译器无关，可忽略。翻译器自身源码应零 warning。

## 3. 单跑一个用例

最直接的手感，是用 A-1（无 read、最简单的 while 循环）走一遍全链：

```bash
# 第 1 步：生成 IR 到文件（不给 out.ir 会打到 stdout）
./C/bin/cmmc_ir Tests/Tests3/inputs/A-1.cmm out.ir

# 第 2 步：A-1 无 read，stdin 给空，irvm 执行
./C/bin/irvm out.ir < /dev/null
# 期望输出两行：120 和 325
```

带 `read` 的用例要把输入按行喂给 stdin（每个 `READ` 指令读一行一个整数）。比如 A-2 要读三个整数 `a b c`：

```bash
printf '40\n30\n20\n' | ./C/bin/irvm out.ir
# 期望输出三行：7、2000、3000
```

把"输入用逗号记、输出也用逗号记"是 Tests3 的约定，方便一行一组地罗列。手工跑时记得把逗号还原成换行喂给 irvm。

## 4. irvm 的输入输出约定

irvm 读两路输入：第一个命令行参数是 `.ir` 文件，标准输入是 `READ` 的数据源。

- **每个 `READ t` 指令**：从 stdin 读一行，解析成整数，赋给临时变量 `t`。stdin 里一个整数占一行。给多了没关系（多余的被忽略），给少了会读到 EOF（按 0 处理）。
- **每个 `WRITE t` 指令**：把 `t` 的整数值打到 stdout，**一个值占一行**。
- **出错时**：IR 语法不合法（如未知指令、操作数类型错）irvm 会把错误打到 stderr 并退出码非 0；翻译器生成的合法 IR 不会触发。

所以"程序跑出来的 write 序列"就是 irvm stdout 的所有行，按出现顺序。脚本里用 `paste -sd, -` 把多行拍成逗号分隔的一行，方便和期望的逗号串比。

## 5. 批量脚本 run_tests_ir.sh 用法

```bash
bash 操作手册及脚本/run_tests_ir.sh 必做    # 默认，测 A/B/C 共 27 组输入
bash 操作手册及脚本/run_tests_ir.sh 选做    # 测 E1/E2（需 cmmc_ir_full，未实现则跳过并提示）
bash 操作手册及脚本/run_tests_ir.sh all     # 先必做再选做
```

脚本流程：

1. 先重编译 `cmmc_ir` 和 `irvm`（编译失败立即报错退出，避免拿旧二进制测出误导结果）。
2. 遍历测试数据数组（`name|reads|expect` 三段式）：按 `|` 拆出用例名、read 输入、期望 write。
3. 每组：`cmmc_ir` 生成 .ir 到 `/tmp/ir_tests/` → 把 reads 的逗号串转成换行喂给 irvm → 把 irvm 的多行输出用 `paste -sd,` 拍成逗号串 → 与 expect 字符串相等比较。
4. 每组输出 PASS（绿）/FAIL（红），FAIL 时打印 expected vs got。
5. 末尾汇总：`==== 必做: X pass, Y fail (共 Z 组) ====`。

> 一个用例有多组输入（如 A-2 有 3 组、C-2 有 4 组），脚本会把每组当独立的一行结果跑，所以"27 组"是按输入组数算的，不是按 .cmm 文件数算的。

## 6. irsim.pyc：老师给的手动调试器（不进自动化）

老师随课程发的 `irsim.pyc` 是一个 **Python 3.8 + PyQt5 的 GUI 单步调试器**：加载一个 .ir 文件，可以单步执行、查看每个操作数/变量的当前值、观察跳转。它对**人工理解某条 IR 在干嘛、定位翻译错误**非常有用。

但它**没有命令行模式**——必须开 GUI 点鼠标，没法批量跑、没法喂 stdin 自动取输出。所以它不进 `run_tests_ir.sh`，只作为人工对照工具。在 WSL/真机上要用它，需要 Python 3.8 解释器（irsim.pyc 是 3.8 字节码，高版本解释器可能拒绝）：

```bash
# 用 uv 装 Python 3.8（推荐，不污染系统）
uv python install 3.8
uv venv --python 3.8 .venv38
source .venv38/bin/activate
pip install PyQt5
python irsim.pyc 某个.ir      # 弹出 GUI
```

> 真机 Ubuntu 20.04 自带 Python 3.8，可直接 `python3 irsim.pyc`，省去装环境。WSL 里没有 GUI 显示则需配 X server。自动化测试不依赖它，装不装都不影响 27/27。

## 7. 各组测试重点

**A 组（基础，5 个文件 / 12 组输入）——必做核心**：把表达式翻译、控制流翻译、`read`/`write`、一维数组这几块挨个验。
- A-1：while 循环累乘 + write，无 read。验算术累乘、LABEL/GOTO/IF 翻译。
- A-2：读三个数做条件判断 + 多个 write。验 `>` 比较翻译、`/`、嵌套 if。
- A-3：循环往数组 `buf[i]` 写 `i*i+3`，再 `read k` 输出 `buf[0], buf[k], buf[5]`。验**一维数组寻址**（`&v_buf` + `i*#4` + `*t`）。
- A-4：双重循环求和（外层 i、内层 j，仅对奇数 i 累加 i*j）。验嵌套 for、`==`/`!=`、`<=`。
- A-5：递归函数 `fib(n)`。验 `PARAM`/`ARG`/`CALL`/递归调用、`RETURN`。

**B 组（综合，3 个文件 / 8 组输入）——必做**：数组 + 函数 + 多参数组合。
- B-1：读 6 个数进数组，找最大值下标和值，再调 `scoreAt` 算 `val*2+1`。验数组写入、循环扫描、函数调用。
- B-2：双层循环 + 双参数函数 `pairVal(a,b) = a*a + a + b`，数满足 `>阈值` 的对数。验多 `PARAM` 顺序、嵌套调用。
- B-3：数组先填 `i+2`，再对每个元素调 `clampAdd(elem, 20)`（返回 `min(elem+3, 20)`）就地更新并累乘。验数组元素既是函数实参又是函数返回值写入目标。

**C 组（经典算法，2 个文件 / 7 组输入）——必做**：典型的"会被面试题化"的两个程序。
- C-1：辗转相除求 GCD。验 `while` + 取模 `/` `%`（`%` 在 IR 里是 `a - a/b*b` 拆出来的）。
- C-2：数位和 `digitSum`，含负数取绝对值。验 `if (x<0) x=-x`、`while (x>0)` 取个位 `x - x/10*10`、`/10` 缩位。

**E1 组（结构体，3 个文件 / 6 组输入）——选做**：待 Task 9 实现结构体翻译后补充。
**E2 组（高维数组 + 数组参数，3 个文件 / 4 组输入）——选做**：待 Task 10 实现后补充。

## 8. 检查清单

回归通过的标准，按重要性排：

- **必做 27/27 组 PASS**（`bash 操作手册及脚本/run_tests_ir.sh 必做` 末行 `==== 必做: 27 pass, 0 fail ====`）。这是上机检查的硬指标。
- **选做 E1/E2 共 10 组 PASS**（Task 9/10 完成后，`bash 操作手册及脚本/run_tests_ir.sh 选做`）。
- **`Cannot translate` 提示**：必做范围（A/B/C）内**不得出现**任何 `Cannot translate`——必做特性必须全支持。选做特性（结构体、高维数组、数组参数）在 cmmc_ir（不开选做宏）下允许打印 `Cannot translate` 并跳过，这是预期降级行为。
- **编译零 warning**：翻译器自身源码（`ir.c/operand.c/translate*.c/main_ir.c/irvm.c`）`-Wall` 下无 warning。flex 生成的 `lex.yy.c` 的 warning 可忽略。
- **多组输入全覆盖**：多组输入的用例（A-2 三组、A-3 三组、A-5 三组、B-1 三组、B-3 三组、C-1 三组、C-2 四组）每组都要 PASS，不能只测第一组。
