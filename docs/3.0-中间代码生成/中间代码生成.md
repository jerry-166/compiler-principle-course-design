## 1. 为什么需要中间代码

实践 1 用 Flex+Bison 把字符流变成了**语法树**（AST），实践 2 在树上做了语义检查，确认程序"类型对、作用域对、函数签名对"。但到此为止我们还不能"运行"程序——语法树是嵌套的、给人看的抽象结构，它描述了 `c = a + b * 2` 这种表达式的层级关系，却没有告诉我们"先算什么、后算什么、每一步把结果放哪"。要真正执行，还需要一层**贴近机器的线性表示**，把嵌套的树"摊平"成一串顺序执行的指令。

这层表示就叫**中间代码**（intermediate code），它夹在语义分析和目标代码生成之间，是从"抽象语法"到"机器指令"的过渡。

### 1.1 为什么不直接生成目标代码

理论上完全可以"语法树 → MIPS 目标代码"一步到位，但课程把这一步拆成实践 3（出 IR）和实践 4（出 MIPS）两个阶段，原因有三：

**第一，分而治之降低难度**。语法树到机器指令之间要处理的事情非常多：表达式求值顺序、寄存器分配、栈帧布局、跳转目标地址……如果一次性全做，每个翻译函数都要同时操心"语义对不对"和"寄存器够不够"，代码会复杂到无法维护。先翻译成与机器无关的 IR，把"语义"这一层固定下来；再在 IR 上做寄存器分配和指令选择（实践 4），每一步只解一个问题。

**第二，每阶段都能独立验证**。IR 是文本的、人能读的、而且——关键——**我们写了一个能直接执行 IR 的虚拟机 irvm**。所以实践 3 出完 IR 后，不用等实践 4 的 MIPS 仿真器，就能马上跑起来、看 `write` 输出对不对。如果直接跳到 MIPS，一个 bug 可能是翻译的锅、也可能是寄存器分配的锅、还可能是仿真器的锅，定位极困难。IR 这一层把"语义翻译正确"和"目标代码生成正确"彻底解耦。

**第三，便于优化**。实践 5 的代码优化（常量折叠、死代码消除等）在 IR 上做最自然——IR 是线性的、三地址的、没有寄存器分配的干扰，优化算法容易实现和验证。如果直接优化机器码，要处理的目标机细节太多。

### 1.2 为什么用三地址码

中间表示有好多流派：抽象语法树（AST）、有向无环图（DAG）、栈式机器代码、三地址码（TAC）、SSA（静态单赋值）等。本项目用**三地址码**，对应 Project_3.pdf 的 irsim IR。选它的理由：

- **贴近机器**：每条指令最多 3 个操作数、一个运算，和大多数 RISC 机器指令（如 MIPS 的 `add $t, $s1, $s2`）形态一致，实践 4 翻译到 MIPS 时几乎是 1:1 映射。
- **与目标机无关**：IR 里没有寄存器名字（只有抽象的"临时变量 t1、t2"）、没有具体栈帧布局，换一台机器只改后端，IR 不动。
- **便于优化**：三地址指令彼此独立、数据流清楚，做常量传播、复写传播都很直接。
- **便于调试**：每行就一件事，肉眼能跟踪执行流；irvm 还能直接跑，对照期望输出。

> **关于 IR 的层次**：编译器里 IR 一般分高层（贴近源语言，如 AST 改造）、中层（三地址码/SSA）、低层（接近机器指令）。本项目用的是**低层线性三地址码**——已经摊平了控制流、展开了表达式，离 MIPS 只差寄存器分配和具体指令选择了。

## 2. 是什么——核心概念

### 2.1 三地址码 TAC

三地址码（three-address code）的规则很简单：**每条指令最多涉及三个"地址"**（变量、临时变量、常量），且每条指令只做一个运算。形如：

```
x := y op z        # 二元运算：x 等于 y 和 z 做 op（op 是 + - * /）
x := y             # 一元赋值/拷贝
x := op y          # 一元运算（本项目把一元负数拆成 0 - y）
```

为什么是"三地址"？因为一条指令的右边最多有两个操作数参与运算（y 和 z），左边放结果（x），加起来三个"地址"。"地址"在这里是个抽象词，可以是源程序变量、编译器生成的临时变量、或者立即数常量。

举个具体例子，源码 `a + b * c` 在语法树里是嵌套的：

```
        +
       / \
      a   *
         / \
        b   c
```

直接没法顺序执行。三地址码把它摊平成两条指令、引入一个临时变量 `t1` 暂存中间结果：

```
t1 := b * c      # 先算 b*c，存到 t1
t2 := a + t1     # 再算 a+t1，存到 t2（t2 就是整个表达式的值）
```

每条指令的操作数都是"现成的"（变量或已算出的临时变量），没有嵌套表达式，机器能一条一条顺序执行。这就是"摊平嵌套"的过程，也是整个翻译的核心动作。

### 2.2 17 种 IR 语句

Project_3.pdf 表 4.6 定义了 irsim IR 的 17 种语句，本项目全部实现（`C/src/ir/ir.h` 里的 `IRKind` 枚举）。按用途分七类：

| 类别 | 语句 | 形式 | 一行例子 |
|------|------|------|----------|
| 标号 / 函数定义 | `LABEL` | `LABEL label1 :` | `LABEL label2 :` |
| | `FUNCTION` | `FUNCTION name :` | `FUNCTION fib :` |
| 赋值 | `:=` | `x := y` | `v_n := #5` |
| 算术 | `+` `-` `*` `/` | `x := y op z` | `t1 := t2 * v_n` |
| 内存（取址 / 读 / 写） | `&` | `x := &y` | `t1 := &v_buf` |
| | `*`（读） | `x := *y` | `t3 := *t2` |
| | `*:=`（写） | `*x := y` | `*t7 := t8` |
| 控制流 | `GOTO` | `GOTO label` | `GOTO label1` |
| | `IF` | `IF x relop y GOTO label` | `IF v_n > #0 GOTO label2` |
| 函数调用 | `ARG` | `ARG x` | `ARG v_n` |
| | `CALL` | `x := CALL f` | `t5 := CALL fib` |
| | `PARAM` | `PARAM x` | `PARAM v_n` |
| | `RETURN` | `RETURN x` | `RETURN #0` |
| 声明 | `DEC` | `DEC x size` | `DEC v_buf 24` |
| 输入输出 | `READ` | `READ x` | `READ v_x` |
| | `WRITE` | `WRITE x` | `WRITE v_sum` |

其中 `relop` 是 6 种关系运算符 `< <= > >= == !=`（`ir.h` 里的 `Relop` 枚举）。

注意 `IF` 语句的形式是 `IF x relop y GOTO label`——**条件成立才跳，否则顺序执行下一条**。所以翻译 `if (a > b) ...` 时，要先 `IF a > b GOTO 真分支`，紧跟一条 `GOTO 假分支`，让条件不成立时跳走。这是控制流翻译的关键技巧，2.5 节会详细讲。

### 2.3 操作数：5 种"地址"

每条指令的操作数来自 `operand.h` 的 `OperandKind` 枚举，共 5 种：

| kind | 含义 | 打印形式 | 举例 |
|------|------|----------|------|
| `OP_VAR` | 源程序变量 | `v_<名字>` | `v_a`、`v_n`（源码变量 `a`、`n`） |
| `OP_TEMP` | 编译器临时变量 | `t<编号>` | `t1`、`t2`（自动递增编号） |
| `OP_CONST` | 立即数常量 | `#<值>` | `#5`、`#0` |
| `OP_LABEL` | 跳转标号 | `label<编号>` | `label1`、`label3` |
| `OP_ADDR` | 变量地址（传引用） | `&v_<名字>` | `&v_q`（结构体/数组参数传址） |

**为什么常量前面要加 `#`？** 因为变量名和常量长得可能一样——变量 `v_5` 和常量 `#5` 不加区分就分不清谁是谁。加了 `#` 前缀，解析器（包括我们的 irvm）一眼能看出"这是数值 5"还是"这是名为 v_5 的变量"。这是 irsim IR 的硬性约定。

**为什么变量名要加 `v_` 前缀？** 因为源程序的变量名（如 `a`、`sum`）可能和 IR 的关键字、标号、临时变量名冲突。加 `v_` 前缀把所有源变量统一放进 `v_xxx` 名字空间，和临时变量 `txxx`、标号 `labelxxx` 区分开，避免任何歧义。源变量 `a` 翻译后一律叫 `v_a`。

### 2.4 运行时栈帧与函数调用

函数调用是编译里最复杂的一件事，因为它涉及四件事：**保存返回地址、传递参数、收返回值、回收栈空间**。本项目用 `ARG / CALL / PARAM / RETURN` 这四条 IR 语句配合 irvm 的栈机制来实现。先讲 IR 怎么写，再讲 irvm 怎么执行。

假设源码是：

```c
int add(int a, int b) { return a + b; }
int main() {
    int s = add(3, 5);
}
```

翻译出的 IR 是：

```
FUNCTION add :
PARAM v_a              # 形参 a
PARAM v_b              # 形参 b
t1 := v_a
t2 := v_b
t3 := t1 + t2
RETURN t3              # 返回 a+b
FUNCTION main :
t4 := #3
ARG t4                 # 第一个实参 3（先 ARG）
t5 := #5
ARG t5                 # 第二个实参 5（后 ARG）
t6 := CALL add         # 调用，结果进 t6
v_s := t6              # s = add(3,5)
```

执行流程（irvm 的实现，见 `C/src/irvm/irvm.c`）：

1. **`ARG x`**：把实参值压进一个全局的**参数栈** `arg_stack`。注意是**正序压**：第一个 ARG 进栈底（arg_stack[0]），第二个进 arg_stack[1]，以此类推。
2. **`x := CALL f`**：irvm 在**调用栈** `call_stack` 上压一个调用帧（CallFrame），记录"返回后该执行哪一行（ret_pc）"、参数栈的当前高度、以及被调函数的形参个数 `param_count`。然后**新建一个作用域**（scope）给被调函数，把指令指针跳到 `f` 的入口。形参个数在第一遍扫描时已经从 `PARAM` 语句数出来了。
3. **`PARAM v`**：被调函数开头连续若干条 `PARAM`。irvm 从参数栈里取：`PARAM` 第 i 个形参取 `arg_stack[arg_base + i]`，其中 `arg_base = arg_top - param_count`。由于 ARG 是正序压的，所以**第一个 ARG 对应 PARAM[0]**，正好匹配源码"第一个实参给第一个形参"。这是本项目 ARG 用正序而非 PDF 教材逆序的关键原因——见 3.6 节。
4. **`RETURN x`**：把 `x` 的值算出来作为返回值，弹出调用帧，**恢复调用者的作用域**，把返回值写到 `x := CALL f` 那个左值（`v_s := CALL add` 里的 `v_s`），把参数栈高度恢复（弹掉用过的实参），指令指针跳回 ret_pc。

**为什么要 scope 栈？** 因为 C-- 支持递归（`fib` 调自己）。每次 CALL 新建一个 scope，递归调用时内层 scope 有自己的 `v_n`，不会覆盖外层的。RETURN 时把 scope 弹掉，调用者的 scope 自然恢复。这比"复制整个符号表"轻量得多。

### 2.5 翻译模式：两个核心函数 + 语法制导

整个翻译器的设计哲学是**语法制导翻译**（syntax-directed translation，SDT）：给每类语法结构（表达式、语句、条件）写一个翻译函数，函数按语法规则递归处理子节点，边处理边往一个全局 IR 链表里追加指令。所有翻译函数集中在 `C/src/ir/translate_exp.c` 和 `C/src/ir/translate.c`。

核心是**两个翻译函数**，对应 PDF 教材的两种翻译模式：

#### 2.5.1 `translate_exp(Node *exp, Operand place)` — 表达式翻译

把表达式 `exp` 的**值**算出来，放到 `place` 指定的位置（place 是个临时变量或源变量）。

```c
void translate_exp(Node *exp, Operand place);
```

比如源码 `c = a + b * 2`，翻译时调用方先 `new_temp()` 造一个 `place`（比如 `t1`），然后 `translate_exp(整条表达式, t1)`，函数内部递归：

```
translate_exp(a + b*2, t1):
    new_temp t2; translate_exp(a, t2)        →  t2 := v_a
    new_temp t3; translate_exp(b*2, t3):
        new_temp t4; translate_exp(b, t4)    →  t4 := v_b
        gen t3 := t4 * #2                    →  t3 := t4 * #2
    gen t1 := t2 + t3                        →  t1 := t2 + t3
```

实跑得到的真实 IR（`c = a + b * 2`，其中 `a=3, b=4`）：

```
t2 := v_a
t4 := v_b
t5 := #2
t3 := t4 * t5       # b * 2
t1 := t2 + t3       # a + (b*2)
v_c := t1           # 赋给 c
```

#### 2.5.2 `translate_cond(Node *exp, Operand l_true, Operand l_false)` — 条件翻译

把条件表达式 `exp` 翻译成**跳转**：条件成立跳到 `l_true` 标号，不成立跳到 `l_false` 标号。它**不产生值**，只产生控制流。

```c
void translate_cond(Node *exp, Operand l_true, Operand l_false);
```

比如 `if (x > 0) ...else...`，翻译时调用方造两个标号 `l1`（真分支入口）和 `l2`（假分支入口），然后 `translate_cond(x > 0, l1, l2)`：

```
translate_cond(x > 0, l1, l2):
    t1 := v_x
    IF t1 > #0 GOTO l1     # 条件成立跳 l1
    GOTO l2                # 否则跳 l2
```

**为什么条件翻译用"传真假标号"而不是回填（backpatching）？** 教材里讲控制流翻译有两种流派：

- **回填**：先翻译时不填跳转目标，把"待回填的跳转"记在链表里，等后面确定了目标地址再统一回填。优点是生成的代码紧凑（没有冗余跳转），缺点是要维护跳转链表、实现复杂。
- **继承属性传真假标号**（本项目用）：调用 `translate_cond` 时就把真/假标号作为继承属性传进去，翻译时直接 `GOTO 已知标号`。优点是实现直观、和教材的 SDT 模式一一对应，缺点是可能产生冗余跳转（比如 `IF ... GOTO l2` 紧跟 `GOTO l2`，逻辑上重复）。

本项目按 PDF 教学法选后者：**简单直观，虽然可能有冗余跳转但语义完全正确**。冗余跳转留给实践 5 的代码优化去清理。教学项目优先把原理讲透、把正确性做扎实，不追求生成的 IR 最精简。

#### 2.5.3 place 哨兵：丢弃结果的约定

有个细节：表达式有时出现在"不需要值"的地方，比如纯表达式语句 `a + b;`（算了不用）或函数调用 `f();`。这时 `place` 该填什么？本项目用一个**哨兵**：传一个 `kind == OP_CONST` 的 Operand（值是什么无所谓）。翻译函数内部用 `place_wants(place)` 判断（`place.kind != OP_CONST` 为真才需要结果）：

```c
static int place_wants(Operand place) {
    return place.kind != OP_CONST;
}
```

合法的 place 永远不会是 `OP_CONST`（结果不会存到常量里），所以这个约定不会误判。翻译函数在每处生成结果前都先问 `place_wants`，不需要就跳过那条赋值——但**副作用必须保留**（比如函数调用 `f();` 即使不用返回值，`ARG` 和 `CALL` 仍要生成，因为 `f` 可能有副作用）。

### 2.6 全局 IR 链表 + `gen()` 系列

所有翻译函数都不"返回"自己生成的 IR 片段，而是直接往一个**全局链表**（`ir_head` / `ir_tail`，定义在 `ir.c`）里追加。每条指令通过 `gen_xxx()` 系列函数构造并挂到链表尾：

```c
extern IR ir_head, ir_tail;          /* 全局链表头/尾 */
void gen_assign(Operand x, Operand y);               /* x := y */
void gen_binop(IRKind, Operand x, Operand y, Operand z);  /* x := y op z */
void gen_label(Operand label);       /* LABEL label : */
void gen_if(Operand x, Relop, Operand y, Operand label);  /* IF x relop y GOTO label */
void gen_arg(Operand x);             /* ARG x */
void gen_call(Operand x, const char *fname);   /* x := CALL fname */
/* ... 共 16 个 gen 函数，每种 IRKind 一个 ... */
```

最后 `main_ir.c` 调用 `ir_print(out)` 遍历整条链表，按 PDF 表 4.6 格式逐行打印。

**为什么用全局链表而不是返回 IR 片段拼接？** 因为拼接式（每个翻译函数返回一段 IR，调用方负责拼）要在函数间不断传递和连接链表节点，代码冗长易错。全局链表让每个 `gen` 调用就是"往尾部加一条"，翻译函数彼此完全解耦——递归调用子表达式时，子函数往链表尾加的指令自然排在父函数之前（因为父函数后面才 gen），顺序天然正确。代价是函数间没有显式的数据流（"我生成哪几条"看不见），但对教学项目这点不透明可以接受，换来的是极简的实现。

## 3. 怎么做——实现

### 3.1 IR 数据结构

IR 的核心数据结构在 `ir.h` 和 `operand.h`，都很简单。

**指令节点**（`struct IR_`）：

```c
typedef enum {
    IR_LABEL, IR_FUNCTION, IR_ASSIGN,
    IR_ADD, IR_SUB, IR_MUL, IR_DIV,
    IR_ADDR, IR_LOAD, IR_STORE,
    IR_GOTO, IR_IF, IR_RETURN,
    IR_DEC, IR_ARG, IR_CALL, IR_PARAM,
    IR_READ, IR_WRITE
} IRKind;          /* 17 种语句 */

typedef struct IR_ *IR;
struct IR_ {
    IRKind kind;
    Operand x, y, z;       /* 三个操作数（按 kind 用不同字段；不用全）*/
    char  *name;           /* FUNCTION/CALL 的函数名；GOTO/LABEL/IF 的标号名 */
    Relop  relop;          /* IF 的关系运算符 */
    int    size;           /* DEC 的字节数 */
    struct IR_ *next;      /* 链表下一节点 */
};
```

一个节点复用：不同 kind 用不同字段。比如 `IR_ADD` 用 `x := y + z`（x/y/z 三个操作数），`IR_RETURN` 只用 `x`（返回值），`IR_DEC` 用 `x`（变量）和 `size`（字节数）。这样省去了为每种指令单独定义结构体的繁琐。

**操作数**（`Operand`，`operand.h`）：

```c
typedef enum { OP_VAR, OP_TEMP, OP_CONST, OP_LABEL, OP_ADDR } OperandKind;
typedef struct {
    OperandKind kind;
    char *name;     /* OP_VAR/TEMP/LABEL/ADDR 的名字 */
    int   value;    /* OP_CONST 的值 */
} Operand;
```

操作数生成器（内部用全局计数器自动编号）：

```c
Operand new_temp(void);            /* OP_TEMP，name = "t1", "t2", ... */
Operand new_label(void);           /* OP_LABEL，name = "label1", ... */
Operand new_const(int value);      /* OP_CONST */
Operand new_var(const char *src);  /* OP_VAR，name = "v_" + src */
```

`new_temp()` 每调一次 `g_temp_counter` 自增，保证临时变量编号全局唯一不重名；`new_label()` 同理。

### 3.2 顶层结构翻译（`translate.c`）

程序结构是 `Program → ExtDefList → ExtDef → Specifier FunDec CompSt`。翻译从 `translate_program(root)` 开始：

```c
void translate_program(Node *root) {
    prepass_check(root);              /* 先扫一遍，遇不支持结构置 cannot_translate */
    if (cannot_translate) {
        fprintf(stderr, "Cannot translate: ...\n");
        return;                       /* 不产生任何 IR */
    }
    while (extdeflist 还有节点) {
        translate_extdef(每个 ExtDef);   /* 函数定义 */
    }
}
```

`translate_extdef` 判断 ExtDef 是不是函数定义（第二个子节点是 `FunDec`），是就调 `translate_fundef`。`translate_fundef` 干三件事：

1. `gen_function(fname)` — 输出 `FUNCTION fib :`；
2. 处理形参：每个 ParamDec 调 `tr_vardec_to_type` 算出类型，`translate_insert_var` 把变量名→类型登记进**翻译用的轻量符号表**（`vtab`，pjw 哈希），并 `gen_param` 输出 `PARAM v_<name>`；
3. `translate_compst(compst)` — 翻译函数体。

`translate_compst` 处理 `CompSt → LC DefList StmtList RC`：

- **DefList**（局部变量声明）：每个 `Dec` 算类型并登记。**数组/结构体类型**还要 `gen_dec(v_name, type_size)` 输出 `DEC v_name <字节数>`，告诉 irvm 预留多少内存。普通 `int` 不需要 DEC。
- **StmtList**（语句序列）：逐条 `translate_stmt`。

注意一个陷阱（`translate.c` 注释里写了）：`addChild` 跳过 NULL 子节点，所以**空的 DefList / StmtList 不在 children 数组里**，不能按固定下标 `children[1]` 取，必须按节点名遍历查找。这是和"教科书式"树遍历的一个实际差异。

**为什么翻译器要自带一张轻量符号表 `vtab`，不直接复用实践 2 的 `symtab`？** 因为实践 2 的 `symtab.c` 带了大量语义错误检查（`sem_error`、`print_error`），链接进来会和 `main_ir` 的 `main` 函数冲突（`sem.c` 也有自己的 `main`）。翻译器要的是"查变量类型用于寻址"，不需要任何错误检查（程序已经过实践 2 验证语义正确），所以另写一张极简的表。同理，结构体类型表 `struct_type_table` 也独立维护。

### 3.3 基础表达式翻译（`translate_exp.c`）

`translate_exp(Node *exp, Operand place)` 按 `exp` 的产生式分情况处理，对应 PDF 表 4.1。核心分支：

**`Exp → INT`**：直接赋常量。
```
源码: 42
IR:   place := #42          （仅当 place 需要结果）
```

**`Exp → ID`**：查符号表得 `v_<名字>`，赋值给 place。
```
源码: a
IR:   place := v_a
```

**`Exp → Exp ASSIGNOP Exp`**（赋值）：右值先算到临时变量 `t`，再赋给左值。左值分两种：
- 普通变量 `ID`：`gen_assign(v_lhs, t)`，即 `v_lhs := t`。
- 数组元素 / 结构体域（`a[i]`、`s.f`）：先 `addr = translate_addr_general(lhs)` 求地址，再 `gen_store(addr, t)`，即 `*addr := t`（写内存）。

**`Exp → Exp PLUS/MINUS/STAR/DIV Exp`**（二元算术）：两个子表达式各算到一个临时变量，再 gen 一条 `place := t1 op t2`。
```
源码: a + b * 2
IR:   t2 := v_a  ;  t4 := v_b  ;  t3 := t4 * #2  ;  t1 := t2 + t3
```

**`Exp → MINUS Exp`**（一元负）：没有"一元减"指令，拆成 `0 - x`。
```
源码: -x
IR:   t := v_x  ;  place := #0 - t
```

**`Exp → LP Exp RP`**（括号）：括号不产生代码，直接 `translate_exp(内部 Exp, place)` 透传。

**完整例子**：源码 `c = a + b * 2`（设 `a=3, b=4`），从声明到赋值完整 IR（实跑）：

```
t2 := v_a              # a
t4 := v_b              # b
t5 := #2               # 2
t3 := t4 * t5          # b * 2 = 8
t1 := t2 + t3          # a + (b*2) = 11
v_c := t1              # c = 11
```

irvm 跑出来 `write(c)` 输出 `11`，与 `3 + 4*2 = 11` 一致。

> **观察：IR 有冗余赋值**。比如 `t2 := v_a` 再 `t1 := t2 + t3`，逻辑上可以合并成 `t1 := v_a + t3`。本项目当前实现先把每个操作数"搬"到一个新临时变量，再做运算，是 PDF 教材"先翻译子表达式到临时变量"的标准写法。这种冗余对正确性无影响，留给实践 5 优化清理。

### 3.4 控制流语句翻译（`translate_stmt`，`translate.c`）

`translate_stmt` 对应 PDF 表 4.2，处理 `IF / WHILE / RETURN / CompSt`。核心技巧是**三标号法**：

**`Stmt → IF LP Exp RP Stmt`**（无 else）：用两个标号 `l1`（真分支）、`l2`（出口，假分支直接落到这）。
```
translate_cond(条件, l1, l2)   # 条件真跳 l1，假跳 l2
LABEL l1 :
    翻译 then 语句
LABEL l2 :                     # 整体出口
```

**`Stmt → IF LP Exp RP Stmt ELSE Stmt`**（有 else）：用三个标号。then 分支执行完要跳过 else，所以多加一个出口标号 `l3`。
```
translate_cond(条件, l1, l2)   # 真跳 l1，假跳 l2
LABEL l1 :
    翻译 then 语句
    GOTO l3                    # then 结束跳出口
LABEL l2 :
    翻译 else 语句
LABEL l3 :                     # 整体出口
```

**`Stmt → WHILE LP Exp RP Stmt`**：三个标号，`l1` 循环头、`l2` 循环体入口、`l3` 循环出口。
```
LABEL l1 :                     # 每轮回这判断条件
translate_cond(条件, l2, l3)   # 真进循环体 l2，假跳出 l3
LABEL l2 :
    翻译循环体
    GOTO l1                    # 回头再判条件
LABEL l3 :                     # 循环出口
```

**完整例子**：经典 PDF 例题 `sgn(x)`（符号函数，`x>0` 返回 1，`x<0` 返回 -1，否则返回 0），源码：

```c
int sgn(int x) {
    if (x > 0) return 1;
    else if (x < 0) return -1;
    else return 0;
}
int main() {
    int a;
    a = read();
    write(sgn(a));
    return 0;
}
```

实跑得到的完整 IR：

```
FUNCTION sgn :
PARAM v_x
t1 := v_x
t2 := #0
IF t1 > t2 GOTO label1          # if (x > 0)
GOTO label2
LABEL label1 :
t3 := #1
RETURN t3                       # return 1
GOTO label3                     # then 结束跳出口
LABEL label2 :                  # else 分支
t4 := v_x
t5 := #0
IF t4 < t5 GOTO label4          # if (x < 0)
GOTO label5
LABEL label4 :
t7 := #1
t6 := #0 - t7                   # -1（0 - 1）
RETURN t6                       # return -1
GOTO label6
LABEL label5 :
t8 := #0
RETURN t8                       # return 0
LABEL label6 :
LABEL label3 :                  # 整体出口
FUNCTION main :
READ t9                         # read(a)（注：a = read() 形式）
v_a := t9
t11 := v_a
ARG t11
t10 := CALL sgn                 # sgn(a)
WRITE t10                       # write(sgn(a))
t12 := #0
RETURN t12
```

irvm 实跑（输入分别 `7` / `-3` / `0`）：

```
输入 7  → 输出 1
输入 -3 → 输出 -1
输入 0  → 输出 0
```

三个分支全对。这里能清楚看到 IF-ELSE 的三标号（`label1` 真分支、`label2` 假分支、`label3` 出口）和嵌套的 else-if 又用了一组 `label4/5/6`。

### 3.5 条件表达式翻译（`translate_cond`，`translate.c`）

`translate_cond` 对应 PDF 表 4.3，核心是**短路求值**（short-circuit evaluation）。

**`Exp → Exp RELOP Exp`**（关系运算）：直接翻译成一条 IF + 一条 GOTO。
```
源码: a < b
IR:   t1 := v_a ; t2 := v_b ; IF t1 < t2 GOTO l_true ; GOTO l_false
```

**`Exp → NOT Exp`**（逻辑非）：真假标号互换。
```
源码: !(a < b)
IR:   translate_cond(a < b, l_false, l_true)   # 注意参数互换
```

**`Exp → Exp AND Exp`**（逻辑与）：**短路**——左侧为假直接跳 `l_false`，左侧为真才求右侧。
```
源码: a && b
IR:
    translate_cond(a, l_mid, l_false)   # 左侧假→整体假
    LABEL l_mid :
    translate_cond(b, l_true, l_false)  # 左侧真才看右侧
```

**`Exp → Exp OR Exp`**（逻辑或）：**短路**——左侧为真直接跳 `l_true`，左侧为假才求右侧。
```
源码: a || b
IR:
    translate_cond(a, l_true, l_mid)    # 左侧真→整体真
    LABEL l_mid :
    translate_cond(b, l_true, l_false)  # 左侧假才看右侧
```

**为什么用短路求值？** 两个理由：

1. **语义正确**。考虑 `a != 0 && b / a > 1`：如果 `a == 0`，不短路的话会去算 `b / a`，触发除零错误。短路求值下，`a == 0` 时左侧为假，整体直接判假，根本不碰右侧的除法。C 语言的 `&&` `||` 就是短路语义，必须这么翻才对。
2. **省算**。如果一侧已经能定结果（`a && b` 中 `a` 假），另一侧就算了也白算（无副作用时），短路直接跳过。

> **数值化处理**：条件表达式有时出现在"要值"的地方（如 `x = (a > b);`），这时用 `translate_exp` 数值化为 0/1：先 `place := #0`，再 `translate_cond(exp, l_true, l_false)`，`l_true` 处 `place := #1`，`l_false` 处啥也不写（保持 0）。控制流上下文（`if`/`while` 的条件）则直接调 `translate_cond` 不数值化。

### 3.6 函数调用：ARG 正序与 read/write 特判

函数调用的实参翻译在 `translate_args`（`translate.c`）：

```c
void translate_args(Node *args, Operand *arg_list, int *arg_count);
```

它把 `Args → Exp COMMA Args | Exp` 链表里的实参，**正序**填进 `arg_list[0..n-1]`（`arg_list[0]` 是第一个实参）。然后调用方循环 `gen_arg(arg_list[i])` 顺序输出 `ARG`。

**为什么 ARG 用正序（`arg_list[0]` 先 ARG），而不是 PDF 教材的逆序？** 因为我们的 irvm 的 PARAM 取参约定是 `PARAM[i] = arg_stack[arg_base + i]`——第一个 ARG（arg_stack[0]）对应 PARAM[0]（第一个形参）。PDF 教材原本假设的运行时模型是"逆序压栈、PARAM 从栈顶取"，所以教材让 ARG 逆序。但本项目 irvm 选择了正序模型（更直观、和源码顺序一致），所以 ARG 也跟着正序。**这是本项目相对 PDF 教材的一个有意偏离，IR 和 irvm 必须配套，否则参数会错位。**

**read / write 特判**：C-- 内置的 `read()` 和 `write(x)` 不翻译成 `CALL read`/`CALL write`（它们不是用户定义函数，irvm 里也没有这两个函数表项），而是直接翻译成 `READ` / `WRITE` 指令：

- `read()`：`gen_read(place)`，即 `READ place`。注意它**必须有 place**（要把读到的值放哪），所以源码里要写成 `a = read();` 形式。单独 `read(a);`（语句形式）会被当表达式语句，place 是哨兵 `OP_CONST`，导致 `READ` 不生成——这是个值得注意的用法约束。
- `write(x)`：先把参数 `x` 翻译到临时变量 `t`，再 `gen_write(t)`，即 `WRITE t`。`write` 返回值约定为 0（`place := #0`）。

经典递归例子 `fact(n)`（阶乘），源码：

```c
int fact(int n) {
    if (n == 0) return 1;
    return fact(n - 1) * n;
}
int main() {
    int n;
    n = read();
    write(fact(n));
    return 0;
}
```

实跑 IR：

```
FUNCTION fact :
PARAM v_n
t1 := v_n
t2 := #0
IF t1 == t2 GOTO label1        # if (n == 0)
GOTO label2
LABEL label1 :
t3 := #1
RETURN t3                      # return 1
LABEL label2 :
t8 := v_n
t9 := #1
t7 := t8 - t9                  # n - 1
ARG t7                         # ARG 正序：n-1
t5 := CALL fact                # fact(n-1)
t6 := v_n
t4 := t5 * t6                  # fact(n-1) * n
RETURN t4
FUNCTION main :
READ t11                       # n = read()
v_n := t11
t15 := v_n
ARG t15
t14 := CALL fact               # fact(n)
WRITE t14                      # write(fact(n))
t12 := #0
RETURN t12
```

irvm 实跑（输入 `5`）：

```
120
```

`fact(5) = 5*4*3*2*1 = 120`，递归调用链正确。

### 3.7 数组寻址（`translate_addr.c`）

数组翻译的核心是**地址计算**。`a[i]` 的元素地址 = 数组首地址 + 下标 × 元素宽度（字节）：

```
addr = &v_a + i * #elem_size
```

读元素用 `*addr`（`gen_load`），写元素用 `*addr := value`（`gen_store`）。

**元素宽度 `type_size`**（`translate_addr.c`）递归计算：

```c
int type_size(Type t) {
    if (t->kind == BASIC) return 4;                  /* int = 4 字节 */
    if (t->kind == ARRAY)
        return type_size(t->u.array.elem) * t->u.array.size;  /* 元素宽度 × 元素数 */
    if (t->kind == STRUCTURE) {
        int sum = 0;                                  /* 逐域累加 */
        for (FieldList f = t->u.structure; f; f = f->tail)
            sum += type_size(f->type);
        return sum;
    }
}
```

- `int` 占 4 字节；
- 一维 `int a[6]` 占 `4 * 6 = 24` 字节，元素宽度 4；
- 二维 `int g[2][3]` 占 `4 * 3 * 2 = 24` 字节，第一维元素宽度 `4*3=12`（一行）。

**一维寻址**（`translate_addr_general` 的 LB 分支）：

```
源码: a[i]
IR:
    base := &v_a              # 数组首地址
    t_idx := translate_exp(i) # 下标值
    t_off := t_idx * #4       # 偏移 = 下标 × 元素宽度（int=4）
    addr := base + t_off      # 元素地址
    # 读：result := *addr
    # 写：*addr := value
```

**DEC 分配内存**：函数体里声明 `int buf[6];` 时，`translate_compst` 检测到类型是 ARRAY，输出 `DEC v_buf 24`（24 = 6×4 字节）。irvm 收到 DEC 就在模拟内存 `mem[]` 里划出 24 个槽位，把起始地址记到 `v_buf` 的 `mem_addr`。之后 `&v_buf` 取的就是这个起始地址。

完整例子（一维数组填充 + 读取，A-3 用例核心片段）：

```
源码: for (i=0;i<6;i++) buf[i] = i*i+3;  然后 write(buf[2]);
IR:
DEC v_buf 24                            # 给 buf 开 24 字节
...
t4 := &v_buf ; t6 := v_i * #4 ; t7 := t4 + t6   # &buf[i]
t9 := v_i * v_i ; t8 := t9 + #3                 # i*i+3
*t7 := t8                                        # buf[i] = i*i+3
...
t23 := &v_buf ; t26 := t23 + #2*#4 ; WRITE *t26  # buf[2]=2*2+3=7
```

irvm 输出 `7`，正确。

### 3.8 选做 4.1：结构体寻址（`translate_addr_general` 的 DOT 分支）

结构体 `s.f` 的域地址 = 结构体首地址 + **域偏移**。域偏移 = 该域**前面**所有域的 `type_size` 之和（按声明顺序累加）：

```c
Type struct_field_offset(FieldList fields, const char *fname, int *out_offset) {
    int off = 0;
    while (fields) {
        if (strcmp(fields->name, fname) == 0) {
            *out_offset = off;       /* 找到了，输出偏移 */
            return fields->type;
        }
        off += type_size(fields->type);   /* 累加前面域的宽度 */
        fields = fields->tail;
    }
    return NULL;
}
```

比如 `struct Rect { int width; int height; }`：`width` 偏移 0（前面没域），`height` 偏移 4（前面一个 int 占 4 字节）。所以 `s.height` 的地址 = `&v_s + #4`。

**结构体类型表独立于实践 2 的 symtab**：如 3.2 节所述，翻译器自带 `struct_type_table`（`register_struct` / `lookup_struct`），不复用 `symtab.c`，避免 `sem_error` 和 `main` 冲突。

**结构体形参按引用传递**：结构体变量传给函数时，不复制整个结构体（太重），而是传地址——`ARG &v_s`。函数内形参的值就是这个地址，再 `&v_p + #offset` 访问域。这是 PDF 4.3.1 规则 10 的要求。`translate_args` 检测到实参是结构体（或数组）变量时，生成 `OP_ADDR`：

```c
if (e->type->kind == STRUCTURE || e->type->kind == ARRAY) {
    addr.kind = OP_ADDR;          /* 打印成 "&v_<name>" */
    addr.name = strdup(e->ir_name);
    arg_list[0] = addr;
}
```

完整例子（E1-1，`struct Rect` 算面积 area=width*height=6*7=42，半周长 halfPerim=width+height=6+7=13）：

```
源码核心:
    struct Rect r; int area, halfPerim;
    r.width = 6; r.height = 7;
    area = r.width * r.height;
    halfPerim = r.width + r.height;
    write(area); write(halfPerim);
IR（实跑）:
FUNCTION main :
DEC v_r 8                            # struct Rect 占 4+4=8 字节
t1 := &v_r
t2 := t1 + #0                        # r.width 地址（偏移 0）
t3 := #6
*t2 := t3                            # r.width = 6
t4 := &v_r
t5 := t4 + #4                        # r.height 地址（偏移 4）
t6 := #7
*t5 := t6                            # r.height = 7
t10 := &v_r ; t11 := t10 + #0 ; t8 := *t11    # 读 r.width = 6
t12 := &v_r ; t13 := t12 + #4 ; t9 := *t13    # 读 r.height = 7
t7 := t8 * t9                        # area = 6*7 = 42
v_area := t7
...
WRITE v_area                         # 42
WRITE v_halfPerim                    # 13
```

irvm 实跑输出 `42`、`13`，正确。

### 3.9 选做 4.2：高维数组 + 数组参数

**高维数组寻址**：`g[i][j]` 的地址 = `&v_g + i × #(行宽) + j × #4`，递归处理。`translate_addr_general` 的 LB 分支递归求 base 的地址和元素类型：

```c
/* g[i][j]：base = g[i]（一个 LB 节点），递归求 g[i] 的地址和"下一维数组类型" */
Operand base_addr = translate_addr_general(exp->children[0], &base_type);
Type elem_type = base_type->u.array.elem;   /* 下一维元素类型 */
/* 偏移 = j × type_size(elem_type) */
gen_binop(IR_MUL, t_off, t_idx, new_const(type_size(elem_type)));
gen_binop(IR_ADD, a, base_addr, t_off);
```

比如 `int g[2][3]`（`type_size` 内层 `int`=4，外层行宽 `4*3=12`）：

```
源码: g[0][2] = 30;  sumCorner = g[0][0] + g[1][2];
IR:
t9 := &v_g ; t10 := #0 ; t11 := t10 * #12 ; t12 := t9 + t11   # &g[0]（行偏移 0*12）
t13 := #1 ; t14 := t13 * #4 ; t15 := t12 + t14                # g[0][1]（列偏移 1*4）
...
t17 := &v_g ; t18 := #1 ; t19 := t18 * #12 ; t20 := t17 + t19 # &g[1]（行偏移 1*12）
t21 := #2 ; t22 := t21 * #4 ; t23 := t20 + t22                # g[1][2]（列偏移 2*4）
```

完整例子（E2-1，`int g[2][3]`，`sumCorner = g[0][0] + g[1][2] = 1 + 30 = 31`，`scaled = g[0][1] * 3 = 2*3 = 6`）：

```
IR（实跑，赋值部分）:
DEC v_g 24                    # 2×3×4 = 24 字节
t1 := &v_g ; t2:=#0; t3:=t2*#12; t4:=t1+t3 ; t5:=#0; t6:=t5*#4; t7:=t4+t6 ; t8:=#1 ; *t7:=t8
# g[0][0] = 1（行偏移 0*12 + 列偏移 0*4）
t17 := &v_g ; t18:=#1; t19:=t18*#12; t20:=t17+t19 ; t21:=#2; t22:=t21*#4; t23:=t20+t23 ; t24:=#30 ; *t23:=t24
# g[1][2] = 30（行偏移 1*12 + 列偏移 2*4）
```

irvm 实跑输出 `31`、`6`，正确。

#### 一个真实的坑：维度嵌套顺序（重要）

高维数组这里踩过一个很有教学价值的坑，必须记下来。

**现象**：最初实现 `tr_vardec_to_type`（把 `VarDec` 节点翻译成数组类型）时，`int g[2][3]` 生成的 IR 里 `g[0][2]` 和 `g[1][0]` 地址会冲突，导致数据互相覆盖、结果错乱。

**根因（Why）**：Bison 处理 `VarDec → VarDec LB INT RB` 是**左递归**归约。源码 `int g[2][3]` 里，`[2]` 先写、`[3]` 后写。左递归下，**后写的维度最后归约，最后归约的在语法树最外层（根）**。所以语法树结构是：

```
Root[3]  ← 源码最后写的 [3]，在根
  └─ Mid[2]  ← 源码先写的 [2]
       └─ ID(g)
```

如果翻译时"看到根节点的 `[3]` 就 `new_array(内层, 3)`"，得到的类型是 `array(3, array(2, int))`——把根维度 `[3]` 当成了最外层。但 **C 语义里源码先写的维度才是最外层**：`int g[2][3]` 应该是 `array(2, array(3, int))`（2 行 3 列）。两者正好相反！

类型搞反的直接后果：寻址时 `i` 维的步长算错。正确类型下 `g[i]` 的步长是"一整行"（`3×4=12` 字节）；类型反了的话步长算成 `2×4=8`，于是 `g[0][2]`（应该在第 0 行末尾）和 `g[1][0]`（应该在第 1 行开头）被算到同一个地址，写一个覆盖另一个。

**修复（How）**：先遍历整棵 VarDec 子树，**收集所有维度到一个数组** `dims[]`，注意收集顺序——`dims[0]` 是根节点的维度（源码最后写的，最内层），`dims[n-1]` 是最靠近 ID 的维度（源码最先写的，最外层）。然后**按 `i=0..n-1` 依次 `new_array` 包裹**：先包最内层，最后包最外层：

```c
Type tr_vardec_to_type(Node *vardec, Type base) {
    int dims[32]; int n = 0;
    Node *p = vardec;
    while (p && p->nchild == 4) {       /* VarDec LB INT RB */
        dims[n++] = (int)p->children[2]->ival;   /* 根→叶：最后写的在前 */
        p = p->children[0];
    }
    Type t = base;
    for (int i = 0; i < n; i++)          /* 从内层往外层包 */
        t = new_array(t, dims[i]);
    return t;
}
```

`int g[2][3]`：收集到 `dims = [3, 2]`（根 3 在前），循环先 `new_array(int, 3)` 得 `array(3,int)`，再 `new_array(array(3,int), 2)` 得 `array(2, array(3, int))`，与 C 语义一致。修复后 `g[i][j]` 的行步长正确算成 `3×4=12`，地址不再冲突。

**教训**：Bison 左递归产生的语法树，节点深度顺序和源码书写顺序是**反的**，处理"维度/列表"这类有顺序语义的结构时必须反转。这是个典型的"语法树形状 vs 源码语义"的陷阱，值得所有写递归下降/Bison 翻译的人警惕。

### 3.10 Cannot translate：prepass 与"要么全有要么全无"

必做模式（`cmmc_ir`，不开 `ENABLE_STRUCT` / `ENABLE_HIGH_DIM_ARRAY` 宏）遇到结构体、高维数组、数组参数时**不翻译**，但要保证输出"要么完整 IR 要么零 IR"——不能翻译一半崩了留半截 IR，那样下游 irvm 会跑出错乱结果。

实现上用**两遍扫描**（`prepass_check`，`translate_addr.c`）：在真正翻译前先扫一遍所有函数的形参和局部变量类型，遇到不支持的（`param_unsupported` / `var_unsupported`）就把全局标志 `cannot_translate` 置 1：

```c
void translate_program(Node *root) {
    prepass_check(root);              /* 第一遍：检测 */
    if (cannot_translate) {
        fprintf(stderr, "Cannot translate: Code contains variables of "
                        "multi-dimensional array type or parameters of "
                        "array type.\n");
        return;                       /* 直接返回，不调 translate_extdef */
    }
    /* 第二遍：真正翻译 */
    while (...) translate_extdef(...);
}
```

**为什么用 prepass（前置扫描）而不是边翻译边报？** 因为边翻译边报的话，已经输出到 `ir_head` 链表的指令没法"收回"——`ir_print` 照样会打印前半截 IR。prepass 在生成任何 IR 之前就判死刑，`translate_program` 直接 return，`ir_head` 保持空，`ir_print` 啥也不输出。这保证下游要么拿到完整正确的 IR、要么拿到空（外加 stderr 的统一提示），绝不会拿到半截。

判定规则（受两个宏控制）：

| 类型 | 必做（都不开宏） | 只开 STRUCT | 只开 HIGH_DIM | 两个都开 |
|------|------|------|------|------|
| 普通变量 / 一维数组局部 | OK | OK | OK | OK |
| 结构体变量（局部/形参） | 不可 | OK | 不可 | OK |
| ≥2 维数组局部 | 不可 | 不可 | OK | OK |
| 数组形参（任意维） | 不可 | 不可 | OK | OK |

### 3.11 双可执行策略：条件编译 DRY

本项目产出两个可执行：

- **`cmmc_ir`**（必做）：不开任何宏。遇结构体/高维数组/数组参数时输出 `Cannot translate`。
- **`cmmc_ir_full`**（选做）：开 `-DENABLE_STRUCT -DENABLE_HIGH_DIM_ARRAY`，全功能。

两者**共享同一份源码**（`translate.c` / `translate_exp.c` / `translate_addr.c`），靠 `#ifndef ENABLE_STRUCT` / `#ifndef ENABLE_HIGH_DIM_ARRAY` 条件编译切换行为：

```c
static int param_unsupported(Type t) {
#ifndef ENABLE_STRUCT
    if (type_has_struct(t)) return 1;     /* 不开 STRUCT：结构体形参不可 */
#endif
#ifndef ENABLE_HIGH_DIM_ARRAY
    if (t && t->kind == ARRAY) return 1;  /* 不开 HIGH_DIM：数组形参不可 */
#endif
    return 0;
}
```

两个宏都开时，`prepass_check` 直接 `return`（什么都不检测），`param_unsupported` / `var_unsupported` 永远返回 0，全部翻译。

**为什么用条件编译而非两套代码？** DRY（Don't Repeat Yourself）原则。表达式翻译、控制流翻译、寻址计算这些核心逻辑两套都要，只有"支持不支持结构体/高维"这一处分叉。条件编译让分叉点用 `#ifdef` 局部表达，避免维护两份几乎相同的代码。代价是阅读时要留意 `#ifdef`，但相比重复代码的维护成本小得多。

### 3.12 irvm 设计要点（`irvm.c`）

`irvm` 是配套的 IR 虚拟机（不是交付物，是验证工具），负责把翻译器输出的 `.ir` 文本读进来、按语句执行、把 `WRITE` 的值打到 stdout。核心设计：

**三张表**：

- **scope 栈**（`Scope`）：每个函数调用新建一个 scope（链式哈希表 + parent 指针），变量查找从当前 scope 往上找 parent。`CALL` 建 scope、`RETURN` 弹 scope，天然支持递归。
- **label 表**（`label_table`）：`name → 行号`，全局，第一遍扫描时填。
- **func 表**（`func_table`）：`name → 入口行号 + 形参个数 + 形参名列表`。第一遍扫描时，对每个 `FUNCTION name :`，往下数连续的 `PARAM` 行算出 `param_count`。

**两个栈**：

- **arg_stack**（参数栈）：`ARG x` 把 `x` 求值后压栈。`CALL` 时记下 `arg_base = arg_top - param_count`。`PARAM v` 取 `arg_stack[arg_base + i]`，`i` 从 0 开始（`param_taken` 计数）。正序匹配。
- **call_stack**（调用栈）：每层 `CallFrame` 记录返回地址 `ret_pc`、参数栈保存点 `arg_top_saved`、调用者 scope、返回值左值 `ret_lvalue`。

**执行流程**：从 `main` 函数入口开始（`func_get("main")`），逐行解析语句。`GOTO`/`IF` 改 `pc`；`RETURN` 弹调用帧恢复现场；`x := CALL f` 压调用帧、新建 scope、跳 `f` 入口；赋值语句解析 `lhs := rhs1 op rhs2` 或 `lhs := rhs1`。

#### 一个真实的坑：DEC 字节分配（重要）

irvm 这里也踩过一个 bug，同样有教学价值。

**现象**：相邻的两个数组（比如 `int a[6]; int b[6];`，各 24 字节）会互相覆盖，`a[5]` 的值被 `b[0]` 冲掉。

**根因（Why）**：最初 DEC 的实现是按 `bytes / sizeof(long)` 分配内存槽位：

```c
/* 错误版本 */
int slots = bytes / sizeof(long);   /* 64 位机上 sizeof(long)=8，24/8=3 个槽 */
```

理由是"模拟内存 `mem[]` 是 `long` 数组，一个槽放一个 long，按 long 分配省空间"。但**翻译器算地址时用的是字节偏移**：`a[i]` 地址 = `&v_a + i*4`（4 是 `int` 的字节宽度）。`i=0..5` 时偏移是 `0, 4, 8, 12, 16, 20`，跨越 0~20 字节。如果 irvm 只给 `a` 分了 3 个槽（按 `24/8`），那 `a[3]`（偏移 12）就跑到分配范围外了——刚好覆盖到下一个数组 `b` 的内存。

核心矛盾：**翻译器的寻址用"字节"为单位，irvm 的内存分配也得用"字节"为单位，两者必须一致**。`bytes/sizeof(long)` 在 64 位机上除以 8，等于只分了 1/8 的空间，相邻数组必然挤在一起互相踩。

**修复（How）**：DEC 直接按字节分配，**1 字节 = 1 个 mem 槽**：

```c
/* 正确版本 */
int slots = bytes;              /* 24 字节 = 24 个槽 */
if (slots < 1) slots = 1;
if (mem_top + slots > MEM_SIZE) { ... }
v->mem_addr = mem_top;
mem_top += slots;
```

这样 `a` 占 `mem[addr_a .. addr_a+23]`，`b` 占 `mem[addr_b .. addr_b+23]`（`addr_b = addr_a + 24`），各管各的，互不干扰。代价是 `mem[]` 要开大点（本项目开了 1M long，足够）。

**教训**：当两个模块（翻译器和虚拟机）约定了一种寻址单位（字节），任何一方的分配/计算都必须严格按这个单位来，不能"自作主张"换算成别的单位（long、word 等）。**抽象层边界上的单位约定是契约，违约就数据错乱**。这个 bug 在 32 位机上不会暴露（`sizeof(long)=4`，恰好等于 `int` 宽度），只在 64 位机上才显现——典型的"平台相关隐藏 bug"。
