## 一、必须修复的 BUG

### 1. 解释器 MUS 指令缺少 break — fall-through 到 ADD（严重）

**位置**: `C/src/interpret/interpret.cpp` 第 136-142 行

```cpp
case MUS:
    S[top]=-S[top];
    // ← 缺 break，直接 fall-through 到下面的 ADD
case ADD:
case ADD1:
    top--;
    S[top]=S[top]+S[top+1];
    break;
```

**问题**：编译器生成 `MUS` 用于一元取负（如 `-x`、`-5`），期望只做取反不改变栈高度。但解释器 fall-through 到 ADD 后会额外弹一个元素做加法，导致栈指针错位，后续所有栈操作在错误位置执行。所有包含负数的表达式都会给出错误结果。

当前测试程序 `test.pl` 恰好没有负数运算，所以未暴露。

**修复**：在第 137 行后加 `break;`。

---

### 2. 解释器 RED 字符输入丢失 getch() 返回值（中）

**位置**: `C/src/interpret/interpret.cpp` 第 116-125 行

```cpp
case RED:
    if(instruction.address==0) {
        printf("Your Input:");
        scanf("%d",&temp);
    } else
        getch();       // ← 返回值被丢弃
    S[S[top]]=temp;    // ← temp 是旧值（0 或上次 scanf 的值），不是刚输入的字符
```

**问题**：字符型 `read` 功能完全不工作——用户输入被忽略，存储的是上次的残留值。

**修复**：`temp = getch();`

---

### 3. PL/0 编译器 GETPOSITION 函数参数被忽略（中）

**位置**: `C/src/PL/PL.cpp` 第 225-241 行

```cpp
int GETPOSITION(char * id)  // 声明接收 id 参数
{
    strcpy(NAMETAB[0].name, id);  // 第 228 行：正确用了 id
    ...
    while(strcasecmp(NAMETAB[i].name, CurSymbol->value.lpValue))  // 第 234 行：却用了全局 CurSymbol
        i=NAMETAB[i].link;
```

**问题**：第 228 行正确将参数 `id` 设为哨兵，但第 234 行搜索时用了全局 `CurSymbol->value.lpValue` 而非参数 `id`。当前所有调用点碰巧都传 `CurSymbol->value.lpValue`，所以不触发错误——但这是真正的代码缺陷，函数签名与实现语义不一致。

**修复**：将第 234 行的 `CurSymbol->value.lpValue` 改为 `id`。

## 二、需要关注的隐患

### 4. C-- lexer 未闭合的块注释在 EOF 处静默吞掉（中）

**位置**: `C/src/lex/lexer.l` 第 53-60 行

`/*` 注释如果在文件结束前未闭合，`input()` 返回 0 时循环退出但不报错。parser 收到的 token 流不完整，给出令人困惑的 "syntax error" 而非明确的 "unclosed comment"。

**修复**：循环结束后检查 `c == 0` 时输出错误信息。

---

### 5. 解释器 .pld 二进制格式依赖隐含假设（中）

编译器端 `INSTRUCTION` 有 `lineNumber` 字段但 `WriteObjCode` 逐字段写入跳过了它；解释器端 `INSTRUCTION` 没有 `lineNumber`。正确性依赖于 `sizeof(OPCOD) == sizeof(int)` 且无 padding。GCC 下成立，但 C 标准不保证枚举大小等于 int。

**风险**：极低（双方都用 GCC 编译），但代码设计脆弱。

---

### 6. CALL 函数中 GETPOSITION 返回 0 后直接索引 NAMETAB（低）

**位置**: `C/src/PL/PL.cpp` 第 1380-1381 行

`GETPOSITION` 未找到标识符时返回 0 并报 error(33)，但 CALL 直接用 `NAMETAB[i].kind` 判断。`NAMETAB[0]` 是合法哨兵所以不崩溃，但会报一个多余的不精确错误信息（"应该是过程" 而非 "标识符没有定义"）。
