## 0. 先建立大图：编译器到底在干什么

你写的高级代码（C--），计算机 CPU 是看不懂的。CPU 只懂自己的指令（比如 MIPS 指令）。**编译器就是一个翻译官**，把 C-- 一路翻译成机器能跑的东西。

这个翻译分好几道工序，一道接一道：

```
源代码 (test.cmm)
    │
    ▼  ① 词法分析   ← 实践 1.1（本文档重点）
  Token 序列         把字符拼成"词"
    │
    ▼  ② 语法分析   ← 实践 1.2（本文档重点）
  语法树 (AST)       检查"词"组成的句子合不合语法
    │
    ▼  ③ 语义分析   ← 实践 2
  带类型的语法树      检查"句子有没有意义"（类型对不对、变量有没有定义）
    │
    ▼  ④ 中间代码    ← 实践 3
  IR (irsim 能跑)    翻译成一种中间语言
    │
    ▼  ⑤ 目标代码    ← 实践 4
  MIPS 汇编          翻译成最终机器指令
    │
    ▼  ⑥ 优化       ← 实践 5
  更高效的代码
```

**为什么是这种流水线结构？** 因为一步翻译太难（字符直接到机器指令，跨度太大，谁也写不出）。拆成小步，每步只解决一个简单问题，每步的输出是下一步的输入。这就是"分而治之"。

本文档讲最前面两步：**词法分析**和**语法分析**。

## 1. 词法分析（Lexical Analysis）

### 1.1 为什么需要它？

看这行 C-- 代码：

```c
int age = 18;
```

在计算机眼里，它只是一串字符：`i n t 空格 a g e 空格 = 空格 1 8 ;`。

如果让语法分析直接面对这些零散字符，会极其痛苦——它得自己判断哪几个字符组成一个有意义的单位。于是**词法分析先做一道预处理**：把字符拼成一个个"词"，这些"词"叫 **Token（词法单元 / 单词）**。

上面那行经过词法分析，变成：

```
TYPE(int)   ID(age)   ASSIGNOP(=)   INT(18)   SEMI(;)
```

干净多了。**词法分析的任务就是：吃进字符流，吐出 Token 序列。** 顺便还要做两件事：
- **丢掉注释和空白**（空格、换行、制表符）——它们只是给人看的，对语法没意义。
- **记录位置**（第几行）——后面报错时要告诉用户"第 X 行有错"。

### 1.2 是什么？先讲懂几个概念

**(1) Token（词法单元）**

就是"词的种类"。比如 `int`、`float` 都属于 `TYPE` 这一种 Token；`+`、`-`、`*`、`/` 都是运算符 Token，但每种各有名字（PLUS、MINUS、STAR、DIV）。一个 Token 通常带两部分信息：
- **类型**：它是什么种类的（比如 INT、ID）
- **值**：具体内容（比如数字 `18`、变量名 `age`）

**(2) 正则表达式（Regular Expression）—— 描述"什么样的字符串算一个 Token"**

词法规则本质上是模式匹配。比如"标识符 ID"的规则是：以字母或下划线开头，后面跟若干字母/数字/下划线。用正则表达：

```
[a-zA-Z_][a-zA-Z0-9_]*
```

- `[a-zA-Z_]` 表示"一个字母或下划线"
- `[a-zA-Z0-9_]*` 表示"若干个（含 0 个）字母/数字/下划线"
- 合起来就是 ID 的规则

正则表达式就是描述字符串模式的紧凑写法。常见符号：
- `*` 重复 0 次或多次
- `+` 重复 1 次或多次
- `?` 出现 0 次或 1 次
- `[abc]` a 或 b 或 c 中的一个
- `|` 或

**(3) 状态机（DFA/NFA）—— 正则的"执行者"**

正则表达式怎么变成能跑的程序？背后是一个"状态机"：想象一条流水线，字符一个一个进来，机器根据当前字符从"当前状态"跳到"下一个状态"，跳到某个特殊状态时，就识别出一个 Token。**Flex 帮你自动把正则转成状态机**，你不用手写状态机，只写正则就行。

### 1.3 怎么做？用 Flex

Flex 是一个**词法分析器生成器**：你给它一份描述词法规则的文件（`.l`），它自动生成一份 C 代码（`lex.yy.c`），这份 C 代码就是一个能做词法分析的程序。

**Flex 文件结构（三段式，用 `%%` 分隔）：**

```
{定义部分}      ← 给正则起别名，方便后面用
%%
{规则部分}      ← 正则 + 匹配到后执行的 C 动作
%%
{用户代码部分}  ← 比如 main 函数
```

**一个最小例子（识别 C-- 的 ID 和 INT）：**

```c
/* 定义部分 */
digit       [0-9]
letter      [_a-zA-Z]
id          {letter}({letter}|{digit})*
int_num     [1-9][0-9]*|0

%%
/* 规则部分：左边的正则匹配到，就执行右边花括号里的动作 */

{int_num}   { printf("INT: %s\n", yytext); }
{id}        { printf("ID: %s\n", yytext); }

[ \t\n]+    { /* 空白字符，直接吞掉，什么都不做 */ }

"//".*      { /* 单行注释，吞掉 */ }

.           { /* 兜底：其他字符原样打印或报错 */ }

%%
/* 用户代码部分 */
int main(int argc, char **argv) {
    if (argc > 1) {
        if (!(yyin = fopen(argv[1], "r"))) {
            perror(argv[1]);
            return 1;
        }
    }
    while (yylex() != 0);   /* 反复调用 yylex 直到输入结束 */
    return 0;
}
```

**关键变量/函数：**
- `yylex()`：Flex 生成的核心函数，每调用一次识别并返回一个 Token。
- `yytext`：当前匹配到的那个字符串（比如识别 INT 时，yytext 就是 `"18"`）。
- `yyin`：输入文件指针，告诉 Flex 从哪读字符。

**编译运行流程：**

```bash
flex lexical.l                       # 生成 lex.yy.c
gcc main.c lex.yy.c -lfl -o scanner  # 编译成可执行文件 scanner
./scanner test.cmm                   # 对 test.cmm 做词法分析
```

> 注意 `-lfl` 不能少，否则链接报错。

## 2. 语法分析（Syntax / Grammatical Analysis）

### 2.1 为什么需要它？

词法分析只管"拼词"，不管"句子通不通"。比如：

```
int = 18 age ;
```

词法分析能认出这些 Token（TYPE、ASSIGNOP、INT、ID、SEMI），但这句**语法是错的**。**语法分析的任务就是：检查 Token 序列是否符合语言的语法规则**，符合的话搭出一棵"语法树"。

### 2.2 是什么？先讲懂几个概念

**(1) 文法（Grammar）—— 语言的语法规则**

用一种形式化的方式描述语法，叫"上下文无关文法"。它由一堆**产生式（Production）**组成。产生式长这样：

```
左边 → 右边1 | 右边2 | ...
```

意思是"`左边`这个语法单位，可以写成`右边1`，或`右边2`……"。举 C-- 的例子：

```
Stmt → Exp SEMI                  // 一条语句可以是"表达式加分号"
     | RETURN Exp SEMI           // 也可以是 return 表达式;
     | IF LP Exp RP Stmt         // 也可以是 if (表达式) 语句
     | WHILE LP Exp RP Stmt      // 也可以是 while (表达式) 语句
     | CompSt                    // 也可以是一个语句块
```

这就精确描述了"C-- 里什么算一条语句"。

**(2) 终结符 vs 非终结符**

- **终结符（Terminal）**：就是 Token，是不能再拆的最小单位（INT、ID、`if`、`;` 等）。在文法里通常是大写名字（IF、SEMI）。
- **非终结符（Non-terminal）**：能继续展开的语法单位（Stmt、Exp、Program）。在文法里通常是大写开头。
- 文法从**初始符号**（C-- 里是 `Program`）出发，不断用产生式展开，最终全变成终结符（Token），就生成了一句合法代码。反过来，给定 Token 序列，看能不能从 Program 出发"归约"出来，这就是语法分析。

**(3) 语法树（AST）—— 语法分析的产物**

分析通过后，按展开过程搭成一棵树。比如 `a = 1;` 大致长这样：

```
        Stmt
         |
      Exp SEMI
         |
    Exp ASSIGNOP Exp
    (ID=a) (=)   (INT=1)
```

这棵树是后面所有阶段（语义、中间代码……）的基础。所以前面说"代码风格要好、模块接口要清晰"——因为后面阶段都建立在词法/语法分析之上。

**(4) 二义性、优先级、结合性**

文法有时会有"二义性"——一句话能分析成两种树。比如 `1 + 2 * 3`，是先加后乘、还是先乘后加？C-- 给了明确的**优先级表**（附录 A 表 1）解决：

| 优先级 | 运算符 | 结合性 | 说明 |
|--------|--------|--------|------|
| 1 (最高) | `( )` `[ ]` `.` | 左结合 | 括号、数组访问、结构体访问 |
| 2 | `-`(取负) `!` | 右结合 | 一元运算符 |
| 3 | `*` `/` | 左结合 | 乘除 |
| 4 | `+` `-` | 左结合 | 加减 |
| 5 | `<` `<=` `>` `>=` `==` `!=` | 左结合 | 关系比较 |
| 6 | `&&` | 左结合 | 逻辑与 |
| 7 | `\|\|` | 左结合 | 逻辑或 |
| 8 (最低) | `=` | 右结合 | 赋值 |

> 数值越小优先级越高。所以 `1+2*3` 先算 `2*3`（优先级 3 高于 4），结果是 7。`a = b = 1` 因为 `=` 右结合，先算 `b = 1`。

### 2.3 怎么做？用 Bison

Bison 是**语法分析器生成器**：你给它一份语法规则文件（`.y`），它生成 C 代码，做语法分析。**Bison 会调用 Flex 提供的 `yylex()` 拿 Token**，所以两者是搭档关系。

**Bison 文件也是三段式：**

```
{声明部分}   ← 声明 Token、运算符优先级等
%%
{规则部分}   ← 文法产生式 + 每条规则对应的动作（C 代码）
%%
{用户代码}   ← main 等
```

**规则部分长这样（示意）：**

```yacc
Stmt : Exp SEMI            { /* 匹配到"表达式;"时要做的事 */ }
     | RETURN Exp SEMI     { /* ... */ }
     | IF LP Exp RP Stmt   { /* ... */ }
     ;
```

> 在实践 1.2 里，语法分析只要**判断对错**（能分析出来就对，分析不出就报语法错误），暂时不需要真的搭树或算值——搭树是后面阶段的事。

**Flex + Bison 协作的编译流程：**

```bash
flex lexical.l                       # 词法：生成 lex.yy.c
bison -d parser.y                    # 语法：生成 parser.tab.c 和 parser.tab.h
gcc parser.tab.c lex.yy.c -o parser  # 一起编译成语法分析器
./parser test.cmm                    # 检查 test.cmm 的语法
```

## 3. C-- 语言文法详解（你要写的代码就照这个）

下面是 C-- 的**完整文法**，全部来自附录 A，我按类别帮你梳理，每个都配大白话解释。

### 3.1 Token（词法单元，实践 1.1 要识别的）

| Token | 含义 | 正则该怎么写（你来想） |
|-------|------|----------------------|
| `INT` | 整数 | 一串数字，首位非 0（除非就是 `0`）。也支持八进制 `0237`、十六进制 `0xFF32` |
| `FLOAT` | 浮点数 | 数字+小数点，小数点前后都得有数字（`0.7`、`12.43`）。也支持指数 `01.23E12` |
| `ID` | 标识符 | 字母/下划线开头，后跟字母/数字/下划线，长度 < 32 |
| `SEMI` | `;` | 分号 |
| `COMMA` | `,` | 逗号 |
| `ASSIGNOP` | `=` | 赋值 |
| `RELOP` | `>` `<` `>=` `<=` `==` `!=` | 关系运算符（注意 `>=` 要写在 `>` 前面，Flex 会贪婪匹配） |
| `PLUS` `MINUS` `STAR` `DIV` | `+ - * /` | 四则运算 |
| `AND` `OR` | `&&` `\|\|` | 逻辑与/或（注意是双符号） |
| `DOT` `NOT` | `.` `!` | 成员访问 / 逻辑非 |
| `TYPE` | `int` `float` | 类型关键字 |
| `LP` `RP` `LB` `RB` `LC` `RC` | `( ) [ ] { }` | 三种括号（L=左, R=右；P=圆, B=方, C=花） |
| `STRUCT` `RETURN` `IF` `ELSE` `WHILE` | 对应关键字 | 保留字 |

> **⚠️ 关键提醒：Flex 的"最长匹配"原则。** 比如 `==` 必须放在 `=` 之前识别，否则看到 `==` 会先匹配成两个 `=`。`&&` 要在 `&`（但 C-- 没有 `&`）……总之多字符的运算符要先写。保留字（`if`/`while` 等）要先于 ID 识别，否则会被当成普通标识符。

### 3.2 高层定义（High-level Definitions）

```
Program    → ExtDefList                       // 整个程序 = 一串外部定义
ExtDefList → ExtDef ExtDefList | （空）         // 零个或多个 ExtDef
ExtDef     → Specifier ExtDecList SEMI        // 全局变量定义，如 int a, b;
           | Specifier SEMI                   // 结构体定义，如 struct {...}; （int; 也合法，不报错）
           | Specifier FunDec CompSt          // 函数定义 = 返回类型 + 函数头 + 函数体
ExtDecList → VarDec | VarDec COMMA ExtDecList // 变量列表
```

**大白话：** 程序就是一串"外部定义"。每个外部定义要么是全局变量、要么是结构体、要么是函数。

### 3.3 类型描述符（Specifiers）

```
Specifier       → TYPE | StructSpecifier      // 类型 = 基本类型(int/float) 或 结构体
StructSpecifier → STRUCT OptTag LC DefList RC  // 定义结构体，如 struct Complex {...}
                | STRUCT Tag                   // 用已定义的结构体，如 struct Complex a;
OptTag          → ID | （空）                    // 结构体名字（可有可无）
Tag             → ID
```

### 3.4 声明器（Declarators）

```
VarDec   → ID | VarDec LB INT RB              // 变量 = 标识符 或 数组(如 a[10][2])
FunDec   → ID LP VarList RP | ID LP RP        // 函数头 = 名字(参数列表) 或 名字()
VarList  → ParamDec COMMA VarList | ParamDec  // 形参列表
ParamDec → Specifier VarDec                   // 一个形参 = 类型 + 变量
```

**大白话：** 变量要么是普通变量，要么是数组。函数头由函数名和形参列表组成。

### 3.5 语句（Statements）

```
CompSt   → LC DefList StmtList RC             // 语句块 = { 局部变量定义 + 一串语句 }
StmtList → Stmt StmtList | （空）
Stmt     → Exp SEMI                           // 表达式语句
         | CompSt                             // 嵌套语句块
         | RETURN Exp SEMI                    // return 表达式;
         | IF LP Exp RP Stmt                  // if (表达式) 语句
         | IF LP Exp RP Stmt ELSE Stmt        // if (表达式) 语句 else 语句
         | WHILE LP Exp RP Stmt               // while (表达式) 语句
```

**⚠️ 重点：** C-- 规定**变量必须在语句块开头定义**，不能在任意位置写 `int a;`。这是 CompSt 的定义决定的（先 DefList 后 StmtList）。

### 3.6 局部定义（Local Definitions）

```
DefList → Def DefList | （空）
Def     → Specifier DecList SEMI              // 一条定义，如 int a, b = 5;
DecList → Dec | Dec COMMA DecList
Dec     → VarDec | VarDec ASSIGNOP Exp        // 变量 或 变量=初值
```

### 3.7 表达式（Expressions）

表达式种类最多，但归为四类（见附录 A 解释）：

```
Exp → Exp ASSIGNOP Exp        // 赋值
    | Exp AND Exp | Exp OR Exp | Exp RELOP Exp     // 逻辑、关系
    | Exp PLUS Exp | Exp MINUS Exp | Exp STAR Exp | Exp DIV Exp  // 四则运算
    | LP Exp RP               // 括号
    | MINUS Exp | NOT Exp     // 一元取负、逻辑非
    | ID LP Args RP | ID LP RP   // 函数调用（带参/无参）
    | Exp LB Exp RB           // 数组访问 a[i]
    | Exp DOT ID              // 结构体成员访问 a.b
    | ID | INT | FLOAT        // 最基本：变量、整数、浮点数
Args → Exp COMMA Args | Exp   // 实参列表
```

> 表达式的多种运算符要靠 **3.1 节的优先级表**消除二义性。写 Bison 时要在声明部分用 `%left` `%right` 告诉 Bison 优先级和结合性。

### 3.8 注释

- `//` 单行注释：从 `//` 到行尾丢弃。
- `/* ... */` 多行注释：`/*` 到最近的 `*/` 丢弃。**不允许嵌套**，嵌套要报错。

注释在**词法分析阶段**就被丢弃（实践 1.1 就要做），到语法分析时已经看不到了。

4. 语法正确就接受，错误就报错（报错要带行号，行号信息来自词法分析的记录）。

## 5. 常见困惑 FAQ

**Q1：Flex 和 Bison 到底谁先谁后？谁调谁？**
Bison 是主，Flex 是辅。Bison 在分析语法时需要下一个 Token，就调用 Flex 生成的 `yylex()`；Flex 从输入文件读字符、识别出一个 Token 返回给 Bison。数据流：`字符 → Flex → Token → Bison → 语法树`。

**Q2：为什么优先级表里 `>=` 要比 `>` 先识别？**
因为 Flex 用"最长匹配"——它会尽量匹配更长的串。但你写规则时，如果 `>` 写在前面、又恰好能匹配，可能就匹配成 `>` 了。所以多字符运算符要写在前面（虽然 Flex 默认按最长匹配，但规则顺序在长度相同时决定优先级）。

**Q3：C-- 和 C 语言什么区别？**
C-- 是 C 的一个大幅简化子集。没有指针、没有 `for`、没有 `do-while`、没有自增自减 `++`、类型只有 int/float/struct。它就是为教学设计的"刚好够学编译原理"的语言。

**Q4：实践 1.2 要不要报错？报什么错？**
要。语法分析遇到不合文法的就报"语法错误"。但**具体怎么报、报多详细**，以 `Project_1.pdf` 和测试集期望（`Tests/expects/`）为准。本阶段不要求报"语义错误"（类型错、未定义变量那是实践 2 的事）。

**Q5：`int;` 这种没意义的句子要报错吗？**
不要。文法 `ExtDef → Specifier SEMI` 明确允许它，附录 A 说不算错误。
