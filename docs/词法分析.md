# 词法分析（实践 1.1）—— 为什么、是什么、怎么做

> 配套代码：`C/src/lex/lexer.l`
> 适用阶段：C-- 编译器第 1 阶段（词法分析）
> 阅读前提：不需要懂编译原理，本文从"为什么要切词"讲起。

---

## 0. 一句话先说明白

**词法分析**就是给一段源代码"断句"：把一连串的字符（`i`、`n`、`t`、空格、`m`、`a`、`i`、`n`……）切成一个一个有意义的"词"（`int`、`main`、`(`、`)`……），每个词叫一个 **Token（词法单元）**。

就像你读英文 "I love you"，你的大脑会自动把 9 个字符切成 3 个词。词法分析器干的就是这件事，只不过它切的是 C-- 代码。

切完之后，后面的阶段（语法分析）就不用再跟"单个字符"打交道了，而是直接面对"一串 Token"，工作起来轻松得多。

---

## 1. 为什么（Why）

### 1.1 为什么编译器第一步要做词法分析？

想象一下，如果编译器直接对着原始字符流做语法分析会怎样。比如要判断 `if (x > 0)` 是不是合法的 if 语句，它得先在字符流里"数"出 `i`、`f` 后面是空格、再数 `(`、再数 `x`…… 光是找出"if 这个词从哪到哪"就很费劲，更别提还要同时判断语法结构了。

**词法分析的价值就是"分工"**：

- **词法分析器**只管一件事：把字符流切成 Token。它处理的是"哪些字符组合算一个词"这种**局部**问题。
- **语法分析器**只管一件事：拿到的 Token 序列符不符合语法规则。它处理的是"这些词怎么组合成合法程序"这种**结构**问题。

各管一摊，每一摊都简单。这是编译器经典的"分阶段"设计思想——把一个复杂问题拆成一串简单问题，每个阶段只做一件事。

### 1.2 词法分析器还要顺手干几件杂活

除了"切词"，词法分析器作为唯一一个**逐字符读入整个源文件**的模块，还会顺手做：

1. **丢掉注释和空白**：注释（`//...`、`/* ... */`）和空格、换行、制表符对程序含义没影响，词法分析器直接扔掉，不产生 Token，免得后面阶段被这些噪声干扰。
2. **记录行号**：每读到换行符就把行号 +1。这样后面报错时能说出"第 4 行有问题"，而不是只说"某个地方有问题"。
3. **报告词法错误**：遇到语言里根本没定义的字符（比如 C-- 里没有 `~`），它负责当场报错。

### 1.3 选做部分做到哪一档？

C-- 的词法有三处"难点"（指导书明说只有 INT/FLOAT/ID 需要自己写正则）：

- **选做 2.1**：八进制（`0` 开头，如 `0237`）、十六进制（`0x` 开头，如 `0xFF32`）
- **选做 2.2**：指数形式浮点数（`1.05e-4`、`.5E03`、`43.e-4`）
- **选做 2.3**：`/* ... */` 块注释的完整识别与嵌套禁止

本词法分析器**已实现全部三个选做**，并附带非法形式的精准报错（如 `09` 报 `Illegal octal`、`1.05e` 报 `Illegal floating point`）。实现策略是"先把必做主干立稳，再逐个补选做"——选做的正则和报错逻辑比必做绕，分两步走调试成本更低。

## 2. 是什么（What）

这一节讲清楚做词法分析必须懂的概念。先讲概念，再讲工具。

### 2.1 三个核心术语：词法单元、模式、词素

这三个词容易混，先固定下来：

- **词法单元（Token）**：一个"类别标签"。比如 `INT`、`ID`、`PLUS`。词法分析器输出的就是一串 Token。
- **模式（Pattern）**：描述"什么样的字符串算这一类"的规则。比如 ID 的模式是"字母或下划线开头，后跟字母数字下划线"。
- **词素（Lexeme）**：源代码里**实际匹配上的那串字符**。比如源码里的 `score`、`_count`、`x1` 都是 ID 这个 Token 的具体词素。

举个例子，对于源代码 `count = 123;`：

| 词素      | 匹配的模式 | 输出的 Token |
| ------- | ----- | --------- |
| `count` | 标识符规则 | ID        |
| `=`     | 赋值号   | ASSIGNOP  |
| `123`   | 整数规则  | INT       |
| `;`     | 分号    | SEMI      |

注意 Token 是"类别"，词素是"具体内容"。同一个 Token 类型（比如 ID）可以对应无数个不同的词素（`count`、`x`、`foo` 都行）。所以输出 Token 时，对于 INT/ID/FLOAT 这类"值会变"的 Token，我们还会把词素本身也带出来（比如 `INT: 123`、`ID: count`），方便后面阶段用。

### 2.2 正则表达式：描述模式的语言

**模式怎么写出来？** 用正则表达式（Regular Expression）。它是一种紧凑的"字符串模式描述语言"。词法分析其实就是一堆正则表达式的集合。

Flex 里用到的正则语法（挑常用的讲）：

| 写法      | 含义                        | 例子                               |
| ------- | ------------------------- | -------------------------------- |
| `abc`   | 就匹配 `abc` 这串              | `int` 匹配 `int`                   |
| `[...]` | 字符集，匹配其中任意一个字符            | `[0-9]` 匹配 0~9 任一个数字             |
| `[a-z]` | 用 `-` 表示范围                | `[a-zA-Z]` 匹配任一字母                |
| `*`     | 前面的东西重复 **0 次或更多**        | `[0-9]*` 匹配 `""`、`"5"`、`"123"`   |
| `+`     | 前面的东西重复 **1 次或更多**        | `[0-9]+` 匹配 `"5"`、`"123"`，但不匹配空串 |
| `\|`    | 或，左右任选一个                  | `a\|b` 匹配 `a` 或 `b`              |
| `(...)` | 分组，把几个字符当成一个整体            | `(ab)+` 匹配 `ab`、`abab`           |
| `"..."` | 双引号引起来的内容**逐字匹配**（取消特殊含义） | `"&&"` 就是要匹配两个 `&`               |

**几个 C-- 词法要用到的正则，逐步推演：**

- **标识符 ID**：`字母或下划线` 开头，后面跟任意个 `字母/数字/下划线`。
  - 字母或下划线：`[_a-zA-Z]`（Flex 里下划线直接写）
  - 后面跟任意个字母数字下划线：`([_a-zA-Z0-9])*`
  - 合起来：`[_a-zA-Z]+`（因为开头字符本身也属于"字母数字下划线"，所以直接写"一个或多个字母数字下划线"即可，更简洁）
- **十进制整数 INT**：除 `0` 外首位不能是 0。
  - 单独的 `0`：`0`
  - 非零开头：`[1-9][0-9]*`（首位 1~9，后面任意个 0~9）
  - 两种情况用 `|` 合起来（在 Flex 里写成两条规则更清晰）
- **基础浮点 FLOAT**：小数点前后都要有数字。
  - `{digit}+"."{digit}+`（一串数字 + 点 + 一串数字，如 `3.14`、`0.7`）

> **为什么浮点正则要求前后都有数字？** 因为 Appendix A 明确规定 C-- 的基础浮点"小数点前后必须有数字"。所以 `3.14` 合法，但 `.5`（前面没数字）和 `3.`（后面没数字）**在本阶段都不算合法 FLOAT**。`.5` 会被拆成 `.`（DOT）和 `5`（INT），这是符合必做规则的预期行为，不是 bug。`.5` 这种写法只有在选做 2.2 的指数浮点里（`.5E03`）才合法。

### 2.3 Flex：自动生成词法分析器的工具

手写一个词法分析器（用正则 → NFA → DFA 那套理论）很繁琐。**Flex** 是一个工具：你把"<u>正则规则 + 每条规则匹配到后干什么"写进一个 `.l` 文件</u>，它自动帮你生成一份 C 代码（`lex.yy.c`），里面有一个现成的 `yylex()` 函数负责切词。你不用自己实现状态机，Flex 帮你做。

**Flex 源文件（`.l`）的三段式结构**，用 `%%` 分隔：

```
{定义部分}      ← 给正则起别名、写 %option、用 %{ %} 引入 C 头文件
%%
{规则部分}      ← 每行一条：正则  { 匹配到后执行的 C 代码 }
%%
{用户代码部分}  ← 原样复制进 lex.yy.c，通常放 main 函数
```

### 2.4 Flex 工作的两条铁律（必须懂，否则规则写不对）

Flex 在读入字符流时，按以下两条规则决定匹配哪条：

1. **最长匹配**：尽可能多吃字符。比如输入 `>=`，虽然 `>` 也能匹配，但 `>=` 更长，所以匹配 `>=` 这条规则，输出 RELOP 而不是先输出一个 `>`。
2. **规则顺序优先**：当多条规则都能匹配**同样长**的串时，写在前面的规则赢。这条非常关键，决定了**规则该怎么排序**。

**由此推出我们 `.l` 文件里的两条排序原则**（代码注释里也写了）：

- **多字符运算符必须写在单字符前面**。比如 `&&` 必须写在 `&`（如果有）前面、`>=` 必须写在 `>` 前面。否则遇到 `&&`，Flex 会先匹配成两个 `&`（因为单字符规则在前，且"最长匹配"对单字符规则也只吃到一个 `&` 时就可能被截断——实际上 Flex 是贪心的，但靠规则顺序保证 `&&` 优先）。在我们的代码里，`>=` `<=` `==` `!=` `&&` `||` 全部排在对应的单字符之前。
- **关键字必须写在 ID 规则之前**。因为 `int`、`if`、`while` 这些关键字本身完全符合 ID 的字符模式（都是字母开头）。如果 ID 规则写在前面，`int` 会被当成普通标识符 `ID: int` 吃掉，关键字就失效了。所以代码里 7 个关键字全部写在 `{letter}...` 这条 ID 规则之前。

### 2.5 Flex 给我们用的几个现成变量/函数

在 action（每条规则匹配后执行的 C 代码）里，Flex 提供了几个便利：

| 名字         | 含义                                                         |
| ---------- | ---------------------------------------------------------- |
| `yytext`   | 当前匹配到的**词素**（字符串，`char*`）。比如匹配到 `123`，`yytext` 就是 `"123"`。 |
| `yyleng`   | 当前词素的长度（可理解为 `strlen(yytext)`）。                            |
| `yylineno` | 当前行号（需要 `%option yylineno` 开启才可用，自动维护）。                    |
| `yyin`     | 输入文件指针，默认是 `stdin`（键盘）。可以改指向打开的文件。                         |
| `yylex()`  | 启动词法分析。每次识别一个 Token 并执行对应 action。                          |

## 3. 怎么做（How）

### 3.1 整体数据流

```
源代码文件 test.cmm
        │
        ▼
   yyin 指向该文件
        │
        ▼
   yylex() 开始工作（Flex 生成的状态机）
        │  每识别一个 Token
        ▼
   执行对应 action（这里是 printf 输出 Token 名）
        │
        ▼
   输出 Token 流（屏幕）：
     TYPE / ID: main / LP / RP / LC / ...
```

### 3.2 `lexer.l` 逐段讲解

代码文件：`C/src/lex/lexer.l`。下面按三段式结构讲。

#### 第一部分：定义

```
%option yylineno
%{
#include <stdio.h>
#include <stdlib.h>
%}
digit       [0-9]
letter      [_a-zA-Z]
nonzero     [1-9]
```

- `%option yylineno`：开启 Flex 内置行号计数器，`yylineno` 自动随换行符 +1，省得自己维护。报错时用它给出行号。
- `%{ %}`：引入 C 头文件。`stdio.h` 为了 `printf`/`perror`，`stdlib.h` 为了 `atoi`（把 `"123"` 转成整数 `123`）。
- `digit`/`letter`/`nonzero`：给正则起别名，下面规则里用 `{digit}` 引用，比满屏写 `[0-9]` 易读。

#### 第二部分：规则（核心）

规则区**只写"模式 + 动作"**，每行一条，模式和 `{` 之间不能加注释（这是 Flex 的硬要求，加了会报 `unrecognized rule`，我第一次就是这么错的）。

**注释和空白（不产生 Token，直接丢弃）：**

```
"//".*                      { }            // 单行注释：// 后到行尾，丢弃
[ \t\r\n]+                  { }            // 空格、制表符、换行，丢弃
```

> 注：`/*` 块注释的完整识别属于选做 2.3，本阶段遇到 `/*` 先按"未定义"报 Error type A 提示，留待后续扩展成完整的多行注释处理。

**多字符运算符（必须排在单字符前面）：**

```
"&&"   { printf("AND\n"); }
"||"   { printf("OR\n"); }
">="   { printf("RELOP\n"); }
"<="   { printf("RELOP\n"); }
"=="   { printf("RELOP\n"); }
"!="   { printf("RELOP\n"); }
```

`>` `<` `>=` `<=` `==` `!=` 这六种比较运算符统一输出 `RELOP`（关系运算符），因为语法分析阶段它们是同一类（产生式 `RELOP → > | < | >= | <= | == | !=`）。

**关键字（必须排在 ID 前面）：**

```
"int"     { printf("TYPE\n"); }
"float"   { printf("TYPE\n"); }
"struct"  { printf("STRUCT\n"); }
"return"  { printf("RETURN\n"); }
"if"      { printf("IF\n"); }
"else"    { printf("ELSE\n"); }
"while"   { printf("WHILE\n"); }
```

`int`/`float` 统一输出 `TYPE`（类型关键字，产生式 `TYPE → int | float`）。

**标识符 ID：**

```
{letter}({letter}|{digit})*   { printf("ID: %s\n", yytext); }
```

`{letter}` 已含下划线，所以 `_foo`、`__x` 也合法。输出时带上词素（`ID: score`）。

**整数 INT（十进制，两种写法）：**

```
0                    { printf("INT: %d\n", atoi(yytext)); }
{nonzero}{digit}*    { printf("INT: %d\n", atoi(yytext)); }
```

单独的 `0` 是合法整数；非零数字开头后跟任意个数字也是。`atoi` 把词素转成整数输出（`INT: 123`）。

**浮点 FLOAT：**

```
{digit}+"."{digit}+   { printf("FLOAT: %s\n", yytext); }
```

小数点前后都有数字。浮点直接原样打印词素（`FLOAT: 3.14`），不用 `atoi`（它只转整数）。

**单字符符号（20 类，逐字匹配）：**

```
";"  { printf("SEMI\n"); }
","  { printf("COMMA\n"); }
"="  { printf("ASSIGNOP\n"); }
... （共 20 个，略）
```

**兜底（错误类型 A）：**

```
.   { printf("Error type A at Line %d: Mysterious characters \"%s\".\n", yylineno, yytext); }
```

`.` 匹配任何单个字符。凡是前面所有规则都没匹配上的字符（比如 `~`、`@`、`#`），都会落到这里，报 Error type A。这正是 Project_1 样例 1 里 `~` 报错的来源。

#### 第三部分：用户代码

```
int main(int argc, char **argv) {
    if (argc > 1) {
        FILE *f = fopen(argv[1], "r");
        if (!f) { perror(argv[1]); return 1; }
        yyin = f;
    }
    yylex();
    return 0;
}
int yywrap(void) { return 1; }
```

- `main`：读命令行第一个参数作为输入文件名，打开后把 `yyin` 指向它（没参数就读键盘）。
- `yylex()`：启动切词，直到输入结束。
- `yywrap`：`yylex` 读到文件末尾时调用，返回 1 表示"没有更多输入了，结束"。写了它就不用链接 `-lfl` 库。

### 3.3 编译和运行

```bash
# 1. Flex 把 .l 生成 C 代码 lex.yy.c
flex -o C/src/lex/lex.yy.c C/src/lex/lexer.l

# 2. gcc 编译成可执行文件 scanner
gcc C/src/lex/lex.yy.c -o C/bin/scanner

# 3. 运行：对源代码做词法分析
./C/bin/scanner test.cmm
```

> 不需要 `-lfl`，因为我们自己写了 `yywrap` 和 `main`。

## 4. 选做实现详解（2.1 / 2.2 / 2.3）

这一节回答三个问题：**写了哪些规则？每个规则到底长什么样？遇到具体输入 Flex 会选中哪条、为什么？**

读法约定：每一小节先**列出本小节用到的全部规则（带编号①②③）**，每个规则是一个完整的"模式 → 动作"对，然后**用表格演示几个典型输入**，最后一句话点出"为什么是这个规则胜出"。这样你看到表格里某一列"长度"数字，就能直接对上 Flex 的"最长匹配"铁律。

先回顾 2.4 节的两条铁律，下面会反复用到：

- **最长匹配**：多条规则都匹配时，**吃字符最多的那条赢**。
- **等长时按顺序**：长度相同时，**写在文件前面的那条赢**。

选做的核心技巧就是：**把规则设计成"合法形式和非法形式长度天然不同"**，这样靠最长匹配自动区分，不依赖顺序。下面分别看。

---

### 4.1 选做 2.1：八进制和十六进制整数

#### 4.1.1 用到的全部规则（共 7 条）

按 `.l` 文件里出现的顺序，整型相关一共 7 条规则。**每条都是一个"模式 → 动作"对**：

```
①  0[xX][0-9A-Fa-f]*[G-Zg-z_][0-9A-Za-z_]*   →  报错（非法十六进制，如 0x3G）
②  0[xX]                                       →  报错（单独 0x，如 0x;）
③  0[xX][0-9A-Fa-f]+                          →  INT（合法十六进制，如 0xFF）
④  0[0-7]*[89][0-9]*                          →  报错（非法八进制，如 09）
⑤  0                                           →  INT: 0（单独的零）
⑥  0[0-7]+                                    →  INT（合法八进制，如 0237）
⑦  {nonzero}{digit}*                          →  INT（合法十进制，如 234）
```

逐条读一遍这些模式（这是你"看不出哪个是规则"的根源——下面把每条拆开讲）：

- ① `0[xX][0-9A-Fa-f]*[G-Zg-z_][0-9A-Za-z]*`：`0x` 开头，中间若干个十六进制数字（`*` 可以是 0 个），然后**必须出现一个非十六进制字符**（`[G-Zg-z_]`，即 G~Z、g~z 或下划线），后面再跟任意个字母数字。这条专门抓"十六进制里混进了非法字符"，比如 `0x3G`、`0xGH`。
- ② `0[xX]`：就只有 `0x` 或 `0X`，后面什么都没有。比如 `0x;` 里的 `0x`。单独出现非法（十六进制至少要有一个数字）。
- ③ `0[xX][0-9A-Fa-f]+`：`0x` 后面**至少一个**十六进制数字（`+` 表示一个或更多）。正经的合法十六进制，如 `0xFF`。
- ④ `0[0-7]*[89][0-9]*`：`0` 开头，中间若干个 0~7（可以没有），然后**必须出现一个 8 或 9**（`[89]`），后面任意数字。专门抓"八进制里出现了 8 或 9"，比如 `09`。
- ⑤ `0`：就一个零。
- ⑥ `0[0-7]+`：`0` 后面**至少一个** 0~7。合法八进制，如 `0237`。
- ⑦ `{nonzero}{digit}*`：`1~9` 开头，后跟任意个数字。十进制，如 `234`。

①②④ 是"报错"动作，③⑤⑥⑦ 是"输出 INT"动作。

#### 4.1.2 演示：Flex 对每个输入会选中哪条

**例子 A：输入 `0x3G`**（非法十六进制）

| 规则 | 能吃到的最长字符 | 长度 | 能匹配吗 |
| -- | --------- | -- | ---- |
| ① 非法十六进制 | `0x3G` | **4** | ✅ |
| ③ 合法十六进制 | `0x3`（碰到非hex字符 G 就停） | 3 | ✅ |

最长是①（长度 4）→ **①胜出，报错 `Illegal hexadecimal number "0x3G"`**。关键：①比③长 1 个字符，最长匹配只认长度，和谁写前面无关。

**例子 B：输入 `0xFF`**（合法十六进制）

| 规则 | 能吃到的最长字符 | 长度 | 能匹配吗 |
| -- | --------- | -- | ---- |
| ① 非法十六进制 | 吃不动（没有非hex字符） | — | ❌ |
| ② 单独0x | `0x` | 2 | ✅ |
| ③ 合法十六进制 | `0xFF` | **4** | ✅ |

最长是③（长度 4）→ **③胜出，输出 `INT: 255`**。

**例子 C：输入 `09`**（非法八进制）

| 规则 | 能吃到的最长字符 | 长度 | 能匹配吗 |
| -- | --------- | -- | ---- |
| ④ 非法八进制 | `09` | **2** | ✅ |
| ⑤ 单独的0 | `0` | 1 | ✅ |
| ⑥ 合法八进制 | `0`（碰到 9 就停） | 1 | ✅ |

最长是④（长度 2）→ **④胜出，报错 `Illegal octal number "09"`**。

**例子 D：输入 `0237`**（合法八进制）

| 规则 | 能吃到的最长字符 | 长度 | 能匹配吗 |
| -- | --------- | -- | ---- |
| ④ 非法八进制 | 吃不动（没有 8/9） | — | ❌ |
| ⑥ 合法八进制 | `0237` | **4** | ✅ |

最长是⑥（长度 4）→ **⑥胜出，输出 `INT: 159`**。

#### 4.1.3 为什么这么设计

回答你的疑问："是不是先匹配到非法的，然后合法的自然筛出来？"——**不是**。Flex 不是按顺序"先试非法、再试合法"，而是**同时试所有规则，挑吃字符最多的那条**。所以：

- 合法和非法规则**谁写在前面都行**——它们对同一输入吃到的字符数永远不同，最长匹配自动定胜负。
- 只有**两条规则能吃到一样多字符**时顺序才有用，而本小节 7 条规则里合法和非法对同一输入永远不会等长（看上面各表的"长度"列），所以全靠最长匹配，不靠顺序。

**进制转十进制**用 `strtol(yytext, NULL, base)`，第三个参数传进制（8/10/16），C 标准库函数，比自己写转换省事且正确。

### 4.2 选做 2.2：指数形式浮点数

#### 4.2.1 先讲清楚：指数浮点的结构

指数浮点（Appendix A）由三部分依次拼接：

```
基数   e/E   指数
```

- **基数**：一串数字 + 一个小数点，小数点可在任意位置。三种写法：`1.05`（数字.数字）、`.5`（.数字）、`43.`（数字.）。在 `.l` 里用别名 `base` 统一表示这三种。
- **指数符号**：`E` 或 `e`。
- **指数**：可选的 `+`/`-` + 一串数字（**至少一位**）。

合法例子：`01.23E12`、`43.e-4`、`.5E03`。非法例子：`1.05e`（e 后面没数字）、`1.05e+`（有符号但没数字）。

#### 4.2.2 用到的别名和规则（共 3 条）

先把 `base` 别名讲清楚（这是看懂本节规则的前提）。`base` 是定义部分起的一个名字，代表"指数浮点的基数部分"那三种写法：

```
base  ({digit}+\.{digit}*|\.{digit}+)
```

拆开看这个别名（`|` 是"或"，分成左右两支）：

- 左支 `{digit}+\.{digit}*`：**一个或多个数字 + 小数点 + 零或多个数字**。覆盖 `1.05`（数字.数字）和 `43.`（数字.，后面零个数字）。
- 右支 `\.{digit}+`：**小数点 + 一个或多个数字**。覆盖 `.5`（.数字）。

> 注意 `\.`：`.` 在正则里是"任意字符"的元字符，要匹配字面小数点必须写成 `\.`（转义）。

浮点相关一共 3 条规则：

```
⑧  {base}[eE][+-]?{digit}+   →  FLOAT（合法指数浮点，如 1.05e-4）
⑨  {base}[eE]                →  报错（非法指数浮点，如 1.05e）
⑩  {digit}+"."{digit}+       →  FLOAT（基础浮点，无指数，如 3.14）
```

逐条拆：

- ⑧ `{base}[eE][+-]?{digit}+`：基数 + `e/E` + **可选符号**（`[+-]?`）+ **至少一位数字**（`{digit}+`）。注意 `[+-]?` 里的 `?` 表示"0 个或 1 个"，所以符号可有可无。这条要求 **e 后面必须有数字**。
- ⑨ `{base}[eE]`：只到 `e` 为止，后面什么都没要求。专抓"e 后面缺数字"的非法情况。
- ⑩ `{digit}+"."{digit}+`：基础浮点，小数点前后都要有数字，没有指数部分。

#### 4.2.3 演示：Flex 对每个输入会选中哪条

**例子 A：输入 `1.05e-4`**（合法指数浮点）

逐字符看：`base` 先吃掉 `1.05`，剩下 `e-4`。

| 规则 | 整条能吃到的最长字符 | 长度 | 能匹配吗 |
| -- | ----------- | -- | ---- |
| ⑧ 合法指数 | `1.05e-4` | **7** | ✅（e 后有 `-4` 满足 `[+-]?{digit}+`） |
| ⑨ 非法指数 | `1.05e` | 5 | ✅ |
| ⑩ 基础浮点 | `1.05`（碰到 e 就停） | 4 | ✅ |

最长是⑧（长度 7）→ **⑧胜出，输出 `FLOAT: 1.05e-4`**。注意⑧比⑨长了 2 个字符（多了 `-4`），所以⑧⑨谁写前面都无所谓。

**例子 B：输入 `1.05e`**（非法，e 后没数字）

| 规则 | 整条能吃到的最长字符 | 长度 | 能匹配吗 |
| -- | ----------- | -- | ---- |
| ⑧ 合法指数 | 吃不动（e 后必须有数字，但这里没有） | — | ❌ |
| ⑨ 非法指数 | `1.05e` | **6** | ✅ |
| ⑩ 基础浮点 | `1.05` | 4 | ✅ |

最长是⑨（长度 6）→ **⑨胜出，报错 `Illegal floating point number "1.05e"`**。这里⑧根本匹配不了（它的模式硬性要求 e 后有数字），所以不靠顺序，靠的是"⑧匹配不了、⑨能匹配"。

**例子 C：输入 `3.14`**（基础浮点，无指数）

| 规则 | 能吃到的最长字符 | 长度 | 能匹配吗 |
| -- | ----------- | -- | ---- |
| ⑧ 合法指数 | 吃不动（没有 e） | — | ❌ |
| ⑨ 非法指数 | 吃不动（没有 e） | — | ❌ |
| ⑩ 基础浮点 | `3.14` | **4** | ✅ |

最长是⑩（长度 4）→ **⑩胜出，输出 `FLOAT: 3.14`**。

#### 4.2.4 为什么这么设计（纠正一个常见误解）

你可能会担心："合法和非法指数浮点长度会不会一样、要靠顺序区分？" **不会**。看上面的表格：对任何具体输入，合法规则⑧和非法规则⑨吃到的字符数**永远不会相等**——

- 要么⑧比⑨长（输入是合法的，e 后有数字，⑧多吃那串数字）；
- 要么⑧根本匹配不了（输入非法，e 后没数字）。

**没有"两者等长、靠顺序裁决"的情况。** 所以⑧⑨谁写前面都行，全靠最长匹配。这和 4.1 的整数规则是同一个套路：把规则设计成"长度天然不同"。

> **为什么 `.5`（纯小数无指数）还是被拆分？** 因为 Appendix A 规定**基础浮点（规则⑩）小数点前后必须有数字**，`.5` 前面无数字，不算合法基础浮点。`.5` 只有作为指数浮点的基数时（即 `.5E03`，被⑧匹配）才合法。所以 `.5` 单独出现时，⑩匹配不了，被拆成 `DOT`（`.`）+ `INT: 5`，符合规范。

### 4.3 选做 2.3：注释（含嵌套禁止）

#### 4.3.1 用到的全部规则（共 2 条）

```
⑪  "//".*              →  动作：空（丢弃）
⑫  "/*"                →  动作：input() 循环读到闭合符，丢弃整段
```

- ⑪ `"//".*`：`//` 后跟 `.*`（`.` 是任意字符，`*` 是零个或更多）。`.` 默认不匹配换行符，所以 `.*` 会吃到行尾为止。整行注释，action 为空 → 直接丢弃。
- ⑫ `"/*"`：模式只匹配开头的 `/*` 两个字符。真正的"读到闭合符"工作不在模式里做，而在 action 里用 C 代码做。

#### 4.3.2 为什么块注释不能用正则，要用 action 里的循环

块注释 `/* ... */` 之间可以跨多行、可以包含任意字符（包括换行、引号、甚至看起来像闭合符的片段）。Flex 的正则**不好表达"两个标记之间的配对"**这种结构。所以做法是：模式只认 `/*` 这个开头，进到 action 后**用 `input()` 函数逐字符往下读**，手动找闭合的星斜杠：

```c
"/*"  {
    int c, prev = 0;
    while ((c = input()) != 0) {        // input() 读一个字符，EOF 返回 0
        if (prev == '*' && c == '/') {
            break;                       // 凑成星斜杠，注释闭合
        }
        prev = c;
    }
    // 若读到 EOF（c==0）说明注释没闭合，扫描随之结束
}
```

逻辑：记住上一个字符 `prev`。当前字符是 `/` 且上一个是 `*`，就说明遇到了闭合的星斜杠，注释结束，break 跳出循环。整段注释被读入并丢弃，**不产生任何 Token**。

#### 4.3.3 嵌套禁止的行为

C-- 不支持嵌套注释。对输入 `/* /* nested */ */`：

- 词法分析器从第一个 `/*` 开始读，遇到**第一个**闭合符（紧跟 `nested` 那个）就闭合，丢弃 `/* /* nested */` 这一段。
- 剩下的 ` */`（第二个）前面没有匹配的 `/*`，被 Flex 当普通字符处理，`*` 输出 `STAR`、`/` 输出 `DIV`。后续语法分析阶段会因此报 Error type B（语法错误）。

这正是 Project_1 选做样例 6 描述的行为。词法层只负责"正确丢弃第一个注释 + 如实把第二个闭合符拆成 STAR/DIV 输出"，语法错误的判定交给下一阶段。

## 5. 与下一阶段的衔接

本阶段输出的 Token 名（`TYPE`、`ID`、`INT`、`PLUS`、`RELOP`、`LP`……）正是下一阶段 **Bison 语法分析**里 `%token` 要声明的终结符。到时候会做两件事：

1. 给 `.l` 的 action 加上 `return PLUS;`（返回 Token 编号而不是 printf），让 Flex 的 `yylex()` 能被 Bison 调用。
2. 词法分析器从"独立程序"变成"被语法分析器调用的子模块"。

所以现在把每个 Token 的**名字**和**正则**定准确，是为后面打地基。Token 命名和 Appendix A 的产生式一一对应，别乱改。

## 6. 踩坑记录

1. **Flex 规则区不能在 pattern 和 `{` 之间写注释**。第一版把详细注释写在每条规则行尾和规则之间，Flex 报一片 `unrecognized rule`。教训：规则区的解释性注释只能放进 action 的 `{}` 内部（作为 C 注释），或移到 `.l` 顶部/本文档。规则区只保留"模式 + 动作"。
2. **`.l` 文件里任何地方的 C 块注释都不能出现 `*/` 字面量**（包括 `.l` 顶部注释和 action 内部注释）。Flex 扫描 `.l` 时按 C 注释规则解析，注释里写 `*/` 会提前关闭注释，导致后面的内容变成裸字符，报 `bad character`。补选做时我在注释里写了 `/* */`、`*/` 等字面量描述注释闭合，结果 flex 报第 7 行 `bad character`。解决：`.l` 里描述注释符号时改用文字（"星斜杠"）或 `//` 行注释（行注释里可以写 `*/`）。
3. **规则顺序**：多字符运算符在单字符前、关键字在 ID 前，这两条排序是正确性的保证，不能打乱。
4. **`.l` 文件必须 LF 换行**（协作约束）。CRLF 在同伴的 Linux 上会让 Flex 解析出错。
5. **`%option yylineno` 要写在规则区之前的定义部分**，且 `yyrestart` 切换文件时 `yylineno` 不会自动重置（本阶段单文件暂不涉及，多文件时要注意手动重置）。
6. **选做的合法/非法区分全靠"最长匹配"，不靠规则顺序**。无论是八进制/十六进制整数（4.1），还是指数浮点（4.2），都把规则设计成"合法形式比非法形式严格更长，或非法根本匹配不上"。对同一输入，合法和非法两条规则**永远不会吃到一样多字符**，所以顺序无关紧要。如果误以为"要靠顺序区分等长的合法/非法"，反而可能写错规则（比如用一条带可选数字的宽正则去覆盖非法情况，那时才会出现等长、才会被顺序坑）。
