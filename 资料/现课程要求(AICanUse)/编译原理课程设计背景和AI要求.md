### （一）AI 可直接使用的开发环境、工程结构、工具、硬件架构、编译链路

#### 1. 固定软件环境（AI 生成代码必须适配该版本）

* 操作系统：Ubuntu 20.04 GNU Linux
* 编译工具：GCC 7.5.0
* 词法 / 语法生成工具：GNU Flex 2.6.4、GNU Bison 3.5.1
* MIPS 仿真工具：QtSpim 9.1.9、课程配套 irsim 虚拟机小程序

#### 2. MIPS 架构开发基础

* MIPS：RISC 精简指令集架构（ISA）
* SPIM Simulator：用于运行调试 MIPS32 目标代码

#### 3. C 语言工程固定目录结构（AI 生成工程文件严格遵循）

plaintext
    C/
    ├─ src/
    │  ├─ PL/        // PL编译器源码目录，编译生成PL.exe
    │  └─ INTERPRET/ // 抽象机解释器源码，编译生成INTERPRET.exe
    └─ bin/
       ├─ PL.exe
       └─ INTERPRET.exe

配套测试文件：test.pl（PL 源代码）、test.pld、test.lab（运行输出文件）

#### 4. 程序编译运行链路（AI 代码逻辑必须匹配）

1. PL.C → 编译得到 PL.exe（PL 语言编译器）
2. INTERPRET.C → 编译得到 INTERPRET.exe（中间代码解释器）
3. 输入 test.pl 源码，编译生成抽象机中间代码，由解释器执行输出结果

#### 5. 分阶段代码开发任务（AI 可分段生成对应模块代码）

1. 词法分析模块：Flex 实现（实践 1.1）
2. 语法分析模块：Bison 实现递归下降分析（实践 1.2）
3. 语义分析模块（实践 2）
4. 中间代码生成模块（实践 3）
5. MIPS 目标代码生成模块（实践 4）
6. 代码优化模块（实践 5）
7. PL/0 编译器扩展模块（AI 可生成 for/repeat/case、函数、实型、记录类型扩展代码）

### （二）AI 开发硬性约束（代码、文档、输出必须遵守，不可违规）

1. **环境约束**
   
   * 所有代码必须兼容 Ubuntu20.04 + GCC7.5.0、Flex2.6.4、Bison3.5.1；禁止使用高版本 API、新语法、第三方库
   * MIPS 代码适配 QtSpim 9.1.9，不可使用其他仿真器指令格式
   * 工程结构严格遵循给定 C 目录架构，不得自定义文件夹层级

2. **功能开发约束**
   
   * 第五次课（7.2）后不再新增 PL/0 扩充内容，AI 辅助开发仅能基于现有扩充需求（for/repeat/case/ 函数 / 实型 / 记录），不允许新增未提及的语言特性
   * 编译器必须包含：词法错误捕获、语法错误处理、符号表管理、语法制导翻译完整逻辑，AI 生成代码不能省略核心模块

3. **交付文档约束（AI 辅助写报告限制）**
   
   * AI 生成实验报告不能使用空话套话，必须围绕「问题分析 + 解决思路」展开；禁止 AI 批量套模板
   * AI 仅可辅助生成代码注释、模块逻辑说明，完整实验分析、调试过程需学生自主补充

4. **考核与代码合规约束**
   
   * 分组模式分数规则限制：AI 不能直接输出完整成品工程用于三人组队（90% 计分），建议模块化分步生成；单人模式可完整生成全套代码
   * 评优答辩代码要求：AI 生成代码需预留可讲解逻辑点，保证学生能口述代码原理、回答老师提问，不可生成完全无法理解的晦涩代码

5. **演示程序约束**
   
   * AI 必须配套生成对应扩充功能的 test.pl 示例代码，用于上机演示；示例代码需覆盖所有自主拓展语法

6. **输出限制**
   
   * AI 生成录屏脚本、操作步骤仅作参考，不能直接生成完整录屏文件；文件大小控制需由学生自行压缩处理

6. **记录要求**

   * 在做的过程中遇到的各种问题和解决方法都记录下来，后期需要填写相应文档。
